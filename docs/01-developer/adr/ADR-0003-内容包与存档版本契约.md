# ADR-0003｜Schema 驱动内容包与显式存档版本契约

> 状态：Superseded by ADR-0007
> 日期：2026-07-10
> 决策者：项目技术/内容生产基线

## 背景与问题

旧原型已有多份 JSON，但字段、引用、编辑器、版本和运行时状态边界不完整。大型长期游戏需要更新旧地区、迁移存档、定位坏内容，并允许多人/AI 并行生产。若直接把 Godot Resource、场景路径或自由 JSON 当真相，字段变化和文件移动会破坏存档与工具。

## 决策驱动因素

- 内容可独立于引擎校验、审查、生成和迁移。
- 稳定 ID 穿过文件、包、存档和本地化。
- 官方地区可独立版本、依赖、更新和回滚。
- 错误尽可能在作者保存/CI 阶段发现。
- 编辑器与运行时共享默认值和规则。
- 存档升级失败不覆盖玩家原档。

## 备选方案

### A. Godot `.tres/.tscn` 为全部内容真相

与编辑器结合紧，但跨引擎工具、纯 CI、差异和业务迁移受限；场景路径容易渗入身份。拒绝作为业务契约源，保留为表现组合。

### B. 自由 JSON + 运行时容错

初期快，长期产生字段漂移、静默默认和坏存档。拒绝。

### C. 数据库/专用内容服务为源

查询强，但增加基础设施、离线和迁移复杂度；当前 Git 审查与本地生产更重要。暂不选。

### D. JSON Schema 源 + 校验/烘焙 + 版本化包

标准、可审查、可生成工具，运行时读取规范产物。选择。

## 决策

Authoring 内容使用 JSON Schema Draft 2020-12，所有正式对象使用稳定 ID 和拥有包。构建执行结构/语义/引用/预算验证，生成确定性 Baked Package。Godot Resource 只引用稳定资产/内容 ID。

Executable、Content API、Schema Set、Package 和 Save 使用独立版本线。存档只保存稳定身份和状态差异，通过逐版本纯迁移升级；写入采用验证后原子替换与备份轮换。

## 正面后果

- 内容作者、AI、编辑器和 CI 有共同合同。
- 文件重排和显示名变化不破坏存档。
- 地区包可以独立更新并明确兼容范围。
- 坏引用、不可达图和预算缩水在发布前失败。
- 可在不启动 Godot 的情况下做大量内容工具和测试。

## 负面后果与风险

- 需要维护 Schema、绑定、烘焙器、迁移器和错误诊断。
- Authoring 与 Baked 两形态增加构建步骤。
- JSON 不适合所有大资产/高频数据，需要二进制烘焙和 Godot 表现资源。
- 版本纪律增加前期成本，团队可能试图用“临时字段”绕过。

## 验证与放行门

- 1.0 目录、模板实例和错误夹具通过统一校验。
- 相同输入双烘焙得到相同数据/清单哈希。
- 包缺失、循环、版本冲突、哈希/签名错误安全失败。
- 至少三代 SaveVersion 黄金档迁移、失败恢复和回滚测试。
- 文件移动、显示名修改和 ID 别名迁移不丢状态。
- Editor 与 CLI 对同一内容给出相同错误和产物。

## 迁移与回滚

先从 1.0 蓝图建立 Schema，不直接把旧 JSON 当新合同。旧内容经导入器映射、人工审查和新 ID 注册。若 JSON Authoring 在特定领域不适合，可为该领域采用受版本控制的图/表源，但必须编译到同一 Validated Model、使用稳定 ID 和统一包/迁移规则。

## 标准依据

- [JSON Schema Draft 2020-12](https://json-schema.org/draft/2020-12)
