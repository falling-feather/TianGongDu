# 12｜内容、存档与版本契约

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Accepted Baseline
> 最后更新：2026-07-10
> 维护者角色：Runtime/数据契约负责人；Web、内容工具、后端和 QA 会签

## 1. 数据形态

1. **Authoring Source**：JSON、对话/任务图、场景标记与 DCC 源文件。
2. **Validated Model**：C++ ContentCore 通过 Schema、引用、语义和预算检查后的模型。
3. **Baked Package**：规范化二进制索引和资源组成的 `.tgdpack`，带版本、依赖与哈希。
4. **Runtime State**：从不可变定义实例化的 GameCore Session 状态。
5. **Local Persistence**：IndexedDB/原生文件中的快照、操作日志、备份和设置。
6. **Cloud Profile**：服务端修订快照、已接收 Operation 和设备/迁移元数据。

Release 不直接运行 Authoring Source。内容工作台、CLI、Web 预览和发布构建调用同一 C++ 校验/烘焙库。

## 2. Schema 规则

- JSON Schema Draft 2020-12。
- 根对象声明 `$id`、`schemaVersion`、`additionalProperties` 和最小示例。
- 单位进入字段名/类型，例如 `durationTicks`, `distanceMeters`, `sizeBytes`。
- 玩家文本只存本地化 key。
- 默认值只在 Schema/ContentCore 一处定义。
- Schema 后继续检查跨引用、图可达、预算、包依赖和同步语义。
- Native 与 WASM ContentCore 对同一输入产生相同规范模型和诊断代码。

## 3. `.tgdpack` 布局

`.tgdpack` 是项目定义的只读内容容器，不携带可执行代码：

```text
package.manifest.json
indexes/
  entities.bin
  quests.bin
  assets.bin
data/
  gameplay.bin
assets/
  atlases/
  audio/
  fonts/
  fx/
licenses/
```

Manifest 示例：

```json
{
  "packageId": "region_jiangnan",
  "packageVersion": "1.2.0",
  "schemaSetVersion": "1.1.0",
  "contentApiVersion": "1.0",
  "executableRange": ">=1.0.0 <2.0.0",
  "dependencies": [{ "packageId": "core", "range": "^1.3.0" }],
  "files": [{ "path": "data/gameplay.bin", "sha256": "...", "size": 123 }],
  "signature": { "algorithm": "ed25519", "keyId": "release-a", "value": "..." }
}
```

确切签名算法经安全评审锁定；示例表达合同。Web CDN 使用内容哈希 URL，`Cache-Control: immutable`；新 release manifest 原子切换包集合。

## 4. 版本线

| 版本 | 说明 | 示例 |
| --- | --- | --- |
| ExecutableVersion | C++/WASM/Native 代码 | 战斗实现、Axmol/Emscripten 升级 |
| WebShellVersion | HTML/JS/Service Worker/DOM | 缓存策略、桥协议、UI 修复 |
| ContentApiVersion | Runtime 可理解的内容语义 | 新任务/能力节点 |
| SchemaSetVersion | Authoring 文件结构 | 字段拆分、枚举变化 |
| PackageVersion | 单个 `.tgdpack` | 江南内容修复 |
| SaveVersion | 本地/云快照结构 | 世界状态迁移 |
| SyncProtocolVersion | Operation、修订和冲突协议 | 新合并域或错误码 |

不能用一个版本号假装全部同步。客户端、Web Shell、包、存档和服务声明兼容范围。

## 5. 稳定 ID

全局索引记录 ID、类型、拥有包、引入/弃用版本、替代 ID、来源和本地化 key。引用只用 ID，不用路径、显示名、指针或数组下标。

替换 ID 时旧 ID 进入 alias/tombstone；内容和存档迁移器转换，云端保留历史 Operation 可解析映射。禁止重新使用墓碑 ID。

## 6. 本地 Profile

每个 Profile 包含：

```text
ProfileHead
├── profileId / guestOrAccount / localRevision
├── saveVersion / contentApiVersion / packageSet
├── lastCommittedSnapshot / lastSyncedCloudRevision
├── pendingOperationCount / deviceId
└── checksum / migrationHistory

Snapshot
├── global / region / instance state
├── player progression / inventory
├── quests / relationships / records
├── random streams / checkpoints
└── appliedOperationIds compact index

OperationJournal
└── per-device ordered, globally idempotent operations not yet compacted
```

设置与 Profile 分离；输入、显示、音量和多数辅助选项按设备保存，可选同步小型偏好。

## 7. IndexedDB 事务

Web `IStorage` 使用 IndexedDB：

1. GameCore 在安全点生成不可变 Snapshot/Operation bytes。
2. Web Platform 打开 read-write transaction。
3. 写入新快照、操作、校验和与临时 Head。
4. 读回关键 Header/哈希验证。
5. 同一事务提交新 Head；失败则旧 Head 仍有效。
6. C++ 收到成功回调后才向玩家显示“已保存”。

保留最近自动保存轮换、章节/地区关键点和手动导出。`beforeunload` 不是保存机制；每次重大事务立即提交。

调用 `navigator.storage.estimate()` 显示空间，适当时请求 `persist()`。即便获准持久存储，仍提供云同步/导出，因为用户可主动清理站点数据。

## 8. 原生存储等价

Native `IStorage` 用临时文件 + fsync/原子替换 + 备份轮换，但保持异步 API 和同一字节格式。Web 导出档可导入 Windows/移动端；导入前验证版本、包、Profile 所有权和冲突。

平台安全存储只保存刷新 token/设备密钥等小秘密，不保存大型快照。

## 9. 持久 Operation

Operation 只表示可合并的领域事务，不是完整事件溯源：

```text
operationId       UUID/128-bit unique
profileId
deviceId
baseCloudRevision
domain            quest/inventory/relationship/record/settings/...
kind
payloadVersion
payload
logicalSequence
clientCreatedAt   diagnostic only
```

规则：

- 同一 `operationId` 重试只应用一次。
- `logicalSequence` 只在同一 `(profileId, deviceId)` 内单调，不能用来给两台离线设备建立全局顺序；服务端接受顺序由 `cloudRevision` 表达。
- Operation 先本地提交，再排队上传。
- 服务端用 base revision 检测分叉，不信客户端墙钟排序。
- 奖励 Operation 包含来源/`rewardDedupKey`，重复完成不会重复发放。
- 日志达到阈值或地区里程碑时压缩为新 Snapshot，并保留审计摘要。

## 10. 领域合并策略

| 领域 | 自动合并 | 冲突处理 |
| --- | --- | --- |
| 已发现图鉴/兴趣点 | 集合并集 | 删除通常不传播 |
| 单向成就/技能解锁 | 合法最大进度/并集 | 服务端验证前置 |
| 消耗品/货币 | Operation 账本 | 不用两个快照数值相加 |
| 唯一物品/奖励 | `RewardDedupKey` 去重 | 非法重复隔离并诊断 |
| 任务非互斥节点 | 状态机偏序合并 | 无合法共同后继则冲突 |
| 叙事选择/地区裁定 | 不自动覆盖 | 展示设备 A/B、时间线与后果，玩家选一条或复制 Profile |
| 关系维度 | 带来源的 Operation 重放 | 互斥里程碑进入叙事冲突 |
| 设置 | 设备本地为主 | 明确允许同步的偏好按修订 |

禁止全档 Last-Write-Wins；它会静默丢掉另一设备几小时进度。

## 11. 云修订协议

每个 Profile 云端有单调 `cloudRevision`。客户端同步请求提交：本地已知 revision、Snapshot hash、待同步 Operations、包/Schema 版本和幂等 request ID。

服务端事务：鉴权 → 校验设备/Profile → 去重请求/Operation → 加载基线 → 验证/应用或合并 → 生成新 revision → 写审计/快照 → 返回 accepted/rejected/conflicts 和新 head。

网络失败时本地不回滚。409 表示分叉，不是普通错误；客户端取 conflict bundle，在 DOM UI 中解释并解决。解决本身产生新的幂等 Operation。

## 12. 游客与账号链接

首次打开生成本地 guest Profile 和 device ID，无需注册。链接 OIDC 账号时：

1. Web Shell 完成授权码 + PKCE 等提供方推荐流程。
2. Sync Service 交换/验证身份，返回项目 Session token。
3. 客户端列出本地游客档和账号已有云档。
4. 无冲突可上传并执行 ProfileAdoption；双方都有进度时进入合并预览，绝不静默覆盖。
5. 完成后记录 Profile owner 和恢复方式；原游客档保留备份一段时间。

客户端不保存账号密码；Web 不把长期 token放 `localStorage`。具体会话/Cookie 策略由安全评审决定。

## 13. 迁移

Save/Operation/Package 迁移是逐版本、纯、可重复转换。Web 在 IndexedDB 隔离对象存储中迁移，验证后切 Head；Native 在临时目录迁移。云服务支持滚动部署期间的新旧协议窗口，数据库迁移先向后兼容再清旧字段。

迁移失败保留原快照/日志和诊断，不上传半迁移数据。旧客户端遇到过新的不可逆数据时明确要求更新或只读导出。

## 14. 错误分级

| 级别 | 示例 | 行为 |
| --- | --- | --- |
| Fatal Boot | Core 包哈希/签名、未知 Content API | 阻止进入并回滚 release manifest |
| Package Fatal | 可选地区包损坏 | 隔离/重下，不载入依赖档 |
| Local Save Error | IndexedDB 配额/事务失败 | 保留内存状态、导出/释放空间，不谎称保存 |
| Sync Conflict | 两端叙事/任务无合法合并 | 保留双方，要求选择/复制 Profile |
| Sync Transient | 离线、429、5xx | 本地继续，退避重试 |
| Content Error | 未知 ID、图不可达 | 构建失败；Release 不应出现 |

## 16. Web 持久化唯一主路径

正式 Profile 的 Web 主路径确定为“C++ 生成版本化字节 → JS 异步 IndexedDB 事务”。IDBFS 只允许第三方 POSIX 兼容、工具临时目录或迁移实验，不能作为另一套 Profile 真相，也不能与直接 IndexedDB 同时写同一 Head。

F1 IndexedDB v1 起始布局：

| Object store | Key | Value/索引 | 事务角色 |
| --- | --- | --- | --- |
| `profile_heads` | `[channel, profileId]` | 当前 snapshot ID、local/cloud revision、pending count、hash | 最后原子切换 |
| `snapshots` | `[profileId, snapshotId]` | 版本化 envelope + bytes；按 created/region milestone 索引 | 写新不覆盖旧 |
| `operations` | `[profileId, operationId]` | 唯一身份；索引 `[profileId, deviceId, logicalSequence]` 与 `[profileId, status]` | 与 Head 同事务追加；两设备相同 sequence 可共存 |
| `profile_meta` | `profileId` | Guest/account、device、显示摘要、迁移历史 | 不含长期 token |
| `device_settings` | `[channel, settingGroup]` | 输入/显示/音频/辅助 | 与 Profile 分离 |
| `migration_workspace` | `[profileId, migrationId]` | 隔离中的新档、校验与错误 | 成功后切 Head，失败可清理 |

数据库名、store 版本和字段由机器 Schema/迁移 v1 锁定；`internal/preview/stable` 使用不同数据库名和 origin/scope。包 Blob 默认由 Cache Storage/内容寻址缓存管理，不混入 Profile 事务。

多标签优先用 Web Locks 获取 `[channel, profileId]` 独占写锁；不支持时用 BroadcastChannel + 有界 lease（F1 起始心跳 5 秒、过期 15 秒）并由 IndexedDB compare-and-swap Head 最终防并发。非 Leader 可只读、请求接管或打开另一个 Profile，不能静默双写。

## 17. Snapshot/Operation v1 Envelope

正式编码（如 FlatBuffers/Protobuf/自定义）由 G3 数据 Spike 比较后锁定；在此之前字段语义已锁，不能直接序列化内存布局：

```text
SaveEnvelopeV1
  magic / envelopeVersion / payloadEncoding
  profileId / snapshotId / parentSnapshotId
  saveVersion / contentApiVersion / executableVersion
  packageSetId / createdLogicalSequence / checkpointKind
  uncompressedSize / payloadHash / envelopeHash / payloadBytes

OperationEnvelopeV1
  operationId / profileId / deviceId / logicalSequence
  baseCloudRevision / domain / kind / payloadVersion
  causalParentIds[] / rewardDedupKey? / preconditionHash?
  payloadHash / payloadBytes
```

F1 `SaveEnvelopeV1` 字节基线使用 magic `TGDSAVE\0`、版本 `1.0`、显式小端和 176-byte 固定 Header；当前 payload encoding 只有 `raw`，单 payload 上限 16 MiB。`payloadHash = SHA-256(payloadBytes)`；`envelopeHash = SHA-256(从 magic 到 payloadHash 的规范 Header bytes)`，不包含自身或 payload body。编码器忽略调用方传入的两个派生哈希并按规范重算；解码先验证版本、长度、ID、父子关系和枚举，再验证两个 SHA-256。未知版本、截断、尾随、超限或任一哈希不符都失败关闭。该 envelope 可由 Web/Native 共读，但不等于把 C++ 内存布局直接落盘。

`logicalSequence` 是设备内序列；同步回来的其他设备 Operation 以 `operationId` 共存，不能重编号或覆盖。`cloudRevision` 是服务端事务提交序，冲突合并保留因果基线而不信任客户端墙钟。

ID 使用明确的 128-bit 随机/时间有序规范并以规范小写文本只作调试，字节存储固定；不能用显示名或 JS number。`payloadHash = SHA-256(未压缩 payloadBytes)`；`envelopeHash = SHA-256(规范化 header 字段〔排除 envelopeHash/签名本身〕 + payloadHash)`，因此不发生自引用。压缩不是身份。外部导入容器还包含 manifest、所需包、每文件哈希和可选用户口令加密元数据，绝不把“改扩展名的 JSON”当安全备份。

F1 起始压缩策略：累计 500 个待处理 Operation、日志超过 4 MiB 或章节/地区里程碑时尝试生成新 Snapshot；只有新 Snapshot 和 Head 同事务提交后才标记旧 Operation 可归档。至少保留最近 2 个自动安全快照、章节/地区里程碑和用户导出来源；真实阈值由存档体积/恢复时间测试校准。

## 18. 持久事务安全点

必须立即生成并请求本地提交：获得唯一奖励、任务里程碑、关系阶段、Boss 结算、重大选择、工艺永久变更、旅行、ProfileAdoption/冲突解决。普通位置/生命/消耗按 30–90 秒自动安全点或进入安全区合并，但页面隐藏时只提交已构造的合法快照，不现场运行新玩法逻辑。

本地逻辑提交成功只在以下全部成立时返回：bytes 已验证 → snapshot/operations 已写 → Head compare-and-swap 成功 → transaction `complete`。请求发出、单个 `put()` success、`beforeunload` 或内存对象更新都不等于已保存。对唯一奖励/重大选择等关键事务，Web Platform 在浏览器支持时请求 IndexedDB `durability: "strict"`；它仍只是 UA 能力/提示，不等同跨浏览器断电 `fsync` 保证，因此必须保留多 Head、校验、导出和云备份路径。玩家文案“已保存在此设备”表示 origin 内事务已逻辑提交，不宣称硬件物理落盘绝对完成。

故障规则：

- 配额不足：保留上一 Head 和内存中未保存状态，停止继续发唯一奖励，提供释放缓存/导出/重试。
- 快照损坏：隔离坏对象，尝试上一安全快照；不覆盖备份。
- 双标签竞争：只有 CAS 胜者提交，败者转只读并要求重新加载差异。
- 迁移失败：保持原数据库/Head 可读，保存诊断和可导出原档。
- 操作同步成功但本地 ack 失败：下次用 operationId/requestId 重查，不能重新发奖励。

## 19. 七条版本线的兼容规则

| 线 | Stable 读取 | Stable 写入 | 不兼容时 |
| --- | --- | --- | --- |
| Executable/WebShell | 当前签名 release 的配对范围 | 不直接写 Profile 字段 | 阻止混版并重载/回滚，不让旧 Shell 驱动新 ABI |
| Content API | Manifest 声明的兼容范围 | 由可执行语义决定 | 包不挂载；保留旧可运行集合 |
| Schema Set | 当前 Authoring + 有迁移的旧版 | 当前版 | 工具只读/迁移，不静默丢字段 |
| Package | 满足依赖与 executable range | 烘焙器当前格式 | 隔离/重下/回滚 manifest |
| Save | 当前版 + 明确迁移链；每个 Stable 至少支持上一正式版迁入 | 只写当前版 | 新客户端迁移；旧客户端只读导出或要求更新 |
| Sync Protocol | 服务至少支持当前 Stable 与上一支持窗口 | 协商后的最高共同 minor | 426/稳定错误要求更新；不上传未知 Operation |
| DB Schema | expand→migrate→contract 窗口内新旧服务 | 服务各自兼容字段 | 停止部署/roll forward，禁止破坏性回滚 |

兼容范围由 release manifest 和机器合同表达，不能只写在代码条件里。HTML、Service Worker、WASM、包和档形成合法组合后才创建 Session；旧标签页发现 manifest 更新可继续旧 Session 到安全点，但不能用旧 ABI 写新格式。

## 20. `.tgdpack` v1 冻结门

第 3 节目前是逻辑布局，不是已发布字节格式。G3 前必须在同一合同中锁：magic、container version、endianness、alignment、目录/索引编码、压缩算法和块大小、规范 manifest 序列化、签名覆盖范围、key/撤销、路径/数量/尺寸上限、依赖求解、range 下载、临时下载、校验、原子激活、旧包清理和回滚状态机。

交付必须包括 writer、Native/WASM reader、inspector、一个 canonical golden pack 和负向夹具：截断、单字节篡改、错误签名、解压炸弹、路径逃逸、循环依赖、未知 major、断点续传和旧 manifest 回滚。决策影响长期兼容时新建 ADR，不修改 0007 的历史原因。
