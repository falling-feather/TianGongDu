# 10｜技术架构与依赖规则

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Accepted Baseline；F1 锁定精确工具版本和 Axmol Web 可行性
> 最后更新：2026-07-10

## 1. 架构目标

项目必须同时满足：网页即开即玩、横版动作响应、长期地区扩展、内容模板化、浏览器本地存档、跨设备接续和未来原生多端。任何平台便利都不能侵入 Gameplay；任何云能力都不能让网络中断毁掉单机进度。

优先级：稳定边界 → 可验证 Web 体验 → 内容吞吐 → 多端复用 → 性能优化 → 局部便利。

## 2. 技术基线

| 层 | 选择 | 策略 | 原因/边界 |
| --- | --- | --- | --- |
| 游戏核心 | C++20 | 固定 60 Hz、RAII、可原生/WASM 编译 | 单一 Gameplay，多端共享 |
| 2D 宿主 | Axmol 2.11.4 / 2.11.x LTS | 固定 tag+commit；Web Preview 必须过 F1 | 轻量 2D、C++、WebAssembly 与原生平台 |
| Web 编译 | Emscripten SDK | 通过 emsdk 安装；F1 锁精确版本/镜像 | C/C++ → wasm32 + JS loader |
| Web 渲染 | WebGL2 | 必选基线；不把 WebGPU 作为 1.0 依赖 | 与 Axmol Web 路径和现代浏览器一致 |
| Web UI | 语义化 DOM | Canvas 只绘制游戏世界/战斗 | 文本、输入、可访问性和响应布局更可靠 |
| 本地存储 | IndexedDB | 异步平台桥；IDBFS 仅作 POSIX 兼容层 | 浏览器刷新/重启后持久化 |
| 离线/缓存 | Service Worker + Cache Storage + HTTP Cache | 内容寻址文件，显式版本激活 | PWA、弱网、分包和回滚 |
| 内容源 | JSON + JSON Schema 2020-12 | C++ ContentCore 原生/WASM 双形态 | 编辑器/CI/运行时共享合同 |
| 运行内容 | `.tgdpack` + 二进制索引 | 清单、哈希、依赖、版本、签名 | 流式、CDN、缓存和跨端一致 |
| 同步服务 | C++20 + Drogon + PostgreSQL | F1 同步 Spike 后锁版本 | 与 C++ 合同共享，异步 HTTP/数据库 |
| 身份 | 外部 OIDC 提供方 | 访客先玩，后链接账号 | 不自建密码、找回和 MFA 系统 |
| 构建 | CMake Presets + Ninja/平台生成器 | Web/Windows/Server 独立 Preset | 可重现、多目标、不靠 IDE 魔法 |

机器可读版本见 [`../../content/design/technical-baseline.json`](../../content/design/technical-baseline.json)。

## 3. 为什么是 Axmol，但不把核心交给 Axmol

Axmol 官方定位为 C++ 多平台 2D 引擎，并列出 WebAssembly、WebGL2、Windows、移动端等目标；2.11.x 是稳定维护线。它适合“造梦西游式”横版 Sprite/骨骼动画、相机、音频和输入，但官方仍把 WebAssembly 标为 Preview。

因此决策是：

- 使用 Axmol 降低 2D 渲染/平台宿主成本。
- 不让 Axmol Scene/Node/Scheduler 成为玩法状态、任务状态或存档身份。
- Web Preview 必须经 F1 真实构建、浏览器矩阵、长时内存和资源加载验证。
- 若宿主失败，以 SDL/WebGL2 或其他 C++ 2D 宿主替换 Presentation/Platform；Contracts/Runtime/Gameplay/Content 保持。

这不是回到旧版“先写 Canvas Demo 再想架构”，而是先固定可替换边界。

## 4. C++ 目标与依赖图

```text
tgd_contracts (header+library)
      ▲
tgd_runtime
      ▲
tgd_gameplay
      ▲
tgd_presentation_axmol
      ▲
┌─────┴──────────────┐
tgd_platform_web   tgd_platform_native
      ▲                ▲
web_shell          platform launchers

tgd_content_core ── depends on contracts only
tgd_sync_contracts ─ depends on contracts only
tgd_sync_service ─── depends on sync_contracts + Drogon/PostgreSQL
```

### 允许

- Runtime 依赖 Contracts。
- Gameplay 依赖 Runtime/Contracts。
- Presentation 读取快照/事件，把输入映射成命令。
- Platform 实现 `IStorage`, `INetwork`, `IIdentity`, `ILifecycle`, `IClockSource`, `IAssetTransport`。
- ContentCore 复用 Contracts，但不依赖 Gameplay/Axmol。
- Sync Service 复用稳定存档/操作 DTO，不链接客户端渲染和玩法循环。

### 禁止

- Contracts/Runtime/Gameplay include Axmol、Emscripten、Drogon、DOM/JS 绑定头。
- JavaScript 直接修改生命、背包、任务、关系或随机流。
- Axmol Scheduler 决定任务时间或战斗 Tick。
- `#ifdef __EMSCRIPTEN__` 散落在 Gameplay；平台差异只在 Platform/Build。
- Web 与 Native 各自维护存档字段或内容加载规则。
- Server 反序列化客户端任意 C++ 类型或信任客户端时间戳。

CI 用 include/layer lint、链接目标和架构测试守住依赖。

## 5. Web 进程模型

### 5.1 必选单线程构建

所有功能必须在不启用 Pthreads 的 wasm32 构建可玩。渲染、输入和模拟位于浏览器主线程，资源网络/IndexedDB 通过异步 JS API；重工作业切片并在预算安全点提交。

原因：Emscripten Pthreads 依赖 SharedArrayBuffer、COOP/COEP，且官方说明不能用同一二进制自动回退单线程。单线程是最大兼容与第三方嵌入的安全基线。

### 5.2 Pthreads 增强构建

满足 COOP/COEP、Worker 与 SharedArrayBuffer 时，可加载单独的 `web_pthreads` 产物，把主循环代理到 Worker/Pthread，并为解压/寻路/内容解析使用固定 Worker 池。

- 服务端必须返回 `Cross-Origin-Opener-Policy: same-origin`。
- 子资源满足 `Cross-Origin-Embedder-Policy: require-corp` 或可验证替代策略。
- CDN、字体、身份页面和第三方资源必须兼容隔离策略。
- 启动器先能力探测再选择文件，两个产物有相同内容/存档 API。
- Pthreads 性能没有显著收益时不为“看起来高级”启用。

## 6. 主循环与页面生命周期

Axmol/Web Shell 驱动渲染帧，GameCore 使用固定步长累加器：

```text
Poll Platform Events
Accumulate visible elapsed time
Run 0..N fixed 1/60s ticks (N capped)
Build immutable presentation snapshot
Render interpolated frame
Flush low-frequency UI events
Schedule async storage/network/assets
```

单帧最多追赶 4 个 Tick；超过时记录卡顿并丢弃过量显示时间，不能在页面恢复后追赶几分钟。`visibilitychange/pagehide/blur` 清空按键、暂停模拟、提交本地存档事务；恢复后重新校验输入、音频上下文、WebGL context 和资源句柄。

WebGL context lost 时停止提交绘制，保留 GameCore；context restored 后重建 GPU 资源和视图。无法恢复时保存安全点并给出重载入口。

## 7. JavaScript↔C++ 窄桥

桥只处理平台能力：启动配置、DOM 命令/UI 事件、IndexedDB、Fetch、OIDC token、剪贴板、全屏、Service Worker 状态和诊断。

建议接口：

```text
tgd_boot(BootConfig)
tgd_submit_platform_event(PlatformEventBatch)
tgd_submit_ui_command(UiCommand)
tgd_poll_ui_events(UiEventBuffer)
tgd_complete_async_request(RequestId, ResultBytes)
tgd_request_shutdown()
```

- 热路径使用固定布局二进制/环形缓冲，避免每帧 JSON 与大量 Embind 对象。
- DOM UI 由事件驱动或 10–20 Hz 小快照更新，不复制完整 WorldState。
- 每个异步请求带 RequestId、取消、超时和 Session generation，旧回调不能写入新存档。
- JS 异常转为结构化 PlatformError，不能穿过 WASM 边界破坏栈。
- C++ 向 JS 暴露最小 C ABI/受控 binding，不导出任意对象和指针。

## 8. Web Shell 职责

Web Shell 负责：

- 能力探测、选择单线程/Pthreads 产物。
- 加载/实例化 WASM，显示真实分阶段进度。
- Canvas 尺寸、DPR、全屏、页面生命周期和输入前置处理。
- DOM HUD、菜单、字幕、设置、账号/同步冲突与辅助技术。
- Service Worker 注册/更新、内容缓存状态和版本激活。
- IndexedDB 异步存储桥和多标签写入锁。
- OIDC 跳转/令牌安全交换，不把长期秘密交给 C++/本地存储。

它不拥有角色、战斗、任务、背包、掉落、关系、剧情选择和权威计时。

## 9. 本地存储

GameCore 只看异步 `IStorage`：`Read`, `WriteAtomic`, `List`, `Delete`, `EstimateQuota`, `RequestPersistence`。Web 实现优先直接使用 IndexedDB 对象存储保存快照、操作日志和元数据；IDBFS 只为必须使用 POSIX 文件的第三方/工具路径提供兼容。

保存流程：C++ 生成不可变字节快照/操作 → JS 开 IndexedDB 事务 → 写新 revision 与校验 → 提交 → 回调成功 → C++ 更新本地 head。刷新前的 `beforeunload` 不作为唯一保存机会。

浏览器存储默认可能被清理；游戏提供持久存储请求、云同步、手动导出/导入和清楚的本地/云状态。

## 10. 资源与内容流式

启动只内嵌最小字体/错误 UI。其余通过内容清单获取：

1. Web Shell 下载并验证签名的 release manifest。
2. 根据地区/语言/质量档计算包依赖。
3. Service Worker/HTTP Cache 以内容哈希 URL 缓存压缩包。
4. Platform AssetTransport 分块读取，C++ 校验 SHA-256/索引。
5. Runtime 在安全预算内解压/解析，Presentation 延后 GPU 上传。
6. 旧包在新 Session 验证后清理，不覆盖当前可运行版本。

不把全部 1.0 资源塞进 Emscripten preload `.data`。核心 WASM、Boot Pack、地区包、语言包和高清包独立版本。

## 11. Native 复用

Windows/Android/iOS 使用相同 CMake 目标 `contracts/runtime/gameplay/content`，只替换 Platform 和 Axmol 原生宿主入口。

- Native 文件系统实现同一异步语义，即使底层可同步。
- Native HTTP/OIDC/安全存储通过平台接口。
- 内容包、ID、存档、同步协议和回放完全相同。
- 输入映射从统一 Action 定义生成，平台只报告物理输入。
- 商店/IAP 若未来加入，属于平台服务，不直接发放 Gameplay 奖励；服务端验证后产生幂等操作。

每个 PR 的纯核心测试必须原生运行；多端真正发布仍需各自性能、输入和审核门。

## 12. 同步服务边界

同步服务不是实时游戏服务器。它负责身份映射、Profile head、快照、幂等操作、冲突检测、设备管理、配额、审计和数据导出/删除。

参考实现 C++20/Drogon + PostgreSQL；无状态 API 实例可横向扩展。Blob/大快照可放对象存储，PostgreSQL 保存 revision、操作索引、哈希与事务元数据。Redis 只在真实压测证明需要时加入。

客户端在无服务时完整游玩；服务端不可用时只积压已提交本地的操作。

## 13. 错误与诊断

- Boot 阶段错误必须区分能力缺失、WASM 加载、内容包、存储、账号和同步。
- Content/Schema 错误在构建期硬失败，Release 不静默默认。
- IndexedDB/配额错误保留内存快照并引导导出/释放空间，不能宣称已保存。
- 同步 401/409/429/5xx 有独立处理；重试使用退避、抖动和幂等 key。
- C++ 日志带 build/session/profile/device/revision/package/correlation ID，但不记录 token 和完整存档。
- Web 性能 HUD显示 WASM heap、JS heap（可用时）、纹理估算、下载、缓存、Tick、帧、长任务和同步队列。

## 14. F1 烘焙矩阵

| 实验 | 成功条件 | 失败后的动作 |
| --- | --- | --- |
| Axmol Web Host | 三主流桌面浏览器启动/输入/音频/context 恢复 | 修宿主或评估 SDL/WebGL2 Presentation |
| WASM Build | Release 可重现、体积/符号/异常策略明确 | 调整 Emscripten/依赖/模块边界 |
| Single/Pthreads | 单线程完整；线程版同结果且有实测收益 | 保留单线程，延后线程版 |
| Asset Streaming | 分包、缓存、校验、弱网/回滚无坏状态 | 缩包/改容器/改解压策略 |
| IndexedDB | 刷新、崩溃、配额、淘汰提示和导出恢复 | 改异步桥/事务/快照频率 |
| Cloud Sync | 双端离线编辑、重复请求、冲突可解释 | 收缩自动合并域，保留手动选择 |
| Native Replay | Windows 与 Web 同命令流同校验和 | 清除平台时间/浮点/排序差异 |
| Content Workbench | 模板创建、Native/WASM 校验一致 | 改工具宿主，不改内容合同 |
