# 共享上下文与执行规范

> 文档定位：所有新对话必须共同遵守的背景、权威入口和工程纪律。
> 适用范围：本交接包中的主开发、美术、UI/UX、内容关卡和 QA 目标。
> 最后更新：2026-07-13；维护者：技术负责人、制作人。

## 1. 项目与体验方向

《天工渡》是浏览器首发、C++ 核心、Web/Native 复用的 2.5D 斜向全景动作 RPG。F1 唯一纵切为 `f1_rainy_umbrella_trial`“雨夜试伞”：获得控制到任一结算首玩至少 60 分钟，完整 E2E 预算约 70 分钟，不得用加载、挂机、降速、重复刷怪或失败重试凑时。

视觉和体验以《斗战神》为第一指导，学习受控斜向全景、人物/场景尺度、空间纵深、东方材质重量、群战构图和厚重战斗阅读性；《暖雪》只作为俯斜输入与轮廓可读性的第二参考。必须形成《天工渡》原创角色、器物、建筑、UI、剧情和动作语言，不得重制任何参考作品的可识别表达。

5. `docs/09-术语与索引.md`：术语、机器入口、状态词和待决问题。
6. 专业文档：玩法/内容读 04、06、07、11；UI 读 08、10、Action Registry；存档读 10、12、16、17；资产读 14、16、17；QA 读 16–18。

文档与机器合同冲突时停止相关实现，记录事实、影响和建议归属，不得自行选一份继续。

## 3. 不得破坏的工程边界

- GameCore 使用 C++20、固定 60 Hz；随机、命令、事件和快照可序列化、可回放。
- Web Single 是必选基线；Pthreads 只能在证据成立后作为增强。
- Axmol/Canvas 属于 Presentation；DOM/JavaScript/Web Shell 属于 Platform/UI；它们都不拥有战斗、任务、奖励或世界真相。
- Gameplay 不直接控制 Axmol Node、DOM、IndexedDB 或网络。
- Profile 只通过异步桥事务写入 IndexedDB；IDBFS 不是第二份存档真相。
- 奖励以稳定 Persistent Operation 与 `rewardDedupKey` 幂等；只有本地事务确认后才能显示“已保存/已发放”。
- 新 NPC、Boss、任务、敌群和地区通过 Definition/Schema/Stable ID 进入，不为单个内容添加专属 Runtime 类或硬编码 ID。
- 适用时覆盖正常、失败、取消、重试、失焦、刷新、离线、旧版本和迁移；不能只验证快乐路径。

## 4. 执行节奏

优先快速收口边界明确的代码/内容模块：先冻结状态 Owner、合同、错误和夹具，再连续实现一个闭环，运行受影响范围的快速测试，模块结束时集中 Review 和集成验证。GitHub、远端三浏览器和完整 CI 只在形成阶段性候选提交时集中调试，不能重新把大部分时间耗在每个微提交的远端故障上。

任何合同、Schema、Stable ID、存档格式、依赖方向或许可变化必须立即检查，不能推迟到阶段末。

## 5. 多对话与 Git 纪律

- 集成分支是 `codex/f1-development`；角色分支见交接 README，仅用于独立 worktree 并行。
- 开工先记录分支、HEAD、`git status --short`、基线和预计修改文件。
- 现有未提交修改属于用户/前一对话；不得 reset、checkout、覆盖或混入无关提交。
- 每次提交只表达一个可回滚意图，标题以当前版本 `V2.2.1` 开头并使用语义化 scope。
- 提交前检查完整 diff、暂存范围和 `git diff --cached --check`；不提交密钥、个人路径、缓存和不可复现构建产物。
- 角色分支只交回 commit hash，不自动向 `main` 提交、不重写历史、不未经评审合并集成分支。

## 6. 文档纪律

- 02：当前任务、优先级、阻塞、验收与计划变化。

## 7. 每个新对话的首条开工说明

在写文件前输出：任务 ID/标题、单一 Owner/必审、玩家或生产价值、权威输入章节、Stable ID/Schema、交付产物、非目标、测试/人工/性能验收、风险/回退、预计修改文件和文档。若不满足 01 §12 DoR，先补任务卡，不直接铺代码。

交回时必须给出：结果、commit hash、实际测试、证据位置、仍缺内容、是否可合并和下一步。目标只有在权威路径可到达、Web/Native 适用路径成立、负向与恢复路径通过、文档/迁移/诊断同步且证据绑定提交后，才能标为 Verified。
