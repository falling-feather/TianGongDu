## 3. 技术来源

下列为 2.1 技术决策直接使用的原始/官方来源，访问日期均为 2026-07-10：

- [Axmol 官方仓库](https://github.com/axmolengine/axmol)与 [v2.11.4 官方发布](https://github.com/axmolengine/axmol/releases/tag/v2.11.4)：确认其为 C++ 多平台 2D 引擎；2.11.x 是 v2 最终 LTS 维护线，2026-07-06 发布的 2.11.4 是当前 F1 候选；WebAssembly 仍标 Preview，Web 使用 WebGL2。候选版本不等于工具链已锁，最终 tag/commit/hash 仍由 G1 证据决定。
- [Emscripten 官方文档](https://emscripten.org/docs/)：确认 C/C++ 经 LLVM/Binaryen 编译为 WebAssembly，并提供 Web/Node 运行与模块拆分、优化、调试能力。
- [Emscripten 官方 SDK 安装](https://emscripten.org/docs/getting_started/downloads.html)：emsdk 是官方持续测试的安装方式；项目必须锁精确版本而非长期跟随 `latest`。
- [Emscripten 文件系统](https://emscripten.org/docs/api_reference/Filesystem-API.html)：浏览器默认 MEMFS 不持久，IDBFS 可通过 `syncfs()` 持久到 IndexedDB。
- [Emscripten Pthreads](https://emscripten.org/docs/porting/pthreads.html)：浏览器线程依赖 SharedArrayBuffer 与 COOP/COEP，且单线程/多线程需要两套构建。
- [Emscripten Fetch API](https://emscripten.org/docs/api_reference/fetch.html)：C++ 可异步下载资源并将文件持久到 IndexedDB。
- [MDN WebAssembly](https://developer.mozilla.org/en-US/docs/WebAssembly)：WebAssembly 与 JavaScript 协作，在现代浏览器运行 C/C++ 编译结果。
- [MDN WebGL](https://developer.mozilla.org/en-US/docs/Web/API/WebGL_API)：WebGL2 基于 OpenGL ES 3.0 能力并由现代浏览器硬件加速。
- [MDN IndexedDB](https://developer.mozilla.org/en-US/docs/Web/API/IndexedDB_API)：用于大量结构化数据和 Blob 的浏览器事务存储，也可在 Worker 使用。
- [MDN 浏览器存储额度与持久性](https://developer.mozilla.org/en-US/docs/Web/API/Storage_API/Storage_quotas_and_eviction_criteria)：默认存储可能被淘汰；可请求持久存储，但云同步/导出仍是必要备份。
- [web.dev PWA 缓存](https://web.dev/learn/pwa/caching/)：Service Worker/Cache Storage 可支持离线和分层缓存，但必须显式管理版本与带宽。
- [Drogon 官方仓库](https://github.com/drogonframework/drogon)：确认其为 C++17/20 异步 HTTP/WebSocket 框架，并支持 PostgreSQL、Redis 与协程。
- [JSON Schema Draft 2020-12](https://json-schema.org/draft/2020-12)：内容与技术基线 Schema 规范源。

技术文档只用官方文档、标准或上游仓库作为事实依据。二手文章可帮助发现问题，不能独立支持长期决策。

## 7. 研究登记格式

每项真实工艺研究记录：主题、来源、作者/机构、日期、链接/馆藏、可验证事实、团队推断、幻想改写、使用位置、审阅人和许可状态。生成式 AI 输出不能作为事实来源。

## 8. 变更检查

提交文档前回答：

1. 信息是否已有唯一归属？
2. 是未来计划还是已完成事实？
3. 是否改变世界正史、产品边界或长期架构？
4. 是否需要同步 Schema、技术基线、内容目录、测试或迁移？
5. 是否留下来源、负责人、日期和验证方式？

如果答案不清楚，先修归属和证据，不新增“临时文档”。

## 9. 文档与实现状态词

| 状态 | 只表示 | 不表示 |
| --- | --- | --- |
| Draft | 可讨论草案 | 已批准范围 |
| Scope Approved | 数量、目标和边界获批 | 每项内容已可制作或已实现 |
| Accepted Baseline | 当前应遵循的规则/技术方向 | Spike 参数已经锁定 |
| Contracted | Schema/API/状态/错误/版本足以独立实现 | 代码已通过 |
| Implemented | 代码/内容存在且本地路径可达 | 浏览器矩阵、性能或发布门已过 |
| Validated | 指定构建和证据通过目标门 | 已向玩家发布 |
| Released | 签名版本已部署并可回滚 | 永久完成、不再维护 |
| Superseded | 由新决定取代，保留历史 | 可以删除或继续混用 |

README、02、03 和 ADR 使用同一组词。开发演示不得用“基本完成”“差不多可用”替代明确状态。

## 10. 文档维护门

- 00/01/02/03 每个里程碑复审；04–09 在相关内容门复审；10–19 在模块契约或依赖变化时复审。
- 每份文档至少有状态、职责/读者、最后更新和维护者角色；开发组确定后在仓库治理配置中映射到具体 Owner。
- PR 改变接口、Schema、玩家规则、版本兼容、部署或验收时，必须更新唯一归属文档和反向链接。
- 发现文档与可执行证据冲突时，先把状态降为待验证/阻断，不以“代码为准”静默保留错误文档。
- 每个 G1/G2/G3/M1 结束时运行链接/层级/术语校验，并由模块 Owner 确认所属章节仍可执行。
