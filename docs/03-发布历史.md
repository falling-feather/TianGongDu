## 2.2.1-f1.32｜训练两前沿权威重试恢复候选｜2026-07-15

### 已形成的候选事实

- 既有 schema 1.6.0 的八个 cue / 16 条 evidence rule 已为檐守、翻花两个精确训练 frontier 各自提供 `recovery_offer` + `recovery_resume`；App 不扫描 cue/rule 反选 Objective，恢复焦点仍来自当前 Beat author order 的首个未完成 Active frontier。
- 玩家倒地时，App 只从 Combat snapshot 暂存已验证的非零权威 stance。Session、Encounter、Combat 与移动安全点重试成功后，如初始 stance 不同，App 仅排入既有 `switch_stance` 命令；CombatResolver 仍是唯一状态 Owner，App 不直接写 Actor。
- `F1QuestUiRecoveryResumeGate` 把倒地 Tick 作为实际 retry Tick 的下界。仅当 retry 后 stance 与倒地权威 stance 不同时，才要求匹配 `stance_changed` 严格晚于真实 retry；已匹配时不要求该事件，但仍须等待后续三组件同步 Tick 与 Active/non-defeated/matching-stance Actor snapshot。旧候选在 `messageSequence=17` 停留的根因正是把实际 retry Tick 错误要求为倒地 Tick；修复后仍会拒绝早于倒地、重复 mark、需要切换却未生效/非法的 stance、缺少所需 Combat 事件及 Quest/Objective 漂移。
- 正式 renderer 为檐守/翻花分别提供 offer/resume 玩家文案，且不把 attempt-time classification 或“合格计时”等 QA 术语送入玩家 copy。该变更只消费权威 source + focus Objective，不拥有玩法进度。
- 浏览器高水痕/迎风路线在檐守 frontier、循钟声/背风路线在翻花 frontier 各从真实敌人攻击走过倒地→offer→Canvas R→三组件恢复→Combat-owned stance 生效→fresh resume→正常移动回线标→敌人重新攻击，并继续完成 7 Beat、Boss、双结算、奖励与 replay probe。
