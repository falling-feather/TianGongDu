# 16｜测试、CI 与发布门禁

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Active Baseline
> 最后更新：2026-07-10

## 1. 测试目标

测试不是证明“代码跑过一次”，而是降低长期更新时破坏旧地区、旧存档和旧内容的概率。自动化守住确定性、契约和组合边界；人工游玩判断节奏、可读性、情绪和真实深度。

完成证据必须同时回答：规则对不对、内容能不能到达、表现是否可读、性能是否稳定、发布包是否真的可安装。

## 2. 测试层级

| 层 | 运行频率 | 覆盖 |
| --- | --- | --- |
| 静态/格式 | 每次保存/提交 | 编译、分析器、JSON、Schema、ID、引用、文档层级 |
| 单元 | 每次提交 | 纯函数、数值、组件系统、任务节点、迁移 |
| 契约 | 每次 PR | 包、存档、事件、命令、内容 API、本地化参数 |
| 集成 | 每次 PR/夜间 | 多系统、内容加载、存档往返、Godot 适配 |
| 场景 | 夜间/候选版 | 实际场景、输入、UI、Boss、流式、辅助模式 |
| 回放 | 每次 PR 核心集 | 固定命令流、不同渲染帧率、结果校验和 |
| 性能 | 夜间/里程碑 | 固定硬件路线、帧、内存、加载、GC、包体 |
| 游玩 | 每周/里程碑 | 首玩理解、节奏、深度、关系、难度、可访问性 |
| 安装/升级 | 候选版 | 干净机、升级、回滚、卸载、离线、安全模式 |

## 3. 纯逻辑测试

使用固定 Tick、确定性随机源和内存适配器。每个 Gameplay 系统测试正常、拒绝、边界和顺序变化。断言领域事件和最终状态，不依赖内部实现顺序。

重点属性：

- 同一命令流得到相同状态校验和。
- 重复幂等事件不重复奖励。
- 资源不出现 NaN/无穷/越界。
- 效果叠加与移除满足交换/顺序合同。
- 任务图所有正式节点可达且能终止。
- NPC 日程覆盖结束后回到合法状态。
- 归位事务要么完整提交，要么完全回滚。

## 4. 内容契约测试

每个内容包通过：Schema、ID、跨引用、依赖图、可达性、预算、本地化、资产、确定性烘焙和沙盒加载。错误夹具也纳入测试，确保诊断准确而不是意外崩溃。

当前 `npm run validate:design` 保证 1.0 蓝图下限。F1 后由 `tgd-content` 工具扩展，Node 校验保留为快速入口或迁移到统一 CLI 后提供兼容脚本。

## 5. 存档与迁移测试

- 每个正式 SaveVersion 的黄金档能读取并迁移到当前版本。
- 当前状态保存→加载→保存的语义值稳定。
- Boss 各阶段、任务各节点、NPC 迁移、Cell 切换和地区归位均有往返样本。
- 模拟写入中断、磁盘满、权限拒绝、损坏哈希、缺包和降级。
- 迁移失败保留原档；自动保存轮换可恢复。
- 100 次连续切档和 1000 次状态序列不泄漏持续增长。

## 6. Godot 场景测试

Headless 测试负责场景可加载、节点/资源引用、输入路由、UI 焦点、适配器和基础帧运行。需要视觉/音频判断的测试在可见 runner 或专门实验室执行。

每个地区至少有：启动场景、枢纽、最密战斗、最快流式路线、Boss 重试、归位后状态和 UI 压力场景。场景测试不得依赖作者本地导入缓存。

## 7. 战斗与 Boss 自动化

Combat Lab 用脚本代理执行：

- 基础招式输入和取消窗口。
- 30/60/120/144 FPS 渲染下的固定模拟一致性。
- 各敌人单体、同角色群组、跨角色组合和最坏组合。
- Boss 每条阶段边、失败、重试、证据不足、完整归位、辅助模式。
- 长时间 AI 无动作、无限循环、越界和导航卡死检测。

机器人通关只能证明可达，不证明好玩。每个 Boss 仍需至少 3 类人类玩家盲测：类型熟练者、一般动作玩家、需要辅助选项的玩家。

## 8. UI 与可访问性测试

自动化覆盖焦点图、隐藏控件不可交互、键名动态替换、文本溢出、分辨率矩阵、鼠标/键盘/手柄切换和深层面板暂停。截图差异只比较稳定区域并保留人工审查，避免把像素一致误当体验一致。

人工覆盖 08 的视觉、听觉、操作和认知场景。所有关键声谜有静音替代，所有颜色信息有非颜色通道，所有高频输入有替代。

## 9. 性能测试

固定硬件和驱动镜像，使用可复现路线与采样时长。报告 P50/P95/P99、最差帧、卡顿分类、内存/显存峰值、加载、包体和 Shader 编译。

PR 跑小型预算回归，夜间跑完整地区路线，里程碑在最低/推荐/高档硬件实验室复测。趋势回退超过阈值即失败，即使绝对值尚未越硬门。

## 10. CI 流水线

### PR 快速门（目标 15 分钟内）

1. checkout + 工具版本验证。
2. 格式、文档层级、Schema、ID 和引用。
3. `dotnet build` + 分层分析器。
4. 单元/契约测试与小型回放。
5. 受影响内容包增量烘焙和确定性对比。
6. Headless 冒烟、基础资产加载。
7. 依赖/许可证/秘密扫描。

### 夜间门

全量烘焙、全部集成/场景/回放、NPC 长模拟、Boss 遍历、性能路线、存档矩阵、模组隔离和发布包冒烟。

### 候选版门

签名、SBOM、干净机构建/安装/升级/回滚/卸载、离线启动、各语言、输入与辅助矩阵、完整主线自动可达和人工最终游玩。

## 11. 受影响范围

构建图根据程序集、Schema、包依赖、ID 反向引用、资产清单和测试标签选择快速测试。选择器本身必须保守：无法判断时扩大范围，不漏跑。夜间永远全量，防止图算法错误长期隐藏。

## 12. Flaky 与隔离

测试失败不得自动无限重跑变绿。一次受控重跑只用于收集差异，原失败仍记录。Flaky 测试标记负责人、首次日期、原因假设和修复期限；核心发布门测试不能长期隔离。

测试禁止依赖系统时区、真实时钟、互联网、随机端口、作者目录和执行顺序。确需外部服务使用本地契约模拟，少量真实集成放专用环境。

## 13. 缺陷严重度

| 级别 | 定义 | 发布行为 |
| --- | --- | --- |
| Blocker | 无法启动/通关、存档损坏、安全/隐私严重问题 | 禁止合并/发布 |
| Critical | 核心系统错误、稳定崩溃、重大可访问性阻断 | 候选版前为 0 |
| Major | 明显玩法、任务、性能或表现错误，有可行绕路 | 逐项批准，不能批量忽略 |
| Minor | 不阻断的局部瑕疵 | 记录负责人和目标版本 |

严重度基于玩家影响和发生概率，不基于修复难度。

## 14. 游玩研究

每地区至少三轮：灰盒理解测试、内容节奏测试、近最终盲测。记录任务完成、迷失、失败、提示使用、活动时间分布、NPC 记忆、Boss 理解和情绪反馈。

“每地区 5 小时”使用首玩盲测中位数；开发者熟练速通只作为下界。测试主持人不能现场解释设计。发现玩家普遍不理解时先修体验，不在问卷里证明玩家错了。

## 15. 合并门禁

PR 必须提供：目标里程碑、玩家价值、范围、Schema/迁移影响、测试命令和结果、人工路径、性能差异、截图/录像（确有视觉变化时）、已知风险与回滚。

代码覆盖率是线索而非完成指标。没有断言价值的高覆盖不能替代边界测试；关键状态机和迁移要求分支/案例清单。

## 16. 发布清单

- 所有 Blocker/Critical 为 0。
- 内容锁定，包/可执行/存档兼容矩阵确认。
- 三地区主线、支线和后日常可完成。
- 性能、UI、辅助、本地化和音频门禁通过。
- 安装、升级、回滚、卸载和安全模式通过。
- 签名、SBOM、许可证、隐私文本和第三方通知齐全。
- 发布说明、已知问题、客服/诊断流程和事故负责人就绪。
- 发布产物从受控提交可重建，哈希留档。

发布后先监控崩溃/更新完整性，再开始下个版本的大规模合并。
