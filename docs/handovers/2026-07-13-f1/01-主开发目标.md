## 本阶段交付

### A. 完成奖励原子事务

- App 启动时从已加载 Profile 快照初始化 `ProfileProgressCoordinator`，不建立第二份奖励/Operation 状态。
- Gameplay 解析器仍只产出 `{sourceId, rewardId, rewardDedupKey}` 收据；Presentation 只提交 claim，不判定权益。
- 存储忙时可安全排队一个稳定 claim；非法/冲突 claim 失败关闭并输出可诊断错误。
- claim 生成 Persistent Operation，和更新后的 Profile 快照/Head 在既有 IndexedDB 主事务路径提交。
- 只有存储事务确认后才通知 `F1GrayboxLayer` 显示 persisted/committed；失败时保持 pending/unsafe 语义并允许重试。
- 重复结算、回放、刷新或重试同一 `rewardDedupKey` 不增加 Operation 数、不重复权益。
- 两种结算奖励均经过同一数据驱动路径；不得为具体 reward ID 写分支。

### B. 测试与证据

- 扩展/复用 `tests/native/profile_progress.cpp`、`profile_storage_coordinator.cpp` 与 F1 纵切测试，覆盖成功、重复、非法、分配失败/溢出、存储失败和重试。
- Native MSVC/Clang 受影响测试与 Web Single 构建通过；Node 合同测试保持通过。
- 浏览器至少验证两种结算、Operation 数、commit 后 UI、刷新恢复、重复 claim、离线与一次可重试写失败。
- QA 导出只能观察只读状态；不得成为正式业务入口。
