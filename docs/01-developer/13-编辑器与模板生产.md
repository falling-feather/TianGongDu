# 13｜编辑器与模板生产

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Target Specification；F1 实现最小闭环
> 最后更新：2026-07-16
> 维护者角色：工具/内容技术负责人；内容作者、Runtime、Web、安全与 QA 会签

## 1. 原则

大型 C++ 项目不能要求内容作者反复改头文件、重新编译或手写关卡脚本。编辑器是正式生产系统，每个模板必须支持创建、校验、预览、复制、差异、迁移和批量检查。

主工具是浏览器内容工作台：DOM 适合复杂表单、树、图、搜索和多人审查；C++ ContentCore 同时编译为 Native CLI 与 WASM，保证编辑器、CI 和游戏使用同一 Schema/语义，而不是由 TypeScript 再写一套规则。

## 2. 工具体系

```text
apps/content-workbench/          DOM/TypeScript UI
          │ typed bridge
          ▼
content_core.wasm                C++ 校验、规范化、预览数据编译
          │ same sources
          ├── tgd-content CLI    Native C++，CI/批处理
          └── Game Runtime       读取同一 baked model

Preview Host
├── Axmol Web Preview            角色/战斗/地区可玩预览
└── Native Preview               性能/平台对照
```

具体 DOM 框架在 F1 Tool Spike 决定；框架不能进入内容合同。编辑器可作为本地开发服务器或受控内部站点运行，保存通过本地工作区桥/Git API，不直接修改生产 CDN。

## 3. 模板注册表

权威注册表：[`../../content/templates/template-registry.json`](../../content/templates/template-registry.json)，结构由 [`../../content/schemas/template-registry.schema.json`](../../content/schemas/template-registry.schema.json) 约束。

当前 9 类基础模板：核心 NPC、普通敌人、精英机制战、章节 Boss、地区 Boss、任务链、对话、武器体系、大型子地区。每个模板声明版本、必需/可选模块、生成文件、预览器、验证器和迁移来源。

模板升级展示实例差异；作者选择采纳新默认或保留显式值。结构变化必须有批量迁移，不能静默改全部内容。

## 4. 内容浏览器

按类型、包、地区、阶段、负责人、标签、错误和最近修改筛选。每项显示稳定 ID、显示名、模板版本、拥有包、依赖/被引用、校验和 Git 状态。

关键操作：从模板新建、引用图、比较版本、移动文件保持 ID、弃用/墓碑、生成迁移、打开预览、定位场景/图集。删除前显示存档、同步 Operation 和包影响。

## 5. 角色工作室

### 5.1 新建向导

1. 选择 NPC/敌人/Boss/器物模板。
2. 选择拥有包、地区、阶段和负责人。
3. ContentCore 生成稳定 ID、文件、本地化 key、测试夹具和占位表现引用。
4. 作者配置身份、日程、关系、互动、叙事、能力和地区响应模块。
5. WASM 校验立即运行；通过后打开 Axmol Web 沙盒。

### 5.2 核心 NPC 页签

- Identity：角色作用、职业、组织、显示/资源引用。
- Schedule：日历、天气、章节、锚点和异常覆盖。
- Relationship：三阶段关系弧、维度条件和拒绝/恢复。
- Interactions：指导、交易、工艺、同行、研究、谈判等通用模块。
- Narrative：对话、任务、证据、传闻和天工录。
- World Response：地区归位、选择和跨区变量矩阵。
- Persistence：本地快照字段、持久 Operation 和冲突域。
- Diagnostics：不可达状态、缺本地化、引用、预算和同步风险。

### 5.3 预览

选择时间、天气、关系、任务、地区、Web/Native 平台和模拟网络状态。预览日程、互动、对话、战斗/同行、刷新恢复、本地存档、离线操作和另一设备冲突。

灰盒 NPC 从模板到可预览目标中位时间 ≤30 分钟；这只证明生产链，不代表内容写作/资产完成。

## 6. 战斗实验室

Axmol Web 预览加载角色、武器、能力、敌群与场地档；控制随机种子、渲染 FPS、WASM 单线程/Pthreads、输入脚本和辅助选项。

工具显示逐 Tick 命令、成本、移动、命中窗、效果、取消窗、AI、工灯证据、WASM heap、JS long task、draw call 和资源下载。可导出同一回放到 Windows Native 对照。

内容参数可热重载灰盒；Schema/结构变化重新烘焙。热重载不进入正式玩家版本。

## 7. Boss 阶段图

可视化编辑阶段、条件、招式集、AI 权重、环境、证据、音乐/演出、重试点、存档和归位结果。

静态检查：起点唯一、终点可达、无无界循环、条件不矛盾、每阶段有决策变化、重试快照完整、辅助模式可达、刷新恢复合法、持久 Operation 幂等。

自动生成阶段遍历、失败/刷新/重试、证据不足/完整归位和双端同步夹具。

## 8. 任务与对话图

任务只提供受控节点；拖线时检查类型、持久字段、Operation 语义和可达性。模拟器可随机遍历、重复事件、打乱顺序、页面刷新、离线积压和另一设备分叉。

对话通过领域事件连接任务，不直接写状态。预览支持关系、称谓、文本长度、语音时长、移动/打断和字幕。每条叙事选择标记是否可自动合并；不可合并项必须提供冲突 UI 摘要文案。

## 9. 地区工作台

叠加显示 Cell、加载门、相机、碰撞、地标、任务、NPC 锚点、遭遇、音频、天气、归位变体和资源包边界。

预算实时汇总：首轮时长、活动比例、兴趣点、实体/AI、图集/纹理/音频、WASM heap、下载体积、解压、GPU 上传、缓存占用和同步字段。工作台能模拟冷缓存、弱网、离线和包损坏。

归位变体使用差异层，不复制整个地区。包边界与 Cell 边界协同，避免每进一扇门下载零碎文件。

## 10. 资产/武器工作台

武器按工艺域、部件、架势、能力时间轴与用途权衡编辑。自动生成基础、极端构筑、非法组合、低资源、不同敌材质、Web 单线程和低档设备场景。

资产面板预览图集分页、骨骼/帧动画、压缩、透明过绘、纹理内存和 CDN 包归属。Spine 等受许可证约束的运行时只有在许可清单通过后可选。

## 11. 保存、Undo 与协作

- 所有修改进入 Undo/Redo 事务；跨文件操作原子化。
- ContentCore 输出规范键序/换行/数值，减少无意义 diff。
- 图节点有稳定 node ID；视觉重排不改变语义。
- 保存前验证磁盘/远端版本，冲突显示三方比较。
- 二进制源资产使用 Git LFS/锁；内容 JSON 按对象拆分，避免巨型文件。
- 编辑器绝不把未提交草稿上传玩家云同步服务。

## 12. AI 辅助

AI 只能通过受控命令：按模板生成草案、补测试、解释错误、查重复、生成占位本地化。输入包含最小上下文、允许 ID 和预算；输出先显示差异，不直接保存。

新正史、长期 API、批量覆写、删除、迁移和正式台词必须人工确认。AI 产物仍由人类负责人署名并通过 WASM/Native ContentCore 校验和实际预览。

## 13. Headless/CI

所有 GUI 能力有 CLI 等价：

```text
tgd-content create --template template_core_npc --package region_jiangnan
tgd-content validate --package region_jiangnan --profile ci
tgd-content bake --package region_jiangnan --deterministic
tgd-content test --fixture npc_shen_yan --platform web-single
tgd-content report --budgets --references --localization --web-size
```

真实选项在 F1 实现锁定。CI 对 Native/WASM ContentCore 跑同一黄金夹具并比较规范输出。

## 14. 工具安全

- 工作台只访问用户明确打开的工作区；浏览器版不获得任意磁盘路径。
- 文件系统桥校验规范路径，拒绝逃逸/符号链接越界。
- 预览内容在隔离 origin/Worker 或受控本地环境，不执行项目 JavaScript/C++ 插件。
- 内部站点也需要身份、CSRF/CSP 和最小权限；不连接玩家生产数据库。
- 大文件、深图和恶意文本有尺寸/复杂度上限。

## 15. 质量门禁

- 5000 个对象下搜索、增量校验和引用图无明显阻塞。
- 崩溃/刷新不损坏源文件；可恢复未保存草稿。
- 新作者 90 分钟内创建并预览角色、任务和简单 Boss。
- Native CLI、ContentCore WASM 和游戏加载得到同一规范模型。
- 预览可切 Web 单线程/Pthreads/Native 和网络/存储故障。
- Undo、外部修改、模板迁移、图冲突和浏览器多标签有自动测试。
- 工具升级可打开上一个正式版本全部内容。

“作者可用”由内容团队盲测，不由工具作者自评。

## 16. F1 Workbench 最小范围

F1 只做足以证明“内容不靠改 C++ 生产”的闭环：

- 内容浏览器：按 ID/类型/包/阶段搜索，显示引用和校验错误。
- 核心 NPC 表单：创建 `npc_shen_yan` 夹具或同结构副本，编辑身份、简化日程、关系阶段、交互和世界响应。
- Boss 阶段图：创建四时伞祟四阶段、证据、重试和两类结算，运行可达检查。
- 任务最小图：承载“雨夜试伞”的接受、两证据、Boss、结果和恢复，不在 GUI 中实现全部未来节点美化。
- Native/WASM ContentCore 同一诊断；确定性保存、Undo/Redo、外部修改提示。
- 一键烘焙 F1 包并在 Axmol Web Preview 加载；导出黄金 fixture/诊断报告。

F1 明确不做：多人实时协作、生产 CDN 发布、完整 DCC 编辑、语音时间轴、所有地区地图编辑、任意插件市场、自动写正史或浏览器直接提交 Stable。缺少的目标仍保留在本文件，但不得成为 G3 前端“大平台”延期理由。

## 17. 工作区桥合同

浏览器 UI 本身无任意磁盘权限。Local Workbench 通过受控本机桥处理：列出项目根内允许目录、读取带 revision/hash 的文本、原子写入、批量事务、文件监听和 Git 状态；桥不执行内容提供的命令。

- 启动时用户明确选择仓库根，桥解析真实路径并拒绝根外、符号链接逃逸、设备路径和绝对引用。
- 每次写带 `expectedContentHash`；外部变化返回冲突，UI 做三方比较。
- 跨文件保存使用可恢复 journal 两阶段流程：暂存全部新文件与旧 hash → 校验 → 写 `prepare` marker → 逐文件同卷替换 → 写 `commit` marker → 清理；崩溃启动时按 journal 完成或回滚。普通 Windows/文件系统不宣称整个多文件集合具有单次原子替换。
- 生成产物写到可清理 build/cache，不与 Source 混淆；可再生二进制不手工提交。
- ID 生成、墓碑、迁移和批量重命名有审计记录/预览，删除不能只调用文件系统 remove。
- 内部站点模式使用项目 Git 服务 adapter，也遵守同一 ContentCore/事务合同；不直接连接玩家 API/DB。

框架/OAuth/本机 IPC 方案属于 OQ-106，G3 锁定并做威胁模型。

## 18. 模板实例完成度显示

工具同时显示两个独立维度：`LifecycleStage`（Pitch/Contract/Greybox/Content/Polish/Shipped）与 `ValidationState`（Unknown/Invalid/Valid/ValidatedInBuild）。Schema Valid 只代表结构，不能自动把内容升为 Greybox；实际预览/游玩证据带 build ID 回写索引。

每个实例页显示 DoR 缺项、预算、Owner、引用、迁移版本、最后验证构建和陈旧原因。复制模板永远生成新 ContentId/节点 ID/claim namespace；复制显示文本或资产引用不复制身份。

## 19. 系统型 Demo 的轻量关卡编辑器

`F1-SYSTEM-DEMO-01` 先实现能制作当前战斗 sandbox 的最小工具，不直接建设第 3–10 节描述的完整商业 Workbench。当前 `apps/content-workbench/` 已有 owning author draft、封闭形状归一化、确定性 reducer、revision/dirty/last-valid 与 Node 证据；它仍没有可视化 UI、工作区 I/O、compiler/export bridge 或 Preview。第 19.6 节只记录这部分已实现基础，其余本节内容仍是 `TOOLS-001` 的待实现目标，不能整体标记为可用编辑器。

### 19.1 第一版对象与属性

第一版只需要六类可摆放对象：玩家起点、敌人/刷新点、地面障碍、互动点、机关和安全点。所有对象必须有稳定 ID、位置、方向、范围、区域、资源引用和类型专属字段；敌人还要能引用职责、波次、触发和出生顺序，互动/机关要能引用受控的交互种类与目标条件。工具不得把显示名、数组下标或 DOM 节点当运行时身份。

场地画布、对象树和属性面板编辑同一份内存草稿。拖动只提交量化后的坐标；保存前显示规范化差异。第一版允许 2D 顶视/斜向平面预览，不要求 DCC、完整镜头或最终渲染器嵌入。

### 19.2 保存、导出和试玩

- 打开时校验 schema 版本、内容 hash 和工作区边界；未知大版本、损坏 JSON、根外路径和符号链接逃逸失败关闭。
- 保存采用预期 hash + 临时文件 + 同卷替换；多文件导出使用第 17 节 journal。刷新或崩溃后可恢复草稿，但草稿不自动覆盖源文件。
- 导出必须确定：相同源数据生成相同规范 JSON、诊断顺序和内容指纹。生成缓存与源文件分开。
- “启动试玩”只调用受控的已登记构建/启动配置，把导出包交给现有 Web/Windows Preview；内容文本不得携带任意命令。
- 快速重载第一版可以是显式“重新导出并重载”按钮，不要求玩家构建自动监听。结构/schema 变化必须重启会话；纯位置、波次和允许的数值变化才可快速重载。

### 19.3 第一版诊断

保存和试玩前至少拒绝：零/重复/非法 ID、坐标或范围越界、玩家起点/安全点缺失、对象区域无效、波次引用缺失、触发循环、资源 ID 未登记、阻挡玩家起点或安全点、必经交互不可达。断路检查使用明确的离散地面网格/探针并报告阻断对象与起终点；第一版不宣称解决通用导航网格。

诊断由共享工具合同产生稳定 code、severity、对象 ID 和字段路径；UI 只排序和呈现。TypeScript 不复制一套“差不多”的 Gameplay/ContentCore 规则。

### 19.4 所有权与第一周门

关卡工具组拥有 `apps/content-workbench/`、工具专用合同、桥、导出器和测试。Gameplay/Runtime 拥有可加载的 sandbox Definition 与玩法不变量；内容组拥有具体场地实例；UI 组只消费工具 view model；QA 拥有往返与启动证据。共享 Schema 变更先由主开发冻结，禁止工具组直接改 F1 正史 Definition 来迁就界面。

第一周通过条件：作者能从示例 sandbox 新建/移动至少一个对象，配置一组敌人波次，保存后重新打开得到相同规范内容，导出并启动 Web 与 Windows 试玩；重复 ID、缺资源、越界和阻断安全点各有负例；修改位置或波次后可显式重载。Undo/Redo、多人协作、Boss 图、对话图、发布 CDN 和大型内容浏览器留在后续任务。

### 19.5 动态 Sandbox 与开发预览包

作者源使用规范化 `*.sandbox.json`，Runtime 输入使用独立 `.tgdsbx v1` 开发预览包。`.tgdsbx` 采用显式 little-endian、有界 section directory、稳定 ID 名称与 key 双校验、总包 SHA-256 和最多 4 MiB 容量；它不签名、不压缩、不携带资产二进制，只引用 Stable Asset ID。它既不替代正式 `.tgdpack`，也不进入 Profile/奖励事务。

当前发布 `.tgdsbx format 1.1` 的唯一 ContentCore producer、decoder、package validator 与 owning document。1–13 package-core sections 保持原编号/语义，append-only 14/15 保存已冻结的 interaction/mechanism gameplay binding；两节为空也必须存在。精确 Header/Directory/容量与兼容策略见[内容、存档与版本契约 §21](12-内容存档与版本契约.md#21-tgdsbx-v1-package-core-公共合同)。Gameplay 已有消费已校验内存 Definition/binding 的独立 `SandboxSession` Bootstrap，但尚无 package document/provider→Session adapter，也没有 Preview；工具必须调用唯一 producer，不得自行编码第二份 `.tgdsbx` 真相。

`SandboxPackageDocument` 独占字符串池、1–15 全部数组及两个只读 view，使输入字节可在 decode 返回后释放。compiler/provider 在消费只读 view 时必须保证 document 存活；`SandboxSession::initialize` 会把自身拥有的字段复制进固定容量状态，之后不得保留输入 span、Definition 指针或 document 地址。CollisionWorld 与 Presentation prepared assets 的复制/持有边界仍须显式发布。不得扩展只返回 F1 静态裸指针的 provider，也不得给要求 60 分钟的 `VerticalSliceDefinition` 增加“短 Demo”特例。

当前 wire 明确不含 gameplay profile、player runtime identity、loadout、skill/slot/cooldown、敌人 faction/duty/AI、mutable state 或技能表现引用。1–13 中的 interaction、mechanism 与 safe-point 都只是 placement，不隐式拥有 operate、one-shot、retry 玩法，也不能从 base section 直接产出权威事件；14/15 仅编码已冻结的 typed one-shot binding。

producer/decoder 共用唯一 package validator，并复用公开 gameplay binding validator；缺失、重复、悬空、kind-domain 错误、图断路或被 blocker 覆盖的起点/安全点都失败关闭且不返回 partial document。离散地面路径探针与必经 interaction 可达性仍是后续 compiler/save-preview gate。当前 standalone Session 只消费 typed one-shot interaction 链和本地 initial-safe-point retry；package-document adapter、player movement/Actor Runtime、safe-point policy、Actor/profile/loadout、Objective/Wave/Encounter/Combat 聚合会话仍属后续 GAME/Runtime 接线。

快速重载第一版只做“完整读取→校验→构建候选 Session/Collision/Asset Set→安全边界整体替换”。成功后 generation 加一并从沙盒起点开始；失败、stale hash 或缺资源时保留上一份有效 package、Session、checksum 和画面。首周不做运行中字段热补丁，也不宣称完成跨版本战斗状态迁移。

### 19.6 Sandbox authoring foundation 1.1.0

`TOOLS-001.2` 只建立轻量编辑器的 owning author draft 与确定性状态内核。它与冻结的 `.tgdsbx format 1.1`、Content API 1.0 边界对齐，但不实现 wire。作者 JSON 的 `format` 固定为 `tgd.sandbox.authoring`，`schemaVersion` 固定为 `1.1.0`；顶层只有 `runtime` 与 `editor` 两个所有权域。`runtime` 覆盖当前 package-core 13 sections 的逻辑字段以及必需的 interaction/mechanism gameplay binding sections 14/15；`editor` 独占 label、ordinal、canvasX、canvasY。runtime projection 会完整移除 `editor`，但该 projection 不是 `.tgdsbx`、不是 producer 输入承诺，也不参与 canonical package bytes/hash。

首版采用 closed shape：缺失或未知字段、未知格式/版本/枚举、错误 primitive、非安全整数、越过公开存储范围与容量均失败关闭，不做字符串到数值等 coercion。迁移入口只接受 `1.1.0` identity；不存在 `1.0.0` 隐式升级，未来版本必须登记显式逐版本迁移。普通集合按 ID、wave spawn 按 waveId/spawnOrder/actorId、binding 按 owner ID、编辑器项按 ordinal/ID 稳定排序，保存为固定键序与结尾换行的 JSON。

编辑状态只接受 `entity.add`、`entity.update`、`entity.delete`、`document.import`、`document.mark_saved`。每条命令携带 expected revision；真实文档变化才增加 revision，mark-saved 只接受当前 revision。stale 或结构非法操作不会替换 document、revision 或 lastValidDocument；其中 lastValidDocument 仅表示“最后一个通过 JavaScript author shape 的草稿”，绝不表示已通过 DEV validator、可导出或可玩。

JavaScript 结构门只是作者输入的非权威早期反馈。stable ID key/hash、引用存在与 kind/domain、图环与可达性、asset、出生/安全点、typed writer、系统 Demo 非空玩法链及其他 Gameplay/ContentCore 规则全部交给唯一 package validator；正式导出仍必须调用该 validator 和唯一 producer。当前 **TOOLS authoring 切片**不包含 UI、workspace I/O、compiler、`.tgdsbx` bridge 或 Preview，也没有调用根仓库已存在的 standalone `SandboxSession`；结构合法的 fixture 仍不得称为可导出或可玩包。

Node 门由根脚本 `npm run test:content-workbench` 接入 `npm test`，覆盖严格 round-trip、乱序归一化、editor 不泄漏、owning deep copy/freeze、revision/dirty/last-valid 以及 closed-shape 负例。

### 19.7 真实 Preview 边界

当前 `apps/native-shell/` 只有 `.gitkeep`；MSVC/Clang CTest 是 Native GameCore 证据，不是 Windows 玩家程序。`PLATFORM-001` 必须建立真实可见 Windows Axmol 窗口与隔离 Web sandbox，并让两端消费同一个 `.tgdsbx`、Gameplay Session 和只读 Presentation Layer。Platform 只运输文件/fetch 字节、管理窗口生命周期和 generation，不能解析波次、技能、任务或奖励。

现有 `F1GrayboxLayer` 只能作为回归与宿主节奏参考；不得复制其中的 F1 ID、硬编码 blocker、键位、最近敌人选择、任务文案、Profile、IndexedDB 或奖励接线。系统 Demo 输入必须消费 Action Registry；Web sandbox 不注册 F1 Service Worker，不打开 F1 Profile 数据库。若首周第三天仍没有真实 Windows target，Windows 门保持 Open，禁止用命令行测试冒充完成。
