当前本地 Web 构建已能直观看到代码绘制的雨夜斜向全景，并用 `WASD/方向键`、`F`、`Space`、`J`、`K`、`Shift`、`C`、`1/2` 连续操作全部 7 Beat：雨渡交互、沈砚训练、伞巷失败重试/读招清场、棚缝检查/积水疏通/纸鹭诱饵、纸鹭战与路线选择、春冬工痕与共享账簿调查、春/冬肋调校、返程三角组→校准架→纸鹭叠加增援→清场/捷径、四时伞祟春夏秋冬四阶段，以及直接制伏/恢复共同工印两种结算选择和返沈砚交付。返程入口为 3 名 Active 敌人，校准 Objective 完成后增加至 4 名；`reinforce` 保留玩家和既有敌人的资源/位姿，倒地重试会重建已完成 Objective 对应的叠加状态。Boss 作为初始休眠 Actor 由内容节拍单独激活，按 75/50/25/0 生命阈值依序完成任务目标并切换四套轻重招式；季相光环、大型轮廓、血量/姿态 HUD 和攻击预警只读取 C++ 快照。结算完成后解析器只返回稳定奖励 ID 与 `rewardDedupKey` 收据，Web 明示“SAVE RESERVED”，不会把尚未执行的 Profile 原子事务显示成已发奖/已保存。玩家生命归零后仍经独立 `SafePointRetryCommand` 恢复；7 个 authored safe point 已随 Beat 提交，首次自然死亡经浏览器验证从伞巷 `(-5600,-1200)` 恢复，后续工位/Boss/结算安全点也进入只读快照。它们当前只覆盖移动位姿与既有任务进度，尚不是把战斗 Actor 资源、任务、奖励、世界与 Profile 一起原子提交/恢复的跨域事务快照。HUD 的 `AUDIT Q / IDLE / RETRY / TARGETS` 只读同一 C++ 审计快照；移动、纵跃、已接受战斗事件和任务进度会续期活动窗口，无活动 Tick 单列为闲置，失败尝试不会保留在合格桶。这个机制拒绝挂机、失败和跨 Beat 补时，但仍不能判断移动绕圈、文本质量或内容是否真正非重复，因此最终验收仍必须消费人工盲测记录。入口产物为 `build/web-release-single/dist/web/tiangongdu-f1.html`（必须经本地 HTTP 服务访问）；这一入口已有七段功能路径但仍不满足上表 60 分钟流程，真实内容密度/长时资源循环/盲测、多单位增援与职责/令牌调平、正式鼠标/手柄动作适配、奖励持久事务、完整 AI/导航/自主脱战和跨域事务快照均不能跳过。

沈砚训练现通过 5 个内容驱动触发器验证两套架势的核心动作，而非把按键本身当任务进度：重击 Ability 起手、格挡结果、姿态切换、轻击 Ability 起手和闪避结果必须按 Objective 前置链依次发生；格挡完成还会在同一 Beat 把伞偶构件完整替换为纸鹭构件。两轮链只提高操作教学与内容合同密度，不改变 8 分钟目标；当前自动功能路线仍会快速通过，因此还需加入空间推进、多单位/复合条件波次、资源压力和叙事反馈，并由真人盲测证明时长与非重复性。

伞巷雨障处理同样由 `QuestInteractionDefinition` 的稳定 ID、位置、半径和 Objective 前置驱动；Presentation 只画交互锚点和反馈。当前三步会改变任务与敌群激活时机，但尚未写入可保存的世界机关状态，也没有水流模拟、失败替代、多个解法或 NPC 反应；后续扩充必须继续走 `IQuestRuntime` / 内容 Definition，并在引入可持久世界变化时通过 `ISnapshotContributor` 与 Persistent Operation 进入跨域事务，不能在 Web Shell 里追加第二份谜题状态。

## 4. F1 Web 烘焙任务包

### 4.1 F1 硬门

下列链路任一失败都不能进入 M1：

1. C++20 `contracts/runtime/gameplay/presentation/platform` 依赖边界由 CMake 检查。
2. Axmol 2.11.4 LTS 经 Emscripten 生成 Web Single wasm32 + loader，在 WebGL2 下稳定启动、暂停、恢复和销毁。
3. Canvas 仅绘制游戏画面；DOM HUD/菜单能用键盘、手柄、触控模拟和辅助技术操作。
4. “雨夜试伞”含探索、NPC、对话、采集/工艺、普通战斗、Boss、失败重试和结算。
5. 内容按哈希清单分包，首屏只下载壳、WASM、基础 UI 和首段内容；地区资源按需加载。
6. Profile 通过直接异步 IndexedDB 主路径事务保存；刷新、崩溃、离线和浏览器重启可恢复，IDBFS 不作为第二份真相。
7. Native GameCore 测试宿主运行同一自动回放，离散 Gameplay 结果校验和一致；完整 Windows/Axmol 应用留给 F1N。
8. 内容工作台从模板创建 NPC/Boss/任务，并用同一 C++ ContentCore 的 Native/WASM 版本校验与烘焙。

### 4.2 F1 条件/决策 Spike

下列事项必须在 G3 提交实验、风险和书面结论，但“某个增强方案未达到上线门”本身不阻断已经成立的 Web Single + 本地档纵切：

- **Pthreads**：构建最小代表样片，测量收益、COOP/COEP、独立 origin/CDN/OIDC 影响。无显著收益或部署风险过高时，决定 Deferred，Stable 继续 Single。
- **云同步**：锁 Sync IDL/权威矩阵，用 mock 身份验证 Operation 幂等、双端分叉和叙事冲突原型。生产 OIDC、完整服务 SLO 或正式部署可延期，但本地操作日志/导出不能延期。
- **Windows Native 宿主**：G3 只要求纯核心/回放对照；完整 Axmol 窗口、安装和平台体验由并行子门 F1N 放行，必须在 M1 扩产前完成，但不反向阻塞 F1 Web Single 的纵切结论。
- **Safari/移动/PWA 安装**：按第 7 节记录能力和失败原因，不阻断桌面必选矩阵；不得因此宣称这些平台已支持。

若条件 Spike 暴露会破坏既有硬门的基础问题（例如相同存档协议无法表示冲突、Platform 分层无法 Native 编译），该基础问题仍然阻断；可延期的是增强实现，不是合同正确性。

## 9. 功能完成定义

每项功能需提供八类证据：

1. **玩家价值**：解决什么体验问题。
2. **C++ 契约**：API、单位、所有权、稳定 ID 和错误语义。
3. **跨平台**：Web 单线程、Native 纯核心；相关时再含 Pthreads/移动端。
4. **生命周期**：启动、页面隐藏/恢复、刷新、离线、取消和退出。
5. **存档/同步**：本地事务、迁移、幂等、离线积压与冲突。
6. **内容证据**：教程、正式用例和变式/压力用例。
7. **测试/性能**：自动化、人工游玩、下载、内存、帧和弱网。
8. **维护**：文档、编辑器、日志、缓存失效与回滚。

缺任一适用证据只能标记“实现中”或“验证中”。

## 10. 内容完成定义

角色、Boss、子地区和任务仍遵循上一版深度门禁，并增加 Web/同步要求：

- 核心 NPC 的日程、关系和任务在刷新、离线积压和另一设备恢复后不死锁。
- Boss 的失败重试快照可写 IndexedDB，页面异常关闭后回到合法安全点。
- 子地区资源包可独立缓存、校验、清理和重新下载，不破坏持久状态。
- 任务奖励使用幂等 operation ID，重复同步不重复发放。
- 叙事不可自动合并冲突必须保留两份证据并让玩家明确选择。

## 11. Web 性能硬门

F1 起始目标，最终由真实设备校准：

- 推荐桌面 1080p 60 FPS；低档可锁 30 FPS，但规则仍为 60 Hz。
- 单线程 Web 构建 P95 主帧不超过 16.7 ms；页面后台不继续追赶无限 Tick。
- 冷启动首个可交互画面在 20 Mbps/常规延迟下目标 ≤ 8 秒，回访缓存目标 ≤ 3 秒。
- Brotli 后 WASM 目标 ≤ 12 MiB；首段必需资源总传输目标 ≤ 35 MiB。
- 活跃内存目标 ≤ 512 MiB，硬门 ≤ 768 MiB；长时游玩回到枢纽后应明显回落。
- 主线程不出现 >100 ms 的资源解析/解压峰值；内容包异步下载、分段解析和预算提交。

不得用“原生很快”推断 WASM、浏览器缓存或移动设备表现。

## 14. 计划变更规则

- 改变 C++、Web 首发、Axmol/Emscripten、DOM UI、本地存档或同步模型必须写新 ADR。
- 平台可延后，但不能在未评估时宣称“多端已经支持”。
- 内容数量下调必须同步 00、02、07 和目录 JSON。
- 完成事实移入 03；02 只保留当前状态、风险和下一门禁。
