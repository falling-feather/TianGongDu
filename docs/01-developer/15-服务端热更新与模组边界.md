# 15｜服务端、云同步、更新与模组边界

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Accepted Boundary；同步服务在 F1 Spike 验证
> 最后更新：2026-07-10
> 维护者角色：后端负责人；Sync Contracts、Web、数据、安全、运维与 QA 会签

## 1. 产品边界

1.0 是浏览器首发的单人动作 RPG，不是 MMO。账号和云同步用于跨浏览器/设备接续、备份和未来原生多端，不进入每帧战斗权威。

承诺：访客无需注册即可开始；网络中断时已缓存内容和本地档继续；同步服务故障不回滚本地已保存进度。跨端接续是可选增强，但一旦开启必须可解释、可撤销、可导出。

## 2. 服务组成

```text
Static Web/CDN
├── HTML/JS/Service Worker/WASM
├── release manifests
└── hashed .tgdpack bundles

Identity Provider (OIDC)
└── authorization; no game passwords

Sync API (C++20/Drogon)
├── session exchange
├── profiles/devices
├── snapshots/operations/conflicts
├── export/delete
└── update manifest metadata (optional)

PostgreSQL
├── identity mapping/profile heads
├── revisions/operation dedupe
├── conflict/audit metadata
└── migration/version tables

Object Storage (when needed)
└── encrypted/compressed large snapshots and exports
```

Redis、消息队列和微服务不预装。只有压测、异步导出或运维证据证明需要时加入。

## 3. 身份与会话

- 本地 Guest Profile 由随机 `profileId/deviceId` 标识，不等于安全账号。
- Web Shell 使用外部 OIDC 提供方推荐的 Authorization Code + PKCE/安全 Cookie 方案；不自建密码数据库。
- Sync API 将外部 subject 映射项目 user，签发短期项目会话；刷新/撤销策略经安全评审。
- 多 Profile、设备重命名、设备撤销、账号退出和数据导出/删除是正式功能。
- 链接账号前显示本地/云档摘要；双方有进度时先合并预览，不自动覆盖。
- Token 不写普通游戏存档，不记录日志，不放可被任意脚本读取的长期 `localStorage`。

## 4. 同步 API 草案

```text
POST /v1/session/exchange
GET  /v1/profiles
POST /v1/profiles/{profileId}/adopt
GET  /v1/profiles/{profileId}/head
POST /v1/profiles/{profileId}/sync
GET  /v1/profiles/{profileId}/snapshot/{revision}
POST /v1/profiles/{profileId}/conflicts/{conflictId}/resolve
POST /v1/profiles/{profileId}/export
DELETE /v1/devices/{deviceId}
```

`/sync` 请求含 `requestId`, `deviceId`, `baseRevision`, `clientSchema`, `packageSet`, `snapshotHash`, `operations[]`。响应含新 revision、accepted/rejected operation IDs、server operations、conflicts、迁移/更新要求和配额状态。

所有写请求幂等；相同 request ID 返回相同语义结果。API 用稳定错误码，不让客户端解析英文消息。

## 5. PostgreSQL 事务模型

核心表概念：`users`, `profiles`, `devices`, `profile_revisions`, `operations`, `reward_dedup_keys`, `entitlement_grants`, `conflicts`, `snapshots`, `schema_migrations`, `audit_events`。

一次同步在事务中：锁定/比较 Profile head → 去重 request/operations → 验证版本与前置 → 应用领域合并 → 写新 revision/冲突 → 更新 head → 提交。Snapshot Blob 可事务写临时对象后再以哈希引用，失败不产生悬空 head。

Operation 表按 profile/revision 分区或归档策略需压测后确定。审计不保存不必要的完整叙事文本/个人信息。

## 6. 领域合并

详细规则见 12。服务端不得只做 JSON merge 或 Last-Write-Wins。

- 集合解锁：并集 + 前置验证。
- 库存/货币：唯一 Operation/RewardDedupKey 账本，不能比较两个最终数字。
- 任务：按状态机偏序寻找合法共同后继。
- 关系：按来源 Operation 重放；互斥里程碑转冲突。
- 地区归位、关键叙事：不自动覆盖，返回双方分支摘要。
- 冲突解决可选择 A、B 或复制为两个 Profile；不会删除未选分支直到保留期结束。

客户端时间只用于展示，不能决定胜者。服务端 revision 和 Operation 因果前置决定顺序。

## 7. 本地优先同步流程

1. Gameplay 完成持久事务并生成 Operation。
2. IndexedDB 原子提交 Snapshot/Operation/Head。
3. UI 显示“已保存在此设备”；云状态单独显示。
4. 网络可用且已登录时批量同步。
5. 成功后本地更新 cloud head；失败保留队列并退避。
6. 409/conflict 下载差异，在 DOM 面板解释来源、设备和影响。
7. 解决操作先本地提交，再上传；任何时候可导出完整本地档。

不能在网络请求返回前阻塞 Boss 结算/关卡退出，也不能显示一个模糊云图标让玩家猜是否已保存。

## 8. 内容发布与热更新

Web 代码天然通过站点部署更新，但仍需要版本纪律：

- HTML/Service Worker 短缓存并检查新 release manifest。
- 哈希 JS/WASM/`.tgdpack` 使用不可变长缓存。
- Service Worker 先完整下载/校验新最小集合，等待安全激活。
- 已运行 Session 不热换 GameCore/Gameplay/地区规则；标题页或明确重启切版本。
- 同一存档记录 Executable/WebShell/ContentApi/PackageSet，旧标签页不会写入新格式而不迁移。
- CDN 回滚指向上一签名 manifest；旧资源保留到回滚窗口结束。

内容包无 C++/WASM/JavaScript 可执行代码。新增玩法代码发布新 WASM/Native 可执行版本。

## 9. PWA 与离线

Service Worker 预缓存壳、WASM、Boot Pack；地区包在进入/空闲时缓存。离线界面列出已可用地区和云同步待处理数量。

Service Worker 可能在任意时刻终止，所有缓存操作用完整 Promise/`waitUntil`，不把内存变量当事务。旧 Service Worker、多个标签、刷新中更新和 Cache Storage 配额必须测试。

PWA 安装不是强制；普通标签页拥有同等游戏内容。卸载/清站点数据的风险在设置和同步页清楚说明。

## 10. 多端计划

- Web Desktop：1.0 权威发布目标。
- Windows Native：F1N 对照，M1 后 Beta；复用 `.tgdpack`、存档、同步和 Gameplay。
- Android：触控、内存、后台、商店和账号门通过后 Beta。
- iOS：同上并完成平台/支付/浏览器引擎政策审查。

每端是同一 Profile 的设备，不复制账号。设备能力/设置本地保存；进度通过领域 Operation 同步。

## 11. 模组边界

Web 1.0 默认不提供任意本地模组执行。未来允许的扩展：

- 独立命名空间的数据、对话、任务、外观和受预算资源。
- 公开能力/效果/任务节点和 UI 插槽。
- 仅本地 Profile 或明确标记的 Modded Profile。

禁止：

- 加载任意 C++/WASM/JavaScript、原生库或脚本解释器。
- 覆盖官方 ID、WASM、Service Worker、签名和 Profile head。
- 网络访问、任意文件/IndexedDB origin 访问、收集个人信息。
- 将 Modded Operation 上传普通官方 Profile。
- 绕过资源/图深度/解压预算。

原生端也不能因为能访问文件系统就放宽代码执行。未来代码模组需独立沙箱 ADR。

## 12. 安全与滥用

- 所有请求鉴权、授权、大小/速率/复杂度限制。
- Operation payload 按版本/领域白名单解析；不反序列化任意 C++ 类型。
- 服务端验证任务/奖励/库存基本不变量；异常 Profile 隔离，不直接封禁或删除。
- 快照压缩防炸弹，Blob 哈希、配额和保留策略。
- OIDC token audience/issuer/nonce/PKCE 校验由成熟库/提供方流程完成。
- CORS 只允许正式 origin；CSRF/Cookie 策略、CSP、COOP/COEP 共同评审。
- 用户可导出/删除数据；备份和审计的延迟删除政策透明。

## 13. 隐私

云同步最少存：外部身份映射、Profile/设备、游戏进度、版本和必要审计。不同意云同步的 Guest 不上传 Profile。遥测与云同步分别征得同意，不能捆绑。

禁止默认上传输入文本、完整日志、模组文件、个人路径和无关浏览器指纹。设备 ID 是项目随机 ID，不使用硬件序列号。

## 14. 运维与灾备

- PostgreSQL 自动备份、时间点恢复、定期恢复演练。
- Snapshot 对象存储版本/生命周期与数据库 head 一致性检查。
- 多版本服务滚动部署先向后兼容，数据库迁移 expand→migrate→contract。
- 监控同步成功/延迟/冲突/重复/迁移/配额，不记录敏感 payload。
- 服务故障时客户端进入离线队列；恢复后限速排空，避免惊群。
- 数据损坏先冻结相关 revision、保留证据和用户导出，再修复。

## 15. F1 Sync 验收

- Guest 完整游玩并刷新恢复，无账号强制。
- Guest 链接 OIDC，正确处理本地有档/云端有档/双方有档。
- 两设备从同一 revision 离线推进，再上线：可合并域自动合并，叙事冲突不丢分支。
- 同一请求/Operation 重放 100 次只生效一次。
- 服务 5xx/超时/429/断网不破坏本地档。
- 旧 Schema 客户端、新服务和数据库滚动迁移可共存。
- Profile 导出/删除、设备撤销和 token 过期可测试。
- 备份恢复到隔离环境后 revision/hash 一致。

新决策见 [`adr/ADR-0008-本地优先与多端云同步.md`](adr/ADR-0008-本地优先与多端云同步.md)。

## 16. 普通单机进度与官方权益的权威矩阵

离线单机客户端有能力修改自身玩法状态，服务端不可能仅靠上传的领域 Operation 证明每一次战斗是否合法。1.0 不做竞技排行榜、PVP 反作弊或强权威游戏服，因此明确区分：

| 领域 | 权威来源 | 服务端行为 | 异常处理 |
| --- | --- | --- | --- |
| 普通剧情、关系、地区选择、单机材料/货币 | 玩家 Profile；本地先提交 | 校验结构、版本、幂等、引用和基本不变量；用于备份/合并 | 保留/隔离异常分支并允许导出，不因“数值可疑”删档封号 |
| 唯一任务奖励 | 本地 GameCore `RewardDedupKey` + operationId | 去重并校验来源 ID/一次性关系；不重放奖励 | 重复 key 拒绝但保留已有合法 revision |
| 账号拥有权、付费/IAP、发行平台权益 | OIDC/商店/项目服务端签发 `EntitlementGrant` | 只接受服务端/平台验签结果，客户端最终值无效 | 隔离伪造权益；不连带删除普通单机档 |
| 官方活动/未来联机权益 | 未来专门服务合同 | 不能复用“普通云存档”假装权威 | 进入新 ADR；不在 1.0 默认范围 |
| Modded Profile | 本地受限扩展命名空间 | 不进入普通官方 Profile/权益 | 可导出/本地使用，清楚标识来源 |

“防止损坏另一设备档”依靠版本、Schema、Operation 幂等和冲突保留；不是通过秘密客户端算法监控玩家。商业模式未决定前不得预埋付费奖励逻辑。

## 17. Sync API v1 错误包络

G3 前以 OpenAPI 3.1 或等价 IDL 固化完整 body、auth scope、限制和生成类型。所有非 2xx 响应至少返回：

```json
{
  "error": {
    "code": "protocol_upgrade_required",
    "messageKey": "sync.error.protocol_upgrade_required",
    "args": {},
    "retryable": false,
    "retryAfterSeconds": null,
    "correlationId": "...",
    "detailsVersion": 1,
    "details": {}
  }
}
```

| HTTP | 稳定 code 示例 | 客户端行为 |
| ---: | --- | --- |
| 400 | `malformed_request`, `unknown_field` | 不重试；记录契约错误 |
| 401 | `session_required`, `session_expired` | 尝试一次安全刷新/重新登录；本地继续 |
| 403 | `profile_forbidden`, `device_revoked` | 停止该 Profile 同步并解释账号/设备 |
| 404 | `profile_not_found`, `revision_not_found` | 重新拉 Profile 列表；不新建同 ID 覆盖 |
| 409 | `profile_adoption_conflict`, `conflict_already_resolved` | 请求级零写入；刷新 ProfileAdoption 或冲突解决页面 |
| 412 | `precondition_failed`, `request_id_body_mismatch` | 不自动重试；修正基线/调用错误 |
| 413 | `payload_too_large`, `too_many_operations` | 分批/压缩；不丢本地队列 |
| 422 | `request_envelope_invalid`, `batch_contract_invalid` | 请求级零写入；修正 envelope/IDL 后再提交 |
| 426 | `protocol_upgrade_required` | 要求更新或只读导出 |
| 429 | `rate_limited` | 使用 `Retry-After` + 抖动，不阻塞本地 |
| 500/503 | `sync_unavailable`, `storage_unavailable` | outcome unknown：同 requestId 重试/查 head；本地保持未确认队列 |

`POST /profiles/{id}/sync` 只要鉴权、版本和请求 envelope 可处理，就返回 200，并在 body 中逐项给出 `accepted/rejected/conflicts` 与事务后的 `headRevision`；`operation_invalid`、`transition_invalid`、`unsupported_domain` 是 200 响应内单项 `rejected.code`，领域分叉是 `conflicts[]`，都不是 HTTP 409/422。表内确定性 4xx（如 409/412/413/422/426/429）表示请求级失败且不写 Profile head、Operation、RewardDedupKey、EntitlementGrant 或 conflict。连接中断、代理超时或 5xx 可能发生在数据库 commit 后、响应前，outcome unknown；客户端必须用同 requestId 重试或读取 head，不能猜测零写入。

同一 `requestId` + 相同规范 body hash 返回相同语义结果；同 ID 不同 body 返回 412。Operation ID、requestId/bodyHash/终态 revision 的轻量 tombstone 至少保留 Profile 生命周期；完整响应 body 起始缓存 30 天，过期后由 tombstone、Operation 与 conflict 记录重建等价结果，不重复应用。分页、快照下载和导出使用有界 page/cursor/短期 URL；限制值进入 IDL 和响应，而非只藏在代理配置。

JSON wire 不用 JS `number` 承载 64-bit revision/sequence/size：`cloudRevision`, `baseRevision`, `logicalSequence` 等以无前导零的十进制字符串传输，128-bit ID 使用规范小写文本。`requestHash = SHA-256(JCS/RFC 8785 规范化请求 body)`，不包含 Authorization/Cookie/代理头；C++/TypeScript/服务端共享黄金向量，字段顺序和空白不同但语义相同的请求得到同一 hash。若 G3 改用二进制 IDL，必须在协议版本中明确并保留等价幂等测试。

## 18. `/sync` 原子语义

请求可以含多条 Operation。可处理的 batch 一律以 HTTP 200 返回逐项 `accepted/rejected/conflict` 和最终 `headRevision`，并保证该 revision 只引用同一事务中已持久化的 accepted 结果；被 rejected/conflict 的 Operation 不被伪装成已应用。客户端不能把 HTTP 200 理解为“所有 Operation 成功”，也不能在非 2xx 后局部更新本地 cloud head。

Operation 带设备内 sequence 和显式 `causalParentIds`。同一设备/因果链中前驱被 rejected/conflict 后，依赖项返回 `blocked_by_rejected_predecessor`，不能越过前驱应用；没有因果边且属于独立领域的合法项才可继续 accepted。客户端只对 accepted 项更新云确认，rejected/conflict 及其本地效果进入 quarantined/local-only 分支或冲突解决，不得在 Snapshot 仍含被拒效果时把整个本地 Head 标成“已与 headRevision 同步”。

处理顺序：

1. 鉴权并验证 user/profile/device scope。
2. 规范化 body、计算 request hash、检查 idempotency 记录。
3. 验证协议、Schema、包集合、大小和 Operation envelope。
4. 锁定 Profile head；比较 base revision。
5. 逐领域去重/验证/合并，在内存事务模型中生成候选 head/conflicts。
6. 写 accepted Operation、RewardDedupKey/EntitlementGrant、conflict、snapshot metadata、必要的新 revision 与 request result；仅有 rejected 项时 head 可保持不变。
7. 最后更新 Profile head 并提交；失败整体回滚。
8. 对象 Blob 先写临时内容寻址对象并验证，再在 DB 引用；未被提交引用的对象由安全清理任务回收。

并发同步允许数据库 serialization/deadlock 重试，但相同 requestId 的外部语义不变。服务实例崩溃、客户端超时或响应丢失后，客户端重发同 requestId 即可查询结果。

## 19. PostgreSQL v1 不变量

实际 DDL/migration 是 G3 交付并接受数据库专项评审。至少由数据库约束而非应用内存守住：

- `profiles(id)`；每个 owner/Guest 认领状态和当前 `head_revision_id` 合法。
- `profile_revisions(profile_id, revision)` 唯一且单调，parent 指向同 Profile。
- `operations(profile_id, operation_id)` 唯一；`(profile_id, device_id, logical_sequence)` 唯一且有序。`logical_sequence` 仅设备内单调，服务端提交序由 revision 表达。
- `reward_dedup_keys(profile_id, reward_dedup_key)` 唯一；`entitlement_grants(grant_id)` 独立记录 issuer/signature/reference/状态，客户端不能生成。
- `sync_requests(profile_id, request_id)` 唯一并保存 request hash/结果摘要/到期时间。
- `devices(id, user_id)`、Profile 授权连接和撤销时间有 FK/不可绕过检查。
- `conflicts(id, profile_id)` 保留 base/A/B/解决 revision；未解决分支不能级联删除。
- `snapshots(hash)` 内容寻址，revision 引用、大小、加密/压缩版本和对象状态一致。
- 账号删除使用受审计状态机；运行时表不随意 `ON DELETE CASCADE` 清除修复证据。

同步事务使用可证明的行锁/可序列化策略；实现必须在两个服务实例并发 100 次相同/分叉请求下只产生合法 revision。Migration 采用 expand→backfill→verify→contract，运行身份无 DDL 权限；对象存储和数据库一致性有扫描/修复作业。
