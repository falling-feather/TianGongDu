# 01｜开发者文档

> 状态：Active
> 目标读者：C++ 客户端、Web 平台、玩法、工具、内容技术、后端、测试与发布工程师
> 最后更新：2026-07-10
> 维护者角色：技术负责人；各二级手册由表中模块 Owner 会签

这是新开发者进入项目的唯一入口。它回答“C++ 与网页如何分工、代码放哪里、一次改动怎样跨 Web/Native/云同步保持一致”。细节由 10–19 二级手册承载。

## 1. 90 分钟接手路径

1. 阅读 [`00-项目总纲.md`](00-项目总纲.md)，理解浏览器首发、多端复用和产品红线。
2. 阅读 [`04-游戏设计总纲.md`](04-游戏设计总纲.md)，理解玩家循环和两套战斗系统。
3. 阅读 [`01-developer/10-技术架构与依赖规则.md`](01-developer/10-技术架构与依赖规则.md) 与 [`01-developer/11-Runtime与Gameplay组件模型.md`](01-developer/11-Runtime与Gameplay组件模型.md)。
4. 阅读 [`content/design/technical-baseline.json`](../content/design/technical-baseline.json)，确认机器可读的平台基线。
5. 运行 `npm test` 与 `npm run validate:design`，确认设计、链接和技术选项没有漂移。
6. Foundation 阶段先从一个稳定 ID 追踪“Schema → 机器目录 → 模板 → 规划中的模块 Owner → 验收”；F1 代码合入后，再追踪真实的“内容编译 → C++ Runtime → Gameplay → Axmol Presentation → Web/Native Platform → 测试”，不得用空目录冒充实现证据。

Foundation 阶段尚未伪造完整 Axmol 工程。F1 将建立 CMake/Emscripten 构建、Web Shell、15–30 分钟纵切、IndexedDB 存档和同步服务契约实验。

## 2. 总体架构

```text
Authoring JSON + Schema ──validate/bake──► tgd::contracts

apps/web-shell ─────────┬──► tgd::presentation_axmol
                       ├──► tgd::platform_web ──► JS/DOM/PWA/IndexedDB/CDN/Sync
                       └──► tgd::gameplay ──► tgd::runtime ──► tgd::contracts

apps/native-shell ──────┬──► tgd::presentation_axmol
                       ├──► tgd::platform_native ──► Windows/Android/iOS services
                       └──► tgd::gameplay ──► tgd::runtime ──► tgd::contracts

tgd::presentation_axmol ──► tgd::runtime/contracts (只读快照与命令 DTO)
tgd::platform_* ──────────► tgd::runtime/contracts (实现平台接口)
```

箭头表示左侧可依赖右侧；Presentation 与 Platform 彼此不链接，由 app 组合。Contracts 不知道 Axmol、浏览器或数据库；Runtime 不知道具体地区；Gameplay 不直接控制 Sprite/Scene/DOM。Platform 采集物理输入并通过共享映射生成 Action/命令，Presentation 只观察快照、播放反馈和显示当前设备图标；两者都不能成为玩法真相。

## 5. 一条功能的标准生命线

以新增角色为例：

1. 内容工作台从角色模板生成稳定 ID、源数据、本地化 key 和测试夹具。
2. C++ ContentCore 以原生 CLI 和 WASM 两种形式运行同一 Schema/语义校验。
3. 内容被烘焙为带哈希的只读定义和 `.tgdpack` 索引。
4. Runtime 创建通用组件；Gameplay 消费日程、关系、任务、战斗与专业能力。
5. Axmol Presentation 播放动画/声音，DOM 只显示文本密集 UI；二者都不判定任务完成。
6. Web Platform 把本地档事务写入 IndexedDB；同步开启时把幂等操作日志提交云端。
7. 原生单元测试、WASM 集成测试、浏览器游玩和多设备冲突测试共同放行。

Boss、武器、任务和地区复用同一生命线，只替换模板和组件组合。

## 6. C++ 与跨平台规则

- 基线为 C++20；编译器、标准库、Axmol 和 Emscripten 版本由锁文件/CMake Preset 固定。
- 公开 API 使用 `tgd::<layer>` 命名空间；玩家文本只使用本地化 key。
- 所有权优先值语义、RAII、`std::unique_ptr`；跨层禁止裸拥有指针。
- 热路径不抛异常；错误使用 `Result<T, Error>` 或明确状态，Web 构建策略在 F1 锁定。
- API 表达单位，如 `DurationTicks`、`DistanceMeters`、`Bytes`。
- 固定模拟 60 Hz；浏览器 `requestAnimationFrame`、页面降频和原生显示刷新率不能改变规则时间。
- 随机数来自可注入、可序列化随机流；回放可固定种子。
- 存档不能引用指针、Axmol Node、JS 对象、DOM 元素、文件路径或数组位置。
- JavaScript↔C++ 只通过窄桥 DTO/事件；禁止任意 Embind 暴露整个对象图。
- Web 单线程构建是必选基线；Pthreads 是具备 COOP/COEP 时的增强构建，二者行为一致。

## 8. 分支与交付

- 主分支始终保持设计校验和已建立的构建门可用。
- 一个提交表达一个可回滚意图；WASM/Native 差异必须由平台适配而非 `#ifdef` 散落实现。
- PR 标明影响 Web 单线程、Web Pthreads、Native、同步服务和存档的哪些矩阵。
- 改变长期平台、线程、同步、内容包或身份模型必须新建 ADR。
- 发布说明列出命令、浏览器/设备、人工路径、性能、同步冲突和未覆盖风险。

## 9. 完成定义速查

一个功能至少满足：

- Web 单线程路径可达且反馈完整。
- Native 纯核心测试通过；平台差异有适配合同。
- 正常、失败、重试、页面隐藏/恢复、刷新、离线和本地读档可工作。
- 若有同步状态，重复请求、离线积压、双端冲突和账号链接经过验证。
- 内容由模板创建，不向 Runtime 写 ID 特判。
- 自动化、人工游玩、加载体积和帧性能证据齐全。
- 文档、迁移、缓存失效和回滚同步更新。

## 10. 开发组接手包

接手时应明确区分三种事实，避免把蓝图当成已经存在的代码：

| 状态 | 含义 | 当前例子 | 开发行为 |
| --- | --- | --- | --- |
| 已锁定合同 | 没有新 ADR 不得偏离 | C++20、60 Hz、GameCore 与表现分离、Web 单线程基线、本地优先 | 直接实现并建立守卫 |
| F1 待锁参数 | 方向已定，具体值必须实测 | Emscripten 精确版本、异常/RTTI、纹理格式、包分块、Pthreads 收益 | 先做最小 Spike，输出比较和决定 |
| 目标规格 | 表达最终能力，当前尚未实现 | 内容工作台、三地区、云冲突 UI、移动端 | 按 02 的里程碑进入，不得宣称已完成 |

开发组收到仓库后第一个可审计产物不是“很多框架文件”，而是一个从干净 Windows 环境可重复运行的 F1 Bootstrap：锁定工具 → 配置 → 编译 Native 核心 → 编译 Web Single → 启动本地 HTTPS → 运行测试。任何依赖 IDE 私有配置、作者机器路径或手工复制 DLL 的流程都不能作为团队基线。

开工前每条工作流必须能回答：

1. 该状态的唯一 Owner 在哪一层。
2. 对外契约、单位、版本和错误码是什么。
3. 正常、取消、失败、重试、刷新和旧版本怎样处理。
4. 哪个自动检查阻止反向依赖或数据漂移。
5. 哪份证据证明 Web 单线程路径真实成立。

## 11. 团队拓扑与模块所有权

“Owner”表示对一致性和最终合并负责，不表示独自完成所有代码。跨域改动必须由“必审角色”会签；人数较少时一人可兼任多角，但职责不能消失。

| 领域/目标 | 状态与接口 Owner | 主要产物 | 必审角色 |
| --- | --- | --- | --- |
| 产品范围、里程碑、裁定 | 制作人 | 00、02、取舍与验收签字 | 游戏总监、技术、内容、QA |
| 世界正史、章节、角色 | 叙事负责人 | 05–07、任务/对话/研究登记 | 游戏总监、内容、关卡、敏感性审阅 |
| Contracts/Runtime | C++ 核心负责人 | 稳定 ID、Tick、实体、命令/事件、存档 DTO | Gameplay、平台、QA |
| Gameplay/AI | 战斗与系统负责人 | 战斗、工灯、任务、关系、工艺、AI | 核心、关卡、动画、QA |
| Presentation/Axmol | 客户端表现负责人 | View、动画、音频、相机、GPU 资源 | Gameplay、技术美术、性能 |
| Web Shell/Platform Web | Web 平台负责人 | WASM Loader、DOM、PWA、IndexedDB、生命周期 | 核心、UX、安全、QA |
| Native Platform | 平台负责人 | Windows/移动入口、文件、输入、账号适配 | 核心、构建、QA |
| ContentCore/Workbench | 工具与内容技术负责人 | Schema、烘焙、编辑器、迁移、预览 | 内容、核心、QA |
| `.tgdpack`/资产管线 | Runtime 负责人 | 包格式、导入、图集、音频、预算 | 技术美术、Web、发布、安全 |
| Sync Contracts/API/DB | 后端负责人 | OpenAPI/IDL、Drogon、DDL、迁移、灾备 | Web、安全、运维、QA |
| 构建/发布/可观测性 | 构建与运维负责人 | Preset、CI、渠道、部署、回滚、指标 | 各模块 Owner、安全 |
| 质量与放行 | QA/发布负责人 | 测试矩阵、证据清单、缺陷裁定 | 制作人、技术、内容 |

跨层状态所有权速查：玩法时间、碰撞结果、任务、奖励和随机流属于 GameCore；Sprite/相机/音频属于 Presentation；键码、页面生命周期、存储事务和网络属于 Platform；DOM 焦点和可访问描述属于 Web Shell；云 revision 和官方权益 claim 属于 Sync Service。出现两个 Owner 时，设计就是未完成状态。

## 12. 工作项状态与 Definition of Ready

任务流固定为 `Proposed → Ready → In Progress → In Review → Verified → Shipped`，另有 `Blocked` 与 `Rejected`。只有满足下列 DoR 才能进入 `Ready`：

- 玩家价值、范围、非目标和验收路径明确。
- 影响的模块、Owner、必审人和稳定 ID 已确定。
- 接口/数据/错误/版本变化有合同草案；长期取舍已判断是否需要 ADR。
- 正常、失败、取消、刷新、离线、旧档和性能压力中适用项已列出。
- 依赖的内容、占位资产、测试夹具和浏览器环境可获得。
- 风险有触发器和失败回退；不能只写“以后优化”。

不满足 DoR 的想法留在规划或 Pitch，不得以“先写起来”进入正式 Runtime。`Verified` 还必须满足第 9 节 DoD，并附可复现证据；只有进入签名发布构建后才是 `Shipped`。

## 13. 首个 Bootstrap 迭代的共同目标

以下以“至少覆盖 C++ 核心、Web/客户端、Gameplay、工具/内容技术和 QA 的小型完整团队、关键工具/硬件无外部阻塞”为前提，可参考 10 个工作日组织；它不是按人分配的工时承诺。实际团队不足时延长迭代，但保持依赖顺序，防止各组过早各自铺开：

1. 技术负责人建立锁定工具清单、根 CMake/Preset、target 空壳和 include/link 层守卫。
2. 核心组建立 60 Hz `GameSession`、命令、事件、快照和原生回放夹具。
3. Web 组建立 HTTPS Shell、Single WASM 启动、生命周期与 DOM/Canvas 边界；先不用完整美术。
4. Gameplay 组用占位胶囊实现移动、跳跃、轻/重击、守势、闪身和一个工灯证据目标。
5. 工具组用一个 NPC 和一个章节 Boss 贯通 Schema → ContentCore → Bake → Runtime。
6. 存档组定义 IndexedDB v1 主路径并通过刷新/崩溃恢复；IDBFS 仅做兼容实验。
7. QA 建立同一输入在 30/60/144 渲染 FPS 和 Web/Windows 对照的黄金回放。
8. 后端先锁同步 IDL、权威矩阵和数据库唯一约束，再实现最小 API；不得反向拖住离线纵切。

Bootstrap 迭代结束评审只看：干净环境构建录像/日志、架构守卫、可操作 Web 灰盒、刷新恢复、黄金回放和未决参数表。文件数量、类数量和“框架完成百分比”不作为成果。

## 14. 评审与决策节奏

- 每日短同步只暴露阻塞和跨层契约变化，不以长口头汇报替代 issue/文档。
- 每周一次可玩构建评审，从标题页进入，不允许开发控制台跳过关键路径。
- 每周一次契约评审，集中处理 ABI、Schema、存档、API、包格式和迁移；结论回写 10–19。
- 每两周做风险复审，按 02 的触发器决定继续、收缩或启动替代宿主实验。
- 任何例外必须写明范围、Owner、到期日、移除条件和测试；没有到期日的临时例外视为永久债务，默认拒绝。
- 文档与代码同 PR 更新；模块 Owner 每个里程碑复审所属文档，陈旧文档按 Blocker 处理，不靠聊天补充。
