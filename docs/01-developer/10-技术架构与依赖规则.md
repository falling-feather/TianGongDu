> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Accepted Baseline；F1 锁定精确工具版本和 Axmol Web 可行性
> 最后更新：2026-07-11
> 维护者角色：技术负责人；Core、Web、Presentation、构建与后端 Owner 会签

## 1. 架构目标

项目必须同时满足：网页即开即玩、2.5D 斜向全景动作响应、可纵深走位、长期地区扩展、内容模板化、浏览器本地存档、跨设备接续和未来原生多端。任何平台便利都不能侵入 Gameplay；任何云能力都不能让网络中断毁掉单机进度。

优先级：稳定边界 → 可验证 Web 体验 → 内容吞吐 → 多端复用 → 性能优化 → 局部便利。

## 2. 技术基线

| 层 | 选择 | 策略 | 原因/边界 |
| --- | --- | --- | --- |
| 游戏核心 | C++20 | 固定 60 Hz、RAII、可原生/WASM 编译 | 单一 Gameplay，多端共享 |
| 2D 宿主 | Axmol 2.11.4 / 2.11.x LTS | 固定 tag+commit；Web Preview 必须过 F1 | 轻量 2D、C++、WebAssembly 与原生平台 |
| Web 编译 | Emscripten SDK | 通过 emsdk 安装；F1 锁精确版本/镜像 | C/C++ → wasm32 + JS loader |
| Web 渲染 | WebGL2 | 必选基线；不把 WebGPU 作为 1.0 依赖 | 与 Axmol Web 路径和现代浏览器一致 |
| Web UI | 语义化 DOM | Canvas 只绘制游戏世界/战斗 | 文本、输入、可访问性和响应布局更可靠 |
| 本地存储 | IndexedDB | 异步平台桥；IDBFS 仅作 POSIX 兼容层 | 浏览器刷新/重启后持久化 |
| 离线/缓存 | Service Worker + Cache Storage + HTTP Cache | 内容寻址文件，显式版本激活 | PWA、弱网、分包和回滚 |

| 运行内容 | `.tgdpack` + 二进制索引 | 清单、哈希、依赖、版本、签名 | 流式、CDN、缓存和跨端一致 |
| 同步服务 | C++20 + Drogon + PostgreSQL | F1 同步 Spike 后锁版本 | 与 C++ 合同共享，异步 HTTP/数据库 |
| 身份 | 外部 OIDC 提供方 | 访客先玩，后链接账号 | 不自建密码、找回和 MFA 系统 |

机器可读版本见 [`../../content/design/technical-baseline.json`](../../content/design/technical-baseline.json)。

## 3. 为什么是 Axmol，但不把核心交给 Axmol

Axmol 官方定位为 C++ 多平台 2D 引擎，并列出 WebAssembly、WebGL2、Windows、移动端等目标；2.11.x 是稳定维护线。它具备 Sprite/骨骼动画、相机、音频和输入等 2D/2.5D 宿主基础，但斜向全景所需的地面投影、纵深排序、遮挡以及 Sprite/预渲染/有限实时 3D 组合必须由 F1 代表样片验证；官方仍把 WebAssembly 标为 Preview。

因此决策是：

- 使用 Axmol 降低 2D 渲染/平台宿主成本。
- 不让 Axmol Scene/Node/Scheduler 成为玩法状态、任务状态或存档身份。
- Web Preview 必须经 F1 真实构建、浏览器矩阵、长时内存和资源加载验证。
- 若宿主失败，以 SDL/WebGL2 或其他 C++ 2D 宿主替换 Presentation/Platform；Contracts/Runtime/Gameplay/Content 保持。

这不是回到旧版“先写 Canvas Demo 再想架构”，而是先固定可替换边界。

## 4. C++ 目标与依赖图

```text
tgd_gameplay ──► tgd_runtime ──► tgd_contracts

tgd_presentation_axmol ──► tgd_runtime / tgd_contracts
tgd_platform_web ────────► tgd_runtime / tgd_contracts
tgd_platform_native ─────► tgd_runtime / tgd_contracts

web_shell ───────► gameplay + presentation_axmol + platform_web
native_launchers ► gameplay + presentation_axmol + platform_native

tgd_content_core ── depends on contracts only
tgd_sync_contracts ─ depends on contracts only
tgd_sync_service ─── depends on sync_contracts + Drogon/PostgreSQL
```

图中 `A ──► B` 表示 A 可以链接 B。Presentation 与 Platform 是由 app 组合的兄弟适配器，禁止彼此链接。

### 允许

- Runtime 依赖 Contracts。
- Gameplay 依赖 Runtime/Contracts。
- Presentation 只读取快照/表现事件、播放反馈并显示输入 glyph，不采集键码或决定 Action。
- Platform 实现 `IStorage`, `INetwork`, `IIdentity`, `ILifecycle`, `IClockSource`, `IAssetTransport` 和物理输入采集；共享数据驱动 `InputMapper` 在 Platform common 中把物理控制映射为 `ActionId/ActionEdge`，再由 app 提交命令。
- ContentCore 复用 Contracts，但不依赖 Gameplay/Axmol。
- Sync Service 复用稳定存档/操作 DTO，不链接客户端渲染和玩法循环。

### 禁止

- Contracts/Runtime/Gameplay include Axmol、Emscripten、Drogon、DOM/JS 绑定头。
- JavaScript 直接修改生命、背包、任务、关系或随机流。
- Axmol Scheduler 决定任务时间或战斗 Tick。
- `#ifdef __EMSCRIPTEN__` 散落在 Gameplay；平台差异只在 Platform/Build。
- Web 与 Native 各自维护存档字段或内容加载规则。
- Server 反序列化客户端任意 C++ 类型或信任客户端时间戳。

CI 用 include/layer lint、链接目标和架构测试守住依赖。

## 5. Web 进程模型

### 5.1 必选单线程构建

所有功能必须在不启用 Pthreads 的 wasm32 构建可玩。渲染、输入和模拟位于浏览器主线程，资源网络/IndexedDB 通过异步 JS API；重工作业切片并在预算安全点提交。

原因：Emscripten Pthreads 依赖 SharedArrayBuffer、COOP/COEP，且官方说明不能用同一二进制自动回退单线程。单线程是最大兼容与第三方嵌入的安全基线。

### 5.2 Pthreads 增强构建

在响应头已经提供 COOP/COEP、目标页 `crossOriginIsolated === true` 且 SharedArrayBuffer/Worker 可用时，Threads 渠道可加载单独的 `web_pthreads` 产物，把主循环代理到 Worker/Pthread，并为解压/寻路/内容解析使用固定 Worker 池。非隔离 Single 页面不能完成最终能力判断，也不能先加载页面再动态补响应头。

- 服务端必须返回 `Cross-Origin-Opener-Policy: same-origin`。
- 子资源满足 `Cross-Origin-Embedder-Policy: require-corp` 或可验证替代策略。
- CDN、字体、身份页面和第三方资源必须兼容隔离策略。
- Single 入口只能检测浏览器候选能力并在用户确认后导航到独立 Threads origin/channel；目标页再做最终隔离能力探测并选择 Threads 文件。若未来整个 Stable origin 统一隔离，必须另经 OIDC/CDN/嵌入门后才能同页选择。
- 两个产物有相同内容/存档/同步 API，但不同 origin 的 IndexedDB/Guest 档天然隔离，不能宣称本地档自动共享。
- Pthreads 性能没有显著收益时不为“看起来高级”启用。

## 6. 主循环与页面生命周期

Axmol/Web Shell 驱动渲染帧，GameCore 使用固定步长累加器：

```text
Poll Platform Events
Accumulate visible elapsed time
Run 0..N fixed 1/60s ticks (N capped)
Build immutable presentation snapshot
Render interpolated frame
Flush low-frequency UI events
Schedule async storage/network/assets
```

单帧最多追赶 4 个 Tick；超过时记录卡顿并丢弃过量显示时间，不能在页面恢复后追赶几分钟。`visibilitychange/pagehide/blur` 清空按键并暂停模拟，只可 best-effort 提交已经构造好的安全快照；关键事务必须在事件发生时完成，不能依赖 unload 阶段启动/完成 IndexedDB 新事务。恢复后重新校验输入、音频上下文、WebGL context 和资源句柄。

WebGL context lost 时停止提交绘制，保留 GameCore；context restored 后重建 GPU 资源和视图。无法恢复时保存安全点并给出重载入口。

## 7. JavaScript↔C++ 窄桥

桥只处理平台能力：启动配置、DOM 命令/UI 事件、IndexedDB、Fetch、OIDC token、剪贴板、全屏、Service Worker 状态和诊断。

建议接口：

```text
tgd_boot(BootConfig)
tgd_submit_platform_event(PlatformEventBatch)
tgd_submit_ui_command(UiCommand)
tgd_poll_ui_events(UiEventBuffer)
tgd_complete_async_request(RequestId, ResultBytes)
tgd_request_shutdown()
```

- 热路径使用固定布局二进制/环形缓冲，避免每帧 JSON 与大量 Embind 对象。
- DOM UI 由事件驱动或 10–20 Hz 小快照更新，不复制完整 WorldState。
- 每个异步请求带 RequestId、取消、超时和 Session generation，旧回调不能写入新存档。
- JS 异常转为结构化 PlatformError，不能穿过 WASM 边界破坏栈。
- C++ 向 JS 暴露最小 C ABI/受控 binding，不导出任意对象和指针。

## 8. Web Shell 职责

Web Shell 负责：

- 能力探测、选择单线程/Pthreads 产物。
- 加载/实例化 WASM，显示真实分阶段进度。
- Canvas 尺寸、DPR、全屏、页面生命周期和输入前置处理。
- DOM HUD、菜单、字幕、设置、账号/同步冲突与辅助技术。
- Service Worker 注册/更新、内容缓存状态和版本激活。
- IndexedDB 异步存储桥和多标签写入锁。
- OIDC 跳转/令牌安全交换，不把长期秘密交给 C++/本地存储。

它不拥有角色、战斗、任务、背包、掉落、关系、剧情选择和权威计时。

## 9. 本地存储

GameCore 只看异步 `IStorage`：`Read`, `WriteAtomic`, `List`, `Delete`, `EstimateQuota`, `RequestPersistence`。Web 实现优先直接使用 IndexedDB 对象存储保存快照、操作日志和元数据；IDBFS 只为必须使用 POSIX 文件的第三方/工具路径提供兼容。

保存流程：C++ 生成不可变字节快照/操作 → JS 开 IndexedDB 事务 → 写新 revision 与校验 → 提交 → 回调成功 → C++ 更新本地 head。刷新前的 `beforeunload` 不作为唯一保存机会。

浏览器存储默认可能被清理；游戏提供持久存储请求、云同步、手动导出/导入和清楚的本地/云状态。

## 10. 资源与内容流式

启动只内嵌最小字体/错误 UI。其余通过内容清单获取：

1. Web Shell 下载并验证签名的 release manifest。
2. 根据地区/语言/质量档计算包依赖。
3. Service Worker/HTTP Cache 以内容哈希 URL 缓存压缩包。
4. Platform AssetTransport 分块读取，C++ 校验 SHA-256/索引。
5. Runtime 在安全预算内解压/解析，Presentation 延后 GPU 上传。
6. 旧包在新 Session 验证后清理，不覆盖当前可运行版本。

不把全部 1.0 资源塞进 Emscripten preload `.data`。核心 WASM、Boot Pack、地区包、语言包和高清包独立版本。

## 11. Native 复用

Windows/Android/iOS 使用相同 CMake 目标 `contracts/runtime/gameplay/content`，只替换 Platform 和 Axmol 原生宿主入口。

- Native 文件系统实现同一异步语义，即使底层可同步。
- Native HTTP/OIDC/安全存储通过平台接口。
- 内容包、ID、存档、同步协议和回放完全相同。
- 输入映射从统一 Action 定义生成，平台只报告物理输入。
- 商店/IAP 若未来加入，属于平台服务，不直接发放 Gameplay 奖励；服务端验证后产生幂等操作。

每个 PR 的纯核心测试必须原生运行；多端真正发布仍需各自性能、输入和审核门。

## 12. 同步服务边界

同步服务不是实时游戏服务器。它负责身份映射、Profile head、快照、幂等操作、冲突检测、设备管理、配额、审计和数据导出/删除。

参考实现 C++20/Drogon + PostgreSQL；无状态 API 实例可横向扩展。Blob/大快照可放对象存储，PostgreSQL 保存 revision、操作索引、哈希与事务元数据。Redis 只在真实压测证明需要时加入。

客户端在无服务时完整游玩；服务端不可用时只积压已提交本地的操作。

## 13. 错误与诊断

- Boot 阶段错误必须区分能力缺失、WASM 加载、内容包、存储、账号和同步。
- Content/Schema 错误在构建期硬失败，Release 不静默默认。
- IndexedDB/配额错误保留内存快照并引导导出/释放空间，不能宣称已保存。
- 同步 401/409/429/5xx 有独立处理；重试使用退避、抖动和幂等 key。
- C++ 日志带 build/session/profile/device/revision/package/correlation ID，但不记录 token 和完整存档。
- Web 性能 HUD显示 WASM heap、JS heap（可用时）、纹理估算、下载、缓存、Tick、帧、长任务和同步队列。

## 16. CMake Target 与公开边界

第 4 节图中的箭头表示“上层可链接下层”。F1 根工程必须建立下列 target；公开头位于 `include/tgd/<layer>/`，实现和第三方适配位于 `src/`/`private/`，不得把 Axmol/Emscripten/Drogon 通过 public include 泄漏到核心。

| Target | 可公开依赖 | 状态/线程 Owner | 对外唯一入口 | 团队 Owner |
| --- | --- | --- | --- | --- |
| `tgd_contracts` | 标准库受控子集 | 纯值/无全局状态；调用线程 | ID、DTO、`Result/Error`、版本 | C++ 核心 |
| `tgd_runtime` | contracts | `GameSession`、Tick、实体、世界、碰撞、包、存档模型；模拟线程 | `RuntimeFacade` | C++ 核心 |
| `tgd_gameplay` | runtime/contracts | 战斗、AI、任务、关系、工艺；模拟线程 | `GameplayModuleSet` | Gameplay |
| `tgd_presentation_axmol` | runtime/contracts + Axmol(private) | View/相机/动画/音频/GPU；宿主渲染线程 | `PresentationHost` | 表现客户端 |
| `tgd_platform_web` | contracts/runtime + Emscripten(private) | 浏览器能力、异步请求；浏览器/Worker 边界 | `WebPlatformServices` | Web 平台 |
| `tgd_platform_native` | contracts/runtime + 平台库(private) | 文件/网络/输入/生命周期 | `NativePlatformServices` | 平台 |
| `tgd_content_core` | contracts | 校验、规范化、烘焙；CLI/Worker | `ContentCompiler` | 工具/内容技术 |
| `tgd_sync_contracts` | contracts | 同步 DTO/错误/版本；无 DB | `SyncProtocol` | 后端 + Core |
| `tgd_sync_service` | sync_contracts + Drogon/DB(private) | HTTP/事务/审计 | 版本化 API | 后端 |

应用入口只组合 target，不成为新业务层。CI 至少用 CMake link graph、禁止 include 列表和架构测试三道门阻止反向依赖；`friend`、相对路径 include 或复制 DTO 不能绕过。

## 17. 斜向全景权威坐标与碰撞

1.0 Gameplay 使用地面平面 `x/y` 加独立高度 `h` 与离散 `floorLayer`。地面 `x/y` 承担横向和纵深走位、距离、包夹与导航；`h` 只承担纵跃、击飞、落差和落地；`floorLayer` 区分桥上/桥下等合法重叠层。它们是规则状态，不等于屏幕坐标或自由 3D 刚体。核心单位为米、秒对应 Tick、弧度和显式质量/受力类型；Presentation 的斜向投影、屏幕比例和透视夸张不进入存档。

- 地面移动输入先归一为二维向量，再用当前 Cell 的受控 `CameraBasis` 在 Tick 边界转换成世界地面向量。镜头平滑、抖动和像素位置没有输入权；作者切换基向量时必须走显式过渡状态并可回放。
- 静态碰撞由关卡源烘焙为地面凸形/边、阻挡体、高度带、层连接、坡面/台阶、指定跨越锚点、危险区、导航和遮挡语义，加载到 Runtime 碰撞世界。
- 玩家和战斗 AI 使用 GameCore 内固定 Tick 运动/运动学控制器；地面吸附、坡面、纵跃、边缘宽限、层切换、击退和击飞都由规则状态决定。
- 命中与占位查询在地面形状相交后再检查高度区间和 `floorLayer`；查询结果按形状 ID、接触时间和实体稳定键规范化、量化、排序，不得依赖引擎容器或屏幕景深顺序。
- Axmol 物理只可用于非权威碎片、布景摆动和预览，不能决定玩家位置、命中、任务或存档。Runtime 若私有复用独立碰撞库，必须封在 `ICollisionWorld` 内、不得经 Presentation/Platform，并通过 Web/Windows 回放。
- 动态机关是受控 Gameplay Body/状态机，不是任意刚体沙盒。动画 Root Motion、Sprite 边界、遮挡透明度、景深排序和 GPU 粒子没有碰撞权。
- F1 必须用代表 Cell 验证八向移动、纵深绕行、前景遮挡、桥层/落差、地面危险和锁定目标，并在 10,000 Tick Web/Windows 夹具中校验量化后的 `x/y/h/floorLayer` 无平台分叉。

这项决定同时消除“视觉景深是否等于玩法坐标”和 `IPhysicsQuery` 所属层的歧义：正式名称为 Runtime `ICollisionWorld`，第三方碰撞库只能作为 Runtime 私有实现；Presentation 只投影和排序。

## 18. 固定 Tick 与异步提交细则

- 可见帧 delta 先夹到 `0..250 ms`；页面从 hidden 恢复的第一帧 delta 为 0。
- 累加器每帧最多执行 4 Tick。仍超出时保留不超过 1 Tick 插值余量、丢弃多余显示时间并发出 `simulation_overrun` 诊断；不会减慢/加快规则时间追债。
- 物理输入由 Platform 生成单调 `platformSequence`；在下一个 `CollectCommands` 阶段映射到目标 Tick。到达太晚的事件不回写过去 Tick。
- 同 Tick 命令按 `tick → actor stable key → command sequence → command type tie-break` 排序；浏览器 key repeat 不生成新的按下沿。
- DOM/网络/存储/内容异步完成只在 Tick 边界变成 `PlatformCompletionCommand`。回调携带 Session generation；旧 generation 只记录并丢弃。
- 暂停模式仍可泵平台事件、UI 和已开始的存储事务，但不推进 Gameplay Tick。恢复先清空物理持续键，再等待新的输入沿。
- 关闭/刷新前不承诺执行新 Gameplay Tick；重大持久事务必须在事件发生时提交，不能等待 `beforeunload`。

黄金回放覆盖 30/60/144 渲染 FPS、1–4 Tick 追帧、blur/visibility、手柄断连和异步顺序打乱，比较首个/最终离散状态与 Persistent Operation 序列。

## 19. JS↔WASM ABI v0.1 行为合同

第 7 节函数名是稳定方向，F1 G1 必须产出 `tgd_web_abi.h`、机器 IDL/Schema、生成的 TypeScript 类型和双向 mock 契约测试。在头文件锁定前，不允许 Web 与 C++ 分别发明结构。

行为先固定如下：

- ABI 有独立 `major/minor`；major 不兼容，minor 只追加能力。Boot 双方先比较版本/能力，再创建 Session。
- 跨边界数据使用显式小端、定宽整数和 `(ptr, length)` 字节区；不传 STL 类型、C++ 指针、JS 对象、DOM 节点或隐式字符串终止符。
- JS→C++ 输入在调用返回前由 C++ 验证并复制/消费；C++ 不保留 JS 临时 ArrayBuffer 指针。
- C++→JS 使用调用方提供缓冲的 `poll/peek required size` 模式或受控环形缓冲；谁分配谁释放，不跨 allocator 释放。
- 每条消息有 `abiVersion, messageType, payloadVersion, sessionGeneration, sequence, payloadLength`；未知 message/type/过长 payload 结构化拒绝。
- 平台异步请求含 128-bit `requestId`、相对 `timeoutMs` 和 cancel token；超时由 Platform monotonic clock 计算，不用墙钟。Gameplay 内时限仍使用 Tick；完成只提交一次，取消后晚回调不得写入新 Session/Profile。
- UI 高频摘要允许按 key 合并旧值。关键队列为错误、焦点、保存结果、重大选择和持久事务结果预留容量；队列满时 ABI 返回 `Backpressure`，生产者在安全点重试/暂停。若关键消息仍无法入队，Session 进入可诊断安全暂停/PlatformFatal，绝不静默丢弃；Persistent Operation 本体先在持久层提交，不只存在桥队列。
- 桥单消息 F1 上限 256 KiB；大型包/快照通过分块异步传输与流式接口，不塞进 UI 队列。
- 错误最少含 `domain, code, retryable, messageKey, args, correlationId`；JS exception 捕获后转换，C++ exception 不跨边界。
- 任一函数重入策略必须写在 IDL：默认不允许回调在同一栈重新进入 GameCore，完成事件排到下一平台泵/Tick。

契约测试至少覆盖：ABI 不兼容、短缓冲、越界长度、未知枚举、JS 异常、重复完成、取消竞态、旧 generation、队列满和 Session 销毁。

## 21. Action Registry 与输入仲裁

稳定 Action/Context 入口为 [`../../content/design/action-registry.json`](../../content/design/action-registry.json)，Schema 为 [`../../content/schemas/action-registry.schema.json`](../../content/schemas/action-registry.schema.json)。Web/Native/手柄/触控只能把物理控制映射到这些 ID；新增 Action 先更新注册表、08 默认映射、提示本地化和回放夹具，不能在代码中临时发明字符串。

C++ 只读表由 `tools/generate-action-contract.mjs` 从注册表生成到 `action_registry.generated.hpp`，稳定数值 ID、类型、Context、edge policy 与 `clearOnBlur` 均由 `npm run check:action-contract` 防漂移，生成头禁止手改。GameCore 的标量热路径只接收有单调 `platformSequence` 的 `ScalarActionSample`；`vector2` UI 导航和文本/IME 走各自 DTO，不能塞进一个假装通用的标量字段。`SessionInputState` 在 Tick 边界将 `move_x/move_y` 按正交、正手性的版本化 `CameraBasisQ15` 转为世界地面命令；已入队命令保存世界方向，不随之后的镜头切换重解释。

- `valueType` 区分 digital/axis1d/vector2/text；标量轴先在 Platform 设备档做死区、反转和归一化，再进入 `ScalarActionSample`。
- Context 的数值越大、优先级越高，固定顺序为 text_entry(40) > menu(30) > dialogue(20) > gameplay(10) > system(0)。输入从高到低仲裁，首个声明处理该物理触发的 Context 消费它；未处理输入才继续下落。`system` 因而是全局回退而非最高层，例如 Gameplay 中 Escape 可落到 Pause，菜单/文本输入中的同键先成为 Cancel。同优先级禁止；进入/退出高层 Context 时清空 held Gameplay Action。
- 浏览器 key repeat 不制造新的 Gameplay press edge；UI repeat 只由 `repeat_with_ui_delay` 策略生成。
- blur、visibility、手柄断连和映射切换生成明确 release/clear batch；恢复后必须等待新物理沿。
- 最近设备只改变 glyph/提示，不重置动作、不改存档；多个设备同时输入按 sequence 合并，互相冲突的轴采用最近有效来源并记录诊断。
- 普通文本由 DOM `beforeinput/input` 归一成一次 TextCommit；IME 在 composition 期间不提交，只有 `compositionend` 归一成 CompositionCommit。两条路径都只进入 `text_entry`，永不成为 Gameplay Action 或快捷键。
- `Tab` 永远保留给 DOM 焦点导航，不作为 Gameplay 默认键；浏览器正以 Escape 退出全屏或 Pointer Lock 时，该次按键不再派发 Pause/Cancel，下一次物理按下才参与 Action 仲裁。
- 同一控制的短按/长按复用必须延迟短按裁决：释放早于 `holdThresholdMs` 才发短按，跨过阈值只发长按，二者永不同时发出。手柄 Back 的地图/日志复用以此为标准夹具。
- 默认映射见 08；重映射冲突区分硬冲突（同 Context 同触发不可辨）和允许冲突（互斥 Context），UI 给出原因。

G1 契约测试用同一物理输入 fixture 覆盖键盘、鼠标、手柄、触控模拟、deadzone、repeat、blur release、Context 切换和设备断连，并比较最终 Action 序列。
