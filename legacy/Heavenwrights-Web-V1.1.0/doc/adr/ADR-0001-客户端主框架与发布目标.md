# ADR-0001：客户端主框架与发布目标

> 状态：已接受  
> 日期：2026-06-28  
> 关联文档：`../04_技术栈与生产管线.md`、`../01_开发者手册.md`

## 背景

《天工渡》需要一个适合 2D / 类 2.5D 动作 RPG、Web 端优先发布、代码优先和 AI 协作内容生产的客户端技术路线。项目同时需要保留长期性能空间、跨平台可能性和文本化内容工具链。

## 备选方案

| 方案 | 取舍 |
| --- | --- |
| Axmol + C++ Runtime | 贴近 2D 游戏框架，C++ 主体，支持 WebAssembly/WebGL2 方向，适合代码优先和自研规则层。 |
| SDL / SFML / raylib | 底层灵活，但需要自行补齐更多游戏框架能力。 |
| Unity / Godot / UE | 编辑器生态成熟，但二进制配置和 Inspector / Scene 工作流会增加 AI 批量审查与 diff 难度。 |
| 纯 Web 技术长期实现 | 原型效率高，但长期动作手感、资源包、性能和跨端目标仍需重新评估。 |

## 决策

采用：

```text
C++20/23
+ Axmol Engine
+ Heavenwrights Runtime 自研游戏层
+ Python 内容工具链
+ 文本化内容数据
+ WebAssembly / WebGL2 浏览器发布
```

当前 `prototype/web-demo/` 仅作为 M0 浏览器原型，不替代最终 Axmol/C++ 实现。

## 影响

- 核心战斗、技能、NPC、工艺、四维状态不能写死在 Axmol `Node` 或具体场景回调中。
- Runtime 负责固定逻辑帧、实体、碰撞、战斗、技能、工艺、NPC、任务、世界状态、存档和内容加载。
- Axmol 负责应用启动、平台基础、渲染、输入、音频、资源加载、场景壳和 UI 基础。
- Python 工具链负责内容校验、打包、diff、预览和 AI 协作辅助。
- WebAssembly/WebGL2 是必须验证的首要发布链路。

## 复审条件

- Axmol WebAssembly/WebGL2 在加载、音频、资源包、输入或性能上达不到最低可玩标准。
- 依赖许可、构建链路或平台支持发生重大变化。
- 项目转向完全不同的发布平台或生产方式。
- 内容工具链无法支撑文本化、可 diff、可校验的协作方式。
