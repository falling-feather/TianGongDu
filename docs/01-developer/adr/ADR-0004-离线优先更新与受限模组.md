# ADR-0004｜离线优先、签名内容更新与受限模组

> 状态：Accepted
> 日期：2026-07-10
> 决策者：项目产品/技术/安全基线

## 背景与问题

项目是单人端游，但长期更新需要内容分发、回滚和可能的社区扩展。若把剧情/战斗依赖在线服务，会降低可保存性并制造高运维成本；若让模组执行任意 C#/.NET 代码，会把客户端文件、供应链和存档暴露给不必要风险。

## 决策驱动因素

- 无网络时完整游玩 1.0。
- 官方更新可验证、可中断恢复、可回滚。
- 内容包能独立于可执行更新，但不热更任意代码。
- 社区扩展有清楚能力与存档隔离。
- 服务停机不会终结已购买单机内容。
- 隐私与安全复杂度与真实需求相称。

## 备选方案

### A. 全程在线与服务器权威

适合强联网经济/多人，但项目没有此需求；成本和保存风险高。拒绝。

### B. 完全无更新/模组架构

最简单，但不符合长期更新和内容修复目标。拒绝。

### C. 任意代码模组

能力最强，但安全、兼容、AOT、崩溃和支持成本极高。1.0 拒绝；未来需独立沙箱 ADR。

### D. 离线核心 + 签名数据/资源包 + 受限扩展点

满足长期内容与安全边界。选择。

## 决策

主线、战斗、存档、关系、工艺和结局完全本地运行。可选控制面只提供签名版本清单、包下载、诊断/遥测（经同意）和未来社区索引。

官方内容更新为带项目清单、哈希、签名、依赖和兼容范围的 PCK/数据资源包；新增 C# 逻辑必须发布新可执行。安装保留上一版本并原子切换。

1.0 模组只允许受验证数据/资源和公开扩展点，使用独立命名空间与存档块；禁止任意程序集、原生库、网络和任意文件访问。

## 正面后果

- 服务停止或玩家离线不影响游戏。
- 更新损坏可以回滚，来源可验证。
- 官方内容可按地区/语言分包，减少整包替换。
- 模组问题可通过安全模式隔离，不必把本机权限交给内容。
- 单机设计不被服务器延迟和运维牵制。

## 负面后果与风险

- 受限模组表达力低于任意代码，社区可能不满足。
- 需要自建签名清单、下载、原子切换和存档兼容工具。
- 数据驱动系统必须提供足够公开扩展点，否则模组只能换皮。
- 客户端仍无法抵抗本机管理员篡改，不应虚假宣传防篡改。
- 签名密钥和发布供应链成为高价值目标。

## 验证与放行门

- 断网首次/再次启动、完整主线和本地存档工作。
- 下载中断、磁盘满、清单/包篡改、旧版本重放安全失败。
- 更新后启动失败可自动回滚；新旧存档边界清楚。
- 模组无法路径逃逸、加载代码、覆盖官方 ID 或造成无界资源分配。
- 缺模组存档进入安全诊断，不覆盖原档。
- 遥测关闭不影响内容，并验证脱敏。

## 迁移与回滚

F1 先实现本地包清单、挂载、签名验证接口和双版本切换，不急于建立生产服务。无服务时可通过完整安装包发布。若 PCK 差分不可靠，回退完整包下载；可靠性优先于带宽优化。

未来若开放代码模组或多人，必须新 ADR 定义能力沙箱、平台、签名、权限、存档和支持边界，不直接放宽 1.0 检查。
