# 01｜开发者文档

> 状态：Active
> 目标读者：C++ 客户端、Web 平台、玩法、工具、内容技术、后端、测试与发布工程师
> 最后更新：2026-07-10

这是新开发者进入项目的唯一入口。它回答“C++ 与网页如何分工、代码放哪里、一次改动怎样跨 Web/Native/云同步保持一致”。细节由 10–19 二级手册承载。

## 1. 90 分钟接手路径

1. 阅读 [`00-项目总纲.md`](00-项目总纲.md)，理解浏览器首发、多端复用和产品红线。
2. 阅读 [`04-游戏设计总纲.md`](04-游戏设计总纲.md)，理解玩家循环和两套战斗系统。
3. 阅读 [`01-developer/10-技术架构与依赖规则.md`](01-developer/10-技术架构与依赖规则.md) 与 [`01-developer/11-Runtime与Gameplay组件模型.md`](01-developer/11-Runtime与Gameplay组件模型.md)。
4. 阅读 [`content/design/technical-baseline.json`](../content/design/technical-baseline.json)，确认机器可读的平台基线。
5. 运行 `npm test` 与 `npm run validate:design`，确认设计、链接和技术选项没有漂移。
6. 从一个稳定 ID 追踪“Schema → 内容编译 → C++ Runtime → Gameplay → Axmol Presentation → Web/Native Platform → 测试”。

Foundation 阶段尚未伪造完整 Axmol 工程。F1 将建立 CMake/Emscripten 构建、Web Shell、15–30 分钟纵切、IndexedDB 存档和同步服务契约实验。

## 2. 总体架构

```text
Authoring JSON + Schema
           │ validate / bake
           ▼
      tgd::contracts
           │
           ▼
       tgd::runtime
           │
           ▼
      tgd::gameplay
           │ commands / events / snapshots
           ▼
 tgd::presentation_axmol
           │
           ├──────────────┐
           ▼              ▼
 tgd::platform_web   tgd::platform_native
           │              │
    Emscripten/WASM   Windows/Android/iOS
           │
   JS Bridge + DOM UI + PWA
           │
 IndexedDB / CDN / Sync API
```

依赖只能向下：Contracts 不知道 Axmol、浏览器或数据库；Runtime 不知道具体地区；Gameplay 不直接控制 Sprite/Scene/DOM；Presentation 只能观察状态和提交命令；Platform 处理文件、网络、账号、生命周期和平台能力，不能成为玩法真相。

## 3. 目标仓库结构

| 路径 | 职责 | 禁止放入 |
| --- | --- | --- |
| `apps/web-shell/` | HTML、DOM UI、PWA、Service Worker、WASM loader | 战斗/任务规则、第二份存档模型 |

| `src/presentation-axmol/` | Sprite、动画、音频、相机、碰撞适配 | 修改权威玩法状态 |

## 5. 一条功能的标准生命线

以新增角色为例：

1. 内容工作台从角色模板生成稳定 ID、源数据、本地化 key 和测试夹具。
2. C++ ContentCore 以原生 CLI 和 WASM 两种形式运行同一 Schema/语义校验。
3. 内容被烘焙为带哈希的只读定义和 `.tgdpack` 索引。
4. Runtime 创建通用组件；Gameplay 消费日程、关系、任务、战斗与专业能力。
5. Axmol Presentation 播放动画/声音，DOM 只显示文本密集 UI；二者都不判定任务完成。
6. Web Platform 把本地档事务写入 IndexedDB；同步开启时把幂等操作日志提交云端。
7. 原生单元测试、WASM 集成测试、浏览器游玩和多设备冲突测试共同放行。

Boss、武器、任务和地区复用同一生命线，只替换模板和组件组合。

## 6. C++ 与跨平台规则

- 基线为 C++20；编译器、标准库、Axmol 和 Emscripten 版本由锁文件/CMake Preset 固定。
- 公开 API 使用 `tgd::<layer>` 命名空间；玩家文本只使用本地化 key。
- 所有权优先值语义、RAII、`std::unique_ptr`；跨层禁止裸拥有指针。
- 热路径不抛异常；错误使用 `Result<T, Error>` 或明确状态，Web 构建策略在 F1 锁定。
- API 表达单位，如 `DurationTicks`、`DistanceMeters`、`Bytes`。
- 固定模拟 60 Hz；浏览器 `requestAnimationFrame`、页面降频和原生显示刷新率不能改变规则时间。
- 随机数来自可注入、可序列化随机流；回放可固定种子。
- 存档不能引用指针、Axmol Node、JS 对象、DOM 元素、文件路径或数组位置。
- JavaScript↔C++ 只通过窄桥 DTO/事件；禁止任意 Embind 暴露整个对象图。
- Web 单线程构建是必选基线；Pthreads 是具备 COOP/COEP 时的增强构建，二者行为一致。

## 7. 目标本地工作流

F1 后统一入口：

```text
cmake --preset windows-clang-debug
cmake --build --preset windows-clang-debug
ctest --preset windows-clang-debug

emcmake cmake --preset web-release
cmake --build --preset web-release
npm run test:web
npm run validate:design
```

同步服务：

```text
cmake --preset sync-service-debug
cmake --build --preset sync-service-debug
ctest --preset sync-service-debug
```

当前 Foundation 仍可执行 `npm test` 与 `npm run validate:design`。

## 8. 分支与交付

- 主分支始终保持设计校验和已建立的构建门可用。
- 一个提交表达一个可回滚意图；WASM/Native 差异必须由平台适配而非 `#ifdef` 散落实现。
- PR 标明影响 Web 单线程、Web Pthreads、Native、同步服务和存档的哪些矩阵。
- 改变长期平台、线程、同步、内容包或身份模型必须新建 ADR。
- 发布说明列出命令、浏览器/设备、人工路径、性能、同步冲突和未覆盖风险。

## 9. 完成定义速查

一个功能至少满足：

- Web 单线程路径可达且反馈完整。
- Native 纯核心测试通过；平台差异有适配合同。
- 正常、失败、重试、页面隐藏/恢复、刷新、离线和本地读档可工作。
- 若有同步状态，重复请求、离线积压、双端冲突和账号链接经过验证。
- 内容由模板创建，不向 Runtime 写 ID 特判。
- 自动化、人工游玩、加载体积和帧性能证据齐全。
- 文档、迁移、缓存失效和回滚同步更新。
