# 01｜开发者文档

> 状态：Active
> 目标读者：客户端、玩法、工具、内容技术、美术技术、测试与发布工程师
> 最后更新：2026-07-10

这是一名新开发者进入项目的唯一入口。它回答“先读什么、代码应该放哪里、依赖可以怎样流动、一次改动怎样才算完成”。细节由 10–19 二级手册承载，避免把所有规则堆进一个无法维护的长文档。

## 1. 90 分钟接手路径

1. 用 15 分钟阅读 [`00-项目总纲.md`](00-项目总纲.md)，理解产品边界和质量红线。
2. 用 20 分钟阅读 [`04-游戏设计总纲.md`](04-游戏设计总纲.md)，理解玩家循环和两套战斗系统。
3. 用 20 分钟阅读 [`01-developer/10-技术架构与依赖规则.md`](01-developer/10-技术架构与依赖规则.md) 与 [`01-developer/11-Runtime与Gameplay组件模型.md`](01-developer/11-Runtime与Gameplay组件模型.md)。
4. 在仓库根运行 `npm test` 与 `npm run validate:design`，确认设计基础没有漂移。
5. 从一个稳定 ID 出发，沿“Schema → 内容定义 → Runtime 组件 → Gameplay 系统 → Presentation 绑定 → 测试”追踪一条完整链路。
6. 开始任务前确认它属于 02 的哪个里程碑、需要什么验收证据、是否触发 ADR。

Foundation 阶段的仓库尚未假装存在完整 Godot 工程；F1 引擎烘焙会按本文档建立解决方案、客户端宿主和第一个端到端切片。

## 2. 架构总图

```text
Content JSON + Schema
          │ validate / bake
          ▼
Tgd.Contracts  ◄──── Save / Package / Version contracts
          │
          ▼
Tgd.Runtime     ◄──── clock, IDs, events, persistence, streaming
          │
          ▼
Tgd.Gameplay    ◄──── combat, AI, quests, relationships, crafting
          │ commands/events/snapshots
          ▼
Tgd.Presentation.Godot
          │
          ├── apps/client-godot
          └── Tgd.Editor.Godot
```

依赖只能向下：Contracts 不知道 Godot；Runtime 不知道具体地区；Gameplay 不直接控制动画节点；Presentation 可以观察状态和提交命令，但不能成为规则真相；Editor 复用正式 Schema 与校验器，不能维护第二套规则。

## 3. 目标仓库结构

| 路径 | 职责 | 禁止放入 |
| --- | --- | --- |
| `apps/client-godot/` | Godot 项目、启动场景、平台适配 | 业务真相、任意内容解析 |
| `src/Tgd.Contracts/` | ID、DTO、Schema 绑定、版本接口 | Godot 类型、具体玩法 |
| `src/Tgd.Runtime/` | 时钟、事件、存档、包、世界流式 | 具体武器和地区脚本 |
| `src/Tgd.Gameplay/` | 战斗、AI、任务、关系、工艺 | 直接操作 Node/动画 |
| `src/Tgd.Presentation.Godot/` | 状态到画面/声音的适配 | 修改权威玩法状态 |
| `src/Tgd.Editor.Godot/` | 内容编辑器插件与预览器 | 只在编辑器成立的内容格式 |
| `content/` | Schema、模板、源内容、包清单 | DCC 二进制源文件 |
| `assets_src/` | Blender/Krita/Aseprite 等源资产 | 直接发布的缓存产物 |
| `assets_runtime/` | 烘焙后的可发布资源 | 手工不可复现的修改 |
| `server/` | 可选控制面和部署清单 | 单机核心规则 |
| `tools/` | 校验、导入、构建、发布工具 | 客户端帧循环逻辑 |
| `tests/` | 单元、契约、集成、场景、性能测试 | 依赖开发者本机状态的脚本 |

## 5. 一条功能的标准生命线

以“新增角色”为例：

1. 设计者从 06 的角色模板填写角色意图、关系弧、日程和交互预算。
2. 编辑器生成稳定 ID 和最小合法内容，Schema 立即校验。
3. 角色定义被烘焙为不可变运行时数据；表现资源只以资产 ID 绑定。
4. Runtime 创建实体和通用组件；Gameplay 系统消费日程、对话、关系、任务与战斗能力。
5. Presentation 根据状态播放动画、语音和 UI，不自行判定任务完成。
6. 预览器执行角色独立沙盒；集成场景验证与地区、存档和本地化联动。
7. 内容审查检查角色是否真的有关系变化和可玩作用，测试审查检查回归与性能。

新增 Boss、武器、任务和地区沿用同一条生命线，只替换模板与组件组合。

## 6. 编码与数据基本规则

- C# 命名使用英文；面向玩家的名称使用本地化 key，不写死在逻辑中。
- 公开 API 必须表达单位，例如 `DurationMs`、`DistanceMeters`，不能用含糊的 `value`。
- 所有随机数来自可注入的随机源；测试和回放可固定种子。
- 固定模拟为 60 Hz；渲染帧率不能改变伤害、AI 或任务时间。
- Runtime 数据优先不可变；跨系统变化通过命令、事件和显式状态机。
- 禁止用 Godot 节点路径、数组位置、资源显示名作为存档键。
- 任何持久字段的删除或改义都要提供迁移。
- 任何跨层例外都要写 ADR，而不是留一句 TODO。

## 7. 本地工作流

目标命令在 F1 建立后统一为：

```text
dotnet build TianGongDu.slnx
dotnet test TianGongDu.slnx
npm run validate:design
python -m tools.content validate content/
godot --headless --path apps/client-godot --editor --quit
```

当前 Foundation 节点可执行：

```text
npm test
npm run validate:design
```

不得把“我的编辑器能打开”作为提交验证。每次交付要在说明中列出命令、结果、人工游玩路径和未覆盖风险。

## 8. 分支与提交

- 主分支始终保持可验证；中型以上改动使用 `codex/` 或团队约定前缀的短生命周期分支。
- 一个提交表达一个可回滚意图；生成资产和源资产变化要能追溯到同一任务。
- 设计变化先改契约与文档，再改实现；紧急修复可同步提交，但不能永久欠账。
- PR 必须绑定 02 的里程碑、列出玩家价值、风险、测试和性能影响。
- 旧 Web 项目只在归档分支修复历史问题，不向主分支回灌架构。

## 9. 完成定义速查

一个开发任务至少同时满足：

- 对玩家可达且有明确反馈。
- 从内容到表现的链路完整，没有手工隐藏步骤。
- 正常、失败、重试、存档读档和边界条件可工作。
- 自动化、人工游玩和性能证据齐全。
- 文档和迁移同步更新。
- 没有把本次内容硬编码进通用层。
- 审查者可以在干净环境复现。

完整门禁见 02 和 16。
