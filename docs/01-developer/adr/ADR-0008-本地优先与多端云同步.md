# ADR-0008｜本地优先、Guest 可玩与多端云同步

> 状态：Accepted
> 日期：2026-07-10
> 决策者：产品/技术/后端/安全基线
> Supersedes：ADR-0004

## 背景与问题

用户希望考虑多端同步。浏览器数据可能被清理，未来 Windows/移动端需要同一进度；但强制账号/全程联网会破坏网页低门槛和单机可靠性。全档最后写入获胜会静默丢掉另一设备的任务、库存和叙事选择。

## 驱动因素

- 首次打开立即 Guest 游玩。
- 网络/服务故障不阻止已缓存内容。
- 本地明确“已保存”，云状态另行显示。
- Web/Windows/移动端共享 Profile。
- 重复请求、离线积压和双端分叉安全。
- 叙事冲突可解释、双方保留。
- 不自建密码、MFA 和账号找回系统。

## 备选方案

### A. 不做云同步，只导出文件

简单可靠，但浏览器清理风险和多端体验差。保留导出作为兜底，不作为唯一方案。

### B. 强制账号、服务器权威全进度

合并简单但违背即开即玩/离线，增加运维与隐私。拒绝。

### C. 云端整档 Last-Write-Wins

实现最省事，但双端必然丢进度，叙事无法解释。拒绝。

### D. Guest 本地优先 + OIDC 链接 + 修订/领域 Operation

兼顾进入门槛、可靠性和多端。选择。

## 决策

游戏默认创建本地 Guest Profile；每次进度先原子提交本地 Snapshot/Operation，之后可选同步。账号使用外部 OIDC；链接时显式比较本地/云档。

同步服务参考实现为 C++20/Drogon + PostgreSQL。Profile 使用单调 cloud revision；写请求/Operation 幂等。集合/单向进度自动合并，库存/货币使用账本/claim，任务按状态偏序，关键叙事/地区裁定冲突由玩家选择或复制 Profile。

同步服务不接管实时战斗，不要求在线。Native 与 Web 是同一 Profile 的不同设备。

## 正面后果

- 低门槛 Guest 与跨设备备份同时存在。
- 服务故障不破坏本地进度。
- 重复/乱序请求不重复奖励。
- 冲突透明，不悄悄覆盖数小时内容。
- OIDC 降低密码安全和恢复负担。
- C++ 合同可在客户端/服务共享 DTO/校验思想。

## 负面后果与风险

- 领域合并和冲突 UI 比整档上传复杂。
- 本地可修改意味着不能信任在线经济；未来付费权益需服务端权威 Operation。
- OIDC popup/Cookie 与 COOP/COEP/PWA 组合需专项测试。
- PostgreSQL 备份、迁移、数据删除和隐私带来长期运维责任。
- Guest 若不链接账号且站点数据被清理，仍可能丢档；必须清楚提示/导出。
- C++ 服务不是因“同语言”就天然更省成本，需压测和运维能力。

## 验证

- Guest 开始、刷新、离线、导出/导入。
- Guest 链接 OIDC 的本地空/云空/双方有档四类路径。
- 双设备离线分叉后的每个领域合并与叙事冲突。
- 同一 request/Operation 重放 100 次只生效一次。
- 401/409/429/5xx、超时、数据库回滚不破坏本地档。
- 设备撤销、token 过期、Profile 导出/删除和备份恢复。
- Web/Native 同一 Profile、Schema/PackageSet 不兼容提示。

## 迁移/回滚

F1 先做最小 Sync Spike，不把服务设为启动依赖。服务未完成时 Web 仍以 IndexedDB + 导出发布内部版本。若 Drogon 不满足团队运维/生态，允许替换服务实现，但 Sync Protocol、PostgreSQL 数据模型、OIDC 和本地优先合同保持。

若自动合并某领域不安全，回退为保留双方/人工选择，而不是扩大 Last-Write-Wins。
