# 14｜资产管线与性能预算

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Web-first Baseline；数值由 F1 真实设备校准
> 最后更新：2026-07-10
> 维护者角色：技术美术负责人；美术指导、动画、Web/表现、音频、许可与 QA 会签

## 1. 美术方向

1.0 采用横版 2D/2.5D 关卡阅读：多层视差背景、2D 角色与特效、有限动态光/Shader、明确前中后景和可交互轮廓。目标是“造梦西游式易读横版动作”的进入方式，加上《天工渡》的工艺细节、社会空间和更复杂战斗，而不是追求重型全 3D。

表现方案优先顺序：Sprite 帧动画/图集基线 → 经许可的骨骼动画 → 少量预渲染/Shader 特效。不能让某个专有动画工具成为全部角色的不可替换前提。

## 2. 管线原则

- 源资产、导入配置、运行时产物、内容包和 CDN 文件分层。
- 所有发布资源可由锁定工具从源重建。
- 先定网页屏幕贡献、下载/内存预算，再定分辨率和帧数。
- 同类资产共享图集、骨骼、动画状态、材质和碰撞规范。
- 资源包按玩家使用路径分，而不是按美术部门目录分。
- 冷启动、弱网、缓存清理和低内存从灰盒开始测试。

## 3. 目录

```text
assets_src/<domain>/<region>/<asset_id>/
  source.aseprite / source.psd / source.blend / audio project
  references/
  license.yaml
  import.yaml

assets_runtime/<package>/<asset_id>/
  atlases/
  animations/
  audio/
  fx/
  manifest.fragment.json
  previews/
```

资产 ID 为小写英文稳定键。显示名、作者和许可进入 metadata。导入器不能依赖绝对路径、作者机器字体或未锁插件。

## 4. 角色动画

### 4.1 Sprite 图集基线

- 统一逻辑朝向、脚底锚点、受击盒原点和像素/世界单位。
- 动画定义首帧、关键姿态、接触、恢复和循环语义；Gameplay 窗口仍由 Tick 定义。
- 图集按角色家族/生命周期分页，不能把全地区角色塞进一张图。
- 透明边界裁剪但保留一致锚点；预览自动检查抖动、脚滑和裁切。
- 常见角色共享动作语法，核心 NPC 通过职业动作、轮廓、配色和局部独特动作区分。

### 4.2 骨骼动画

Axmol 提供 Spine 扩展，但实际使用前必须确认编辑器/Runtime 许可、网页包体、版本锁和移动端性能。内容合同只保存通用 Skeleton/Animation ID，不让 Gameplay 依赖 Spine 类型。

骨骼、附件、蒙皮和约束有平台预算；远处/群众降低更新率或切 Sprite 烘焙。Root Motion 只作视觉参考，规则位移在 Runtime。

## 5. 场景与图集

横版子地区按 Cell 组织：背景远景、中景结构、可玩层、前景遮挡、交互/碰撞、特效与音频区。可重复构件实例化，独特地标保留轮廓。

图集默认 2048²；4096² 只有目标浏览器/设备验证、包体和纹理内存允许时使用。UI、角色、环境、特效按过滤模式、色彩空间和生命周期分组。

传输格式与 GPU 压缩在 F1 Render Spike 锁定。不能假设所有浏览器支持同一压缩扩展；清单按能力选择资源变体，始终有受预算的兼容版本。

## 6. 音频

总线：Master、Voice、Music、Ambient、SFX、Combat、UI、Accessibility。首次用户手势后再恢复 Web Audio，处理标签页静音/挂起。

- 短 SFX 驻留/批量解码，长音乐/语音流式。
- 编码格式通过浏览器能力探测选取，清单提供兼容变体。
- 环境声按地区/时间分层，不下载整区长立体声。
- 声纹玩法提供视觉/触觉等价。
- 语音按行 ID、语言、版本、时长、响度和许可管理。

## 7. VFX 与 Shader

特效先服务读招、命中、证据和环境状态。透明过绘是网页 2D 主要风险之一：限制全屏雾、叠层粒子和大面积半透明，低档减少装饰但保留关键提示。

Shader 数量、编译时间和变体进入构建报告。WebGL2 context 恢复后所有 GPU 资源可由 CPU/包缓存重建；不能把唯一状态存在纹理/FBO。

## 8. 内容分包

```text
shell                 HTML/CSS/JS/Service Worker
runtime               WASM + loader
boot                   字体、错误 UI、序章最小资源
core_shared            通用战斗/UI/音频
region_jiangnan_*
region_longquan_*
region_huizhou_*
locale_zh_cn
quality_standard/high
```

Boot 不预载三地区。子地区进入前后台下载依赖，安全枢纽显示真实进度。哈希 URL 永久缓存，新 manifest 切换版本；旧版本只在新版本完整验证后清理。

Service Worker 预缓存仅限壳和最小启动资源。地区包按玩家进度/网络空闲缓存，不在首次访问消耗数百 MiB。

## 9. 下载与启动预算

F1 起始目标（Brotli/实际传输口径）：

| 项目 | 目标 | 硬门/说明 |
| --- | ---: | --- |
| HTML/CSS/JS Shell | ≤ 1 MiB | 不含 source map |
| 主 WASM | ≤ 12 MiB | 超出需模块/依赖审查 |
| Boot + 首段内容 | ≤ 22 MiB | 与 Shell/WASM 合计首访 ≤35 MiB |
| 冷启动可交互（20 Mbps） | ≤ 8 s | P95 测试网络；显示分阶段进度 |
| 回访缓存启动 | ≤ 3 s | 不强制联网等 manifest 超时 |
| 单个子地区增量包 | 目标 ≤ 25 MiB | 按真实路线/弱网验证 |
| 单语言语音包 | 独立下载 | 不阻断无语音基础体验 |

Source map、调试符号和高质量源资产不进入玩家 CDN 路径。

## 10. 帧与内存预算

桌面推荐档 1080p/60 FPS；低档 30 FPS。Gameplay 始终 60 Hz。

| 项目 | P95 目标 | 硬门 |
| --- | ---: | ---: |
| 总可见帧 | ≤ 16.7 ms | 连续长帧必须诊断 |
| Gameplay Tick | ≤ 3.5 ms | ≤ 5 ms |
| Axmol 更新/提交 | ≤ 4 ms | ≤ 6 ms |
| DOM 更新 | ≤ 2 ms/批 | 不允许每帧大树 diff |
| 单次内容提交 | ≤ 4 ms | 不出现 >100 ms 长任务 |
| 活跃 WASM heap | ≤ 512 MiB | ≤ 768 MiB |
| JS heap（壳/UI） | ≤ 96 MiB | ≤ 160 MiB |
| 估算 GPU 纹理 | ≤ 384 MiB | ≤ 512 MiB |
| 自动存档主线程 | ≤ 2 ms | 序列化/IndexedDB 异步 |

浏览器实现和设备差异大，报告必须注明浏览器、版本、硬件、DPR、质量档和是否 Pthreads。

## 11. 场景起始预算

- 同时高威胁敌人 ≤12；横版可读性通常更低。
- 活跃高逻辑实体 ≤80；群众使用低频/抽象代理。
- 每帧 Axmol draw call 目标 ≤500；同材质/图集合批。
- 可见骨骼角色按平台和屏幕尺寸分更新率。
- 粒子、灯光、遮罩和 RenderTexture 有显式数量/像素预算。
- DOM 常驻可交互节点目标 ≤300；长列表虚拟化。
- 每个 Cell 定义最坏战斗、最快穿越、归位变体和 context restore 场景。

这些数值由 F1 压力样片校准，未校准不等于无预算。

## 12. 内存与生命周期

WASM32 线性内存增长不一定能无代价回收给浏览器，因此从启动就重视峰值和碎片：

- 地区 arena/对象池在离开时整体重置。
- 大型解压缓冲复用并限并发。
- 图集上传完成后按策略释放 CPU 副本，但 context restore 所需数据可从 Cache/包重读。
- 资源引用图和所有权可视化；未使用包/Cell 按 LRU 与安全水位卸载。
- 30–60 分钟循环游玩后回枢纽，WASM/JS/GPU 指标应回到稳定平台线。
- 页面隐藏停止渲染/音频和非必要预取，不在后台膨胀队列。

## 13. 质量档与多端

Standard/High 控制 DPR、背景层、阴影/光效、粒子、动画更新率和纹理。浏览器默认不直接使用物理 4K 内部分辨率；按 GPU/屏幕/帧率选择渲染比例。

Windows 原生可增加更高资源档，但内容/碰撞/玩法不变。移动端使用独立图集/音频/触控档，必须从同一源资产生成，不能手工维护“手机版图”。

## 14. Git LFS 与许可

Git LFS 跟踪不可再生源资产和大二进制；可确定生成的图集/缓存不进 Git。启用前锁定托管配额、备份、客户端版本和文件锁策略。

每个资产有 `license.yaml`。字体、音频、Spine Runtime/编辑器、素材包和 AI 生成资产必须在构建时产出许可报告；未确认的参考图不进入发布包。

## 15. 性能工作流

1. 模板选择 Web 性能档。
2. 内容工作台实时显示下载、解压、WASM/JS/GPU 内存和 draw call。
3. 每子地区维护冷缓存、热缓存、弱网、最密战斗、快速穿越、长时回环六条路线。
4. CI 在固定浏览器/硬件跑 trace 和回放，比较趋势。
5. 超预算先改生命周期、包边界、图集/合批和设计，再微优化 C++。
6. 优化后复测可读性、DOM 辅助和 Native 对照。

发布保留每地区 Web 性能报告、网络瀑布、缓存状态和长时内存曲线。

## 16. 视觉方向冻结边界

1.0 基线是“高清手绘/赛璐璐轮廓的东方幻想横版”，不是严格像素画、写实 3D 或直接复刻《造梦西游》的角色造型。借鉴的是清楚的横版阅读、低进入门槛和长期内容结构；人物比例、服饰、器物、色彩和 UI 必须形成《天工渡》自身语言。

- 角色轮廓先表达职业/武器/动作方向，再添加纹样和工艺细节；缩小到战斗尺寸仍能辨认。
- 环境以可走层、危险层、可交互工事、前景遮挡和远景叙事分层；高细节背景不能吞掉读招。
- 工灯证据使用“材料/受力/工序”的形变、线向、纹理和声光，不统一涂成魔法荧光。
- UI 使用温纸、墨、木、金属与灯色；可读文本在干净底层上，仿旧纹理只作边缘装饰。
- 江南主色倾湿润青灰、靛蓝、油纸暖色；龙泉为铁灰、炉红、矿砂金；徽州为墨黑、石白、松绿。关键敌我/危险色不随地区语义漂移。
- 真实工艺参考需登记来源；概念图只作方向，不把未授权参考直接描摹进资产。

正式风格在 G2 前用同一“雨夜试伞”构图制作 2–3 套小样，按读招、角色识别、生产成本、压缩后画质和多端缩放选择；选择后记录为美术基线，不能让每地区变成不同画风。

## 17. F1 最小资产集合

F1 只生产足以测试管线和性能的代表样本：主角一套灰盒/近最终轮廓、沈砚一套核心 NPC、漏雨伞偶和褪色纸鹭两个敌人家族样本、四时伞祟阶段样本、镇口/作坊两个 Cell kit、基础 HUD/字体、雨/风/证据/VFX、一段环境音乐和必要 SFX。

每项至少有 source、import、license、runtime、preview 和预算记录。可以使用占位音画，但必须清楚标记、可由锁定工具替换，并禁止携带未授权网络素材进入对外构建。F1 用这组样本校准：角色屏占比、逻辑单位→像素映射、图集页、动画方案、透明过绘、音频格式和单子地区包上限；校准前不批量制作 24 名核心 NPC 最终动画。

## 18. 动作与命中表现合同

每个战斗动作的源资产标记 anticipation/contact/recovery/loop 视觉帧和脚底锚点，但 Gameplay 的 startup/active/recovery Tick 是唯一规则。导入报告比较视觉接触点与规则命中窗，偏差超阈值阻断预览；修动画或数据，不能在 Presentation 偷补伤害。

命中停顿、相机、闪白、音效和震动由 `FeedbackTag` 组合，并有减少闪烁/减震/静音替代。Boss 阶段关键读招至少使用轮廓/运动、声音/字幕和地面/环境中两种通道；颜色/音高不能是唯一答案。
