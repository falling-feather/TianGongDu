当前处于 **F0：设计与工程基础**。本阶段交付 00–19 文档体系、ADR、1.0 机器可读内容目录和自动化规模校验。下一阶段是 **F1：引擎烘焙**，目标不是堆内容，而是证明正式架构可以支撑一条完整生产生命线。

里程碑按证据放行，不按“做了几天”放行。时间估算只用于资源安排，不能替代出口条件。

## 3. F1 引擎烘焙任务包

F1 必须同时证明以下链路，不能只做一场漂亮战斗：

1. `Tgd.Contracts`、`Tgd.Runtime`、`Tgd.Gameplay` 与 Godot 表现适配的依赖边界可编译检查。
2. 内容 JSON 经 Schema 校验和烘焙后加载；错误 ID、循环依赖和版本不匹配会清楚失败。
3. 角色编辑器能从模板创建 NPC，配置日程、关系、对话、任务和表现引用，并在独立沙盒预览。
4. Boss 编辑器能配置阶段图、招式、工灯证据、失败/重试和归位结局。
5. 一段 15–30 分钟灰盒含探索、NPC 日程、对话选择、采集/工艺、普通战斗、Boss、失败重试、存档读档和结算。
6. Windows Debug 与 Release 构建能在干净目录启动；日志、崩溃信息和版本号可追踪。
7. 固定 60 Hz 模拟在 30/60/120/144 FPS 渲染条件下得到一致结果。

## 5. 功能完成定义

每项功能需提供七类证据：

1. **玩家价值**：解决了哪个体验问题，玩家如何触发和感知。
2. **契约证据**：Schema、API、稳定 ID、迁移与错误提示。
3. **系统证据**：正常、失败、取消、重试、存档、跨区和边界路径。
4. **内容证据**：至少一个教程用例、一个正式用例、一个压力/变式用例。
5. **测试证据**：单元、契约、集成或场景测试，以及人工游玩记录。
6. **性能证据**：目标硬件和压力场景数据；不能只看开发机平均帧率。
7. **维护证据**：文档、编辑器提示、日志、诊断和可回滚性。

缺任意一类只能称为“实现中”或“验证中”，不能标记完成。

## 6. 内容完成定义

### 角色

一个核心 NPC 完成时必须有稳定 ID、角色功能、三阶段关系弧、日程、两种以上交互、任务链、地区状态响应、演出/语音占位、存档恢复、编辑器预览和本地化检查。

### Boss

一个 Boss 完成时必须有两阶段以上、独立读招、普通战斗与工灯定印的交叉、场景机制、失败重试、辅助选项、归位裁定、战后世界变化和性能压力测试。只调高血量不构成阶段。

### 子地区

一个大型子地区完成时必须有入口/出口与流式边界、主路径和回环、至少一条可选路线、NPC/事件日程、探索语法、战斗与非战斗节奏、地标、环境叙事、归位前后状态、导航和性能证据。

### 任务链

至少包含三次有意义状态变化，支持中断恢复和失败替代，不把所有选择收束成同一句台词；结果至少影响关系、资源、空间、后续任务或地区状态之一。

## 8. 风险前置

当前最高风险按顺序为：

1. Godot .NET 与目标规模下的场景流式、工具稳定性和构建耗时。
2. 内容团队是否能真正通过模板生产，而不是回到手写脚本。
3. 两套战斗系统是否互相增强，而非变成战斗暂停后的答题 UI。
4. 每地区 5 小时是否由独特内容支撑，而非重复模板换皮。
5. 24 个 NPC 的日程、关系和跨区状态是否导致组合爆炸。
6. 资产体量、Git LFS、导入缓存和发布包差分是否可控。

每个风险必须在最早能廉价失败的里程碑验证。F1 不通过，不能用更多美术和剧情掩盖架构问题。

## 9. 计划变更规则

- 数量下调、地区删除或系统合并必须修改 00、02、07 和目录 JSON，并通过产品决策审查。
- 里程碑可以拆小，但出口门禁不能被删除；延期优先缩减并行工作，不牺牲基础设施。
- 已完成事项在验证后移动到 03；02 不保留长篇历史流水账。
- 每两周只更新状态、风险和下一门禁；不把日常任务列表塞进本文件。
