# ADR-0001｜桌面原生优先与 Godot .NET 宿主

> 状态：Superseded by ADR-0005
> 日期：2026-07-10
> 决策者：项目技术/产品基线
> 取代：旧归档分支 ADR-0001（Axmol + C++ + WebAssembly/Web 优先）

## 背景与问题

旧项目用无依赖 Canvas Demo 和 Axmol/WASM 设想快速验证了江南世界、伞器和内容数据，但没有形成适合大型端游的场景、动画、内容预览、角色/Boss 编辑器与稳定发布链。项目目标已经明确为长期更新、地区级 5 小时内容、复杂战斗和大量 NPC 的 PC 单机游戏。

继续围绕 Web 首发优化会让包体、线程、渲染、资产工具和桌面体验长期受错误目标约束；完全自研引擎又会把团队拖入编辑器、导入器和平台基础设施。

## 决策驱动因素

- Windows 原生 60 FPS 和 2.5D 表现能力。
- C# 强类型核心、测试和工具共享。
- 可扩展编辑器，能建设角色/Boss/任务/地区生产工具。
- 场景、动画、音频、UI、资源导入和打包有成熟基础。
- 开放源码、可审查、无按项目收入绑定的引擎许可风险。
- 小团队能在 F1 低成本验证，失败时尚未批量生产内容。
- 内容包/运行时加载与离线发布能力。

## 备选方案

### A. 延续 Axmol + C++ + WebAssembly

优点：继承旧设想、C++ 性能和 Web 发布。缺点：现有项目并无真实 Axmol Runtime，编辑器/内容工作流仍需大量自建；Web 约束与桌面目标冲突。拒绝作为主线，保留旧分支供研究。

### B. Godot .NET

优点：Windows .NET 官方构建、C#、场景/UI/动画/音频/导入/插件一体、开放源码、PCK 能力。风险：大型 C# 项目、工具稳定性、流式和目标视觉组合必须实测。

### C. Unity 或 Unreal

二者都有成熟生态和大项目案例。当前团队需要更轻的开放源码宿主、C# 生产和可控编辑器基础；切入成本、项目结构和依赖策略并不明显优于可验证的 Godot 路线。若 F1 失败，可重新评估，不做永久否定。

### D. 自研 C++ 引擎

能完全控制性能与工具，但在当前阶段会把大量人力投入非差异化基础设施，延迟内容验证。拒绝。

## 决策

正式客户端以 Windows x64 原生为首发目标，使用 Godot 4.7 stable .NET 作为窗口、输入、渲染、音频、场景、资源、UI 和编辑器宿主；C# 承担 Contracts、Runtime、Gameplay、表现适配与编辑器插件。

Web 不再是正式客户端目标，只可用于官网、宣传和独立机制沙盘。Godot 具体补丁、.NET TFM、表现组合和最低硬件在 F1 技术烘焙后锁定。

## 正面后果

- 可以把精力投入玩法、内容工具和地区生产，而非从零建编辑器。
- 核心/工具共享 C# 类型、校验与测试。
- Windows 原生不受浏览器线程、文件系统和包体优先约束。
- 场景与资产工作流对内容团队可见，支持插件化编辑器。
- 开源引擎允许审查、定制或在必要时维护补丁。

## 负面后果与风险

- 旧 Web Runtime 代码不迁移，只迁设计意图和内容。
- 团队需要建立 Godot C#、场景树、导入与性能能力。
- Godot 编辑器资源与 C# 热重载可能在大项目下暴露摩擦。
- PCK 只是运输格式，仍需自建包清单、签名和版本契约。
- 若表现路线最终偏向高复杂全 3D，Godot 性能/工具需更严格验证。

## 验证与放行门

F1 必须完成：Windows Debug/Release 导出、15–30 分钟全链灰盒、固定 60 Hz、角色/Boss 编辑器雏形、内容烘焙、存档迁移、Cell 流式、目标风格压力场景、PCK 挂载与回滚。

未满足 10/14 的硬门时禁止扩充最终内容。失败报告必须区分项目实现问题与引擎能力问题。

## 迁移

1. 旧项目在 `codex/archive-legacy-web-v1` 冻结。
2. 从世界、角色、地区和玩法意图重建新 Schema；不复制旧文件路径和运行时接口。
3. 先建分层核心和一条江南灰盒，再批量迁移内容。
4. 旧概念图仅作参考，权利/资产管线审查后决定是否重制。

## 回滚

若 F1 证据显示 Godot 无法满足性能、工具、导出或稳定性硬门，在大规模内容制作前新建 Superseding ADR，比较 Unity/Unreal/定制宿主。Contracts、内容 Schema 和纯 C# Gameplay 应尽量保持可迁移，Godot 表现层可替换。
