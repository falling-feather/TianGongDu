> 文档定位：长期技术地图，说明本地环境、目录边界、模块职责、扩展规则和开发约定。

> 维护者：项目维护者

## 项目概览

当前仓库同时保留两条线：

- M0 浏览器原型：`prototype/web-demo/`，无外部依赖，用 Canvas、HTML、CSS 和少量 JavaScript 验证首屏体验。
- 长期工程方向：C++20/23 + Axmol + WebAssembly/WebGL2 + 自研 Heavenwrights Runtime + Python 内容工具链。

当前实现服务于验证，不替代最终 Runtime。任何新增规则都应优先保持“规则层、内容层、表现层分离”。

2026-07-06 起，江南地区的长期方向以 `13_江南区域新版大地图设计方案.md` 作为 `07_江南雨巷纵切规格.md` 的扩展优化依据。`07` 中已经验证的纸伞门径、雨巷叙事、稳定 ID、M1 白盒流程和数据拆分继续保留；`13` 将它们升级为 6 个大型子区域、内部点位、长期自给循环和不少于 5 小时的正式章节目标。后续不是完全推翻 `07`，而是把 `07` 的 11 个节点迁移为 `13` 大地图中的主线点位、教学点位或回访点位。

## 本地环境

需要：

- Node.js：运行开发服务器和 Node 测试。
- Python：运行内容校验工具。

常用命令：

```bash
npm test
npm run validate:content
npm run dev
```

`npm run dev` 默认服务地址为：

```text
http://127.0.0.1:4174/
```

### `prototype/web-demo/`

用于快速验证江南雨巷首屏、纸伞 HUD、简化战斗、风息资源、夜市选择和归位反馈。这里可以写表现原型，但不要把长期数据契约只藏在前端代码里。

当前 Web demo 通过开发服务器的 `/content/...` 只读路由加载江南 M1 内容包，并在“江南M1”面板中展示主线、内容包摘要、资源生产面板和 Boss 状态。原型可以保留必要的坐标、镜头、碰撞和绘制逻辑，但主线步骤、子地区、NPC、Boss 和资源数量必须来自 `content/` 数据或能追溯到对应内容 ID。

采用 `13` 的大地图方案后，Web demo 的定位调整为“内容库消费者和白盒验证器”。除非任务明确是短期表现验证，否则不要继续在 `prototype/web-demo/` 中硬编码新的江南人物、地块、建筑、采集点、Boss 或支线；新增内容应先进入内容库和 Schema，再由 demo、未来编辑器或 Runtime 读取。

复杂视觉表现优先通过清晰的渲染函数和数据 ID 边界隔离。当前断桥雨幕 Boss 使用 Canvas 2.5D 多层雨帘、裂伞核和伞骨弧线实现；若后续引入 Three.js 或 Axmol 高精度渲染，不能把玩法判定、任务推进或内容引用绑死在具体渲染库里。

M1 资源生产面板是程序/策划/美术/音频的白盒协作面板，不是正式资源浏览器。它只展示 `AssetPackDef`、`AudioPackDef`、Boss 阶段和资源事件 ID 的当前接入状态；真实 PNG/Aseprite/GLB/音频文件路径、缩略图、帧规格、锚点、碰撞辅助体、压缩格式和混音参数必须进入后续资源 manifest，而不是写死在 Canvas 绘制函数或场景脚本中。

M1 染纸晒场工艺白盒当前在 Web demo 中以 `CRAFT_RECIPES` 和 `world.craft` 验证，但长期接口不能停留在前端临时对象里。每个工艺配方后续都必须能提升为内容层 `CraftRecipeDef` 或等价数据对象，并至少包含：稳定 `id`、入口 `interactionId`、目标 `subregionId/buildingId`、材料需求、授予的 `traitId` 或 `assetVariantId`、四维变化、工艺观测字段、玩法参数、NPC 评价目标和资源事件引用。短键材料如 `rainPaper`、`tungOil` 只允许作为 demo adapter，内容层和正式 Runtime 应映射到稳定物品 ID。

`world.craft` 是工艺结果状态，不是渲染状态。当前字段消费规则为：`paperTension`、`oilCoverage`、`dryingState`、`trialScore` 用于白盒面板和 NPC/任务反馈；`bonuses.guardWind` 影响架伞回风，`bonuses.gatherDamage` 影响收风伤害，`bonuses.dashWindDiscount` 影响借风风息消耗，`bonuses.wetResistance` 影响雨幕压迫/湿滞承受；`umbrellaFill` 只是 Web 原型的 Canvas 占位外观，正式美术必须通过 `variantId` 和资源 manifest 读取。

工艺入口必须由 `InteractionDef + SubregionDef/BuildingDef` 触发。Web demo 可保留侧栏按钮作为白盒调试入口，但可玩路径必须能通过玩家靠近 `subregion.jiangnan_rain_alley.dye_paper_court` / `building.jiangnan_rain_alley.dye_court` 并按 `F` 打开。染纸晒场相关美术需求至少包括 `tile.dye_paper_court`、`building.jiangnan_rain_alley.dye_court`、`asset.variant.umbrella.blue_lantern`，真实贴图、帧规格、锚点和碰撞辅助体仍归后续资源 manifest 管理。

### `content/`

文本化内容数据是策划、程序和 AI 协作的公开接口。每个内容对象必须包含：

- `schemaVersion`
- `id`
- `displayNameKey`
- `tags`
- `designerNote`

引用其他对象时使用稳定 ID。中文显示文本放入 `content/localization/zh-CN.json`，不要写死在逻辑数据中。

江南内容数据当前有两层含义：`07` 形成的是可运行纵切和第一版投放数据基线，`13` 提供的是在该基线上的章节级扩容目标。迁移期间允许旧 `subregion.*` 与新 `large_area.* / point.*` 并存，但任何新内容都必须能说明它属于哪个大区、点位、玩法循环和验收目标。

江南 M1 内容包已经开始按 `07_江南雨巷纵切规格.md` 拆成稳定集合：

- `content/regions/jiangnan_rain_alley.json`：地区入口和索引，负责引用子地区、任务、NPC、敌人、Boss、建筑、交互、资产包和音频包。
- `content/subregions/jiangnan_rain_alley_subregions.json`：子地区拓扑、生产角色和连接门槛。
- `content/quests/jiangnan_rain_alley_quests.json`：主线、主线步骤、支线和回访互动。
- `content/npcs/rain_alley_m1_npcs.json`：关键 NPC 的功能身份、情感身份、行程和四维倾向。
- `content/buildings/jiangnan_rain_alley_buildings.json`：交互建筑和空间功能。
- `content/interactions/jiangnan_rain_alley_interactions.json`：玩家触发、交互类型、效果和反馈目标。
- `content/enemies/rain_alley_m1_enemies.json`：敌人和机关草案。
- `content/bosses/rain_alley_boss.json`：Boss、阶段、归位选择和结局回声。
- `content/assets/jiangnan_rain_alley_assets.json`：美术资源组和变体。
- `content/audio/jiangnan_rain_alley_audio.json`：BGM、环境音、音效和混音规则。
- `content/large_areas/jiangnan_large_areas.json`：`13` 章节化大地图的 6 个 `LargeAreaDef`、36 个 `PointDef`、`07` 子地区迁移表和 3 条自给循环。
- `content/gather_nodes/jiangnan_gather_nodes.json`：19 个 `GatherNodeDef`，承接采集资源、刷新规则、风险引用和美术占位接口。
- `content/recipes/jiangnan_craft_recipes.json`：11 条 `CraftRecipeDef` 和 5 个 `TraitDef`，承接材料、工位、产物、失败产物和玩法修正。
- `content/encounters/jiangnan_encounters.json`：12 个 `EncounterDef`，承接敌群、机关、学习目标、失败处理、奖励和回访规则。
- `content/editors/jiangnan_editor_templates.json`：人物、大区/点位、采集点、建筑交互、工艺配方、敌人与遭遇、任务流程、资源引用 8 类标准编辑器模板。

当前浏览器原型只消费这些内容库中用于白盒验证的切片；完整 Runtime 仍以后续 Axmol/资源服务为准。这些文件同时作为长期 Runtime、策划拆表、资源生产和 QA 验收的共同契约。

`AssetPackDef` / `AudioPackDef` 的 `owner`、`status`、`referenceIds`、`usageIds`、`requiredCount` 和 `requiredStates` 都是生产接口字段，不是备注。Demo、Runtime、工具链只能通过稳定 ID 绑定资源需求；不能通过中文名、临时文件名、当前 Canvas 函数名或具体第三方渲染库对象建立长期引用。

`RegionDef` 在章节化扩展期同时索引 `subregionIds/coreSubregionIds`、`largeAreaIds/pointIds/corePointIds` 以及 `largeAreaSetIds/gatherNodeSetIds/craftRecipeSetIds/encounterSetIds`。前者维持 `07` 纵切和 M1 回归，后者服务 `13` 的大地图章节和正式内容库加载。`chapterTargetDurationMinutes` 只表达章节目标体量，不改变 `recommendedDurationMinutes` 的 M1 纵切验收范围。

`LargeAreaDef` 不能只做地图壳。进入章节 Demo 覆盖范围的大区，最低应具备：至少 1 条任务或回访目标、2 名本地 NPC、1 个可定位建筑、1 个可触发交互、1 个资源或自给循环入口、1 个遭遇或危险反馈。若暂时没有完整对象，`PointDef.contentRefs` 只能使用明确命名空间的占位 ID，并在 `AssetPackDef` 或后续 manifest 中记录资源生产需求。

湖田外圩是 `13` 对 `07` 的第一块新增大区补强样例。当前内容库已把它从单纯点位扩展为外圩水闸补给主线段、外圩水闸支线、田娘阿穗/童舟/巡圩差役三名 NPC、外圩水闸/小舟坞两栋建筑、调水闸门/修补渡舟/巡圩清障三类交互，以及芦苇/香草/鱼篱/油籽采集点、舟修材料/香草药贴/外圩简餐配方、水闸压力/潮锚魇遭遇，并同步进入 `RegionDef.corePointIds`、`questIds`、`npcIds`、内容集合索引与美术资源占位清单。

资源占位 ID 允许出现在 `asset_group.referenceIds`、`audio.usageIds`、`PointDef.contentRefs` 和编辑器模板中，但必须属于明确命名空间，例如 `tile.*`、`ui.*`、`resource.*`、`travel.*`、`record.*`、`daily_commission.*`。`gather.*`、`recipe.*`、`trait.*`、`encounter.*` 已进入正式内容库，不能再只靠前缀白名单放行；新增这四类引用时必须能在对应集合中解析。占位 ID 在生产状态进入 `mvp` 或 `locked` 前，必须落成正式内容对象或 manifest 条目；校验器和测试可以先对白名单命名空间放行，但不允许中文名或临时文件名成为引用。

## 内容库与编辑器策略

`content/` 是长期内容源头，编辑器只是内容库的结构化读写入口。编辑器输出必须是可 diff、可校验、可代码审查的文本数据，例如 JSON、Markdown 或后续约定的 CSV；本阶段不建立只能由某个工具打开的二进制内容源。

标准化编辑器按对象职责拆分：

| 编辑器 | 主要对象 | 必填能力 |
| --- | --- | --- |
| 人物编辑器 | `NpcDef`、`DialogueProfileDef`、`NpcScheduleDef` | 身份、功能角色、情感角色、所在大区/点位、行程、信任/四维倾向、服务、任务钩子、对白分组和立绘/小人资源引用。 |
| 大区/地块编辑器 | `LargeAreaDef`、`PointDef`、`ConnectionDef` | 6 大区、内部点位、坐标/层级、可达门槛、采集点、建筑、敌群、天气/水位/时辰修正、回访状态和 tileset 资源引用。 |
| 采集点编辑器 | `GatherNodeDef`、`ResourceDropDef`、`RefreshRuleDef` | 所属大区/点位、产出资源、刷新规则、风险遭遇、资源占位和回访状态。 |
| 建筑与交互编辑器 | `BuildingDef`、`InteractionDef` | 建筑功能、所属点位、交互类型、触发条件、消耗、产出、反馈目标、关联 NPC、UI 提示和失败/取消处理。 |
| 工艺与配方编辑器 | `CraftRecipeDef`、`MaterialDef`、`TraitDef` | 材料需求、工位/建筑、交互入口、工序观测字段、`resultIds` 产物、质量结果、战斗/探索修正、返修和缺陷输出。 |
| 敌人与遭遇编辑器 | `EncounterDef`、`EnemyDef`、`BossPhaseDef` | 遭遇所属大区/点位、敌群、机关、精英/小 Boss 定位、机制学习点、刷新/回访规则、奖励、失败处理和阶段切换条件；`EncounterDef` 主写 `content/encounters/`，敌人/Boss 只作为引用对象。 |
| 任务与流程编辑器 | `QuestDef`、`QuestStepDef`、`ChoiceDef`、`OutcomeDef` | 主线、支线、日常委托、回访步骤、前置条件、四维变化、解锁、奖励、天工录记录和验收路径。 |
| 资源引用编辑器 | `AssetPackDef`、`AudioPackDef`、`AssetRefDef` | 资源组、用途 ID、生产状态、预览、真实 manifest 引用、锚点/碰撞辅助体和音频混音规则。 |

`jiangnan_editor_templates.json` 的模板契约包含 `draftContract`、`draftFields` 和 `draftSeed`。当前 Web demo 的编辑器实现只在内存中维护每类模板的草稿，按字段生成表单、即时校验必填项/ID 格式/跨内容引用，并导出包含 `editorValidation` 的 JSON 文本。它不直接写盘、不绕过代码审查、不把编辑器状态当作最终内容源。正式写入流程仍然是：编辑器导出文本 -> 人工/AI 审查 -> 写入 `content/` -> 运行校验与测试 -> Git 提交。

编辑器控件应尽量来自内容库已有枚举或索引：大区来自 `LargeAreaDef`，点位来自 `PointDef`，NPC/建筑/交互/采集/配方/特质/遭遇/任务/资源引用来自对应集合，`owner/status` 使用统一选项。新增字段前应先确认是否可复用现有对象字段；只有当内容对象稳定后，才把字段加入模板和校验器。

内容对象继续保留统一基础字段：`schemaVersion`、`id`、`displayNameKey`、`tags`、`designerNote`、`status`、`owner` 和 `sourceNote`。中文显示名、NPC 台词、天工录条目和 UI 文案进入本地化或文本资源；逻辑引用只使用稳定英文 ID。

`image_example/` 只作为美术方向参考库和 prompt/参考图整理区，不是运行时资源目录。参考图、拆图清单、tileset、sprite、BGM、SFX、VFX 和 Three.js/GLB 预览都必须通过 `AssetPackDef`、`AudioPackDef` 或后续 manifest 绑定稳定内容 ID，并记录来源、用途和生产状态。

`AssetPackDef` 的长期字段契约为：`regionId`、`assetGroups[].assetType`、`requiredCount`、`requiredStates`、`referenceIds`、`variants[].sourceQuestId`、`status`、`owner` 和 `sourceNote`。后续 manifest 承接真实文件路径、预览图、帧规格、锚点、碰撞辅助体、压缩格式、LOD、GLB/glTF 版本和授权来源；内容对象只引用 manifest key，不直接引用具体文件路径。

采用 `13` 后，推荐的数据层级为：

```text
RegionDef
-> LargeAreaDef
-> PointDef
-> BuildingDef / InteractionDef / NpcDef / GatherNodeDef / EncounterDef
-> CraftRecipeDef / TraitDef / QuestDef / AssetRefDef / AudioRefDef
```

迁移规则：

1. `07` 的 11 个节点不删除，先映射为 `13` 六大区内的 `PointDef` 或主线切片。
2. 新增人物、地块、建筑、采集、工艺、遭遇、敌人或支线时，先补内容库对象和稳定 ID，再接入 demo 或 Runtime。
3. `prototype/web-demo/` 可以保留临时 adapter，但不能把新内容的唯一来源写在 `main-v2.js` 或 Canvas 绘制函数中。
4. 编辑器首版可以是内部 Web 表单或 JSON 模板生成器，但必须复用同一套 Schema、校验器和本地化表。
5. Three.js、Canvas 或未来 Axmol 只消费 `AssetRefDef` 与地块/遭遇数据，不拥有玩法规则和任务推进。
6. 当前浏览器 demo 的“大地图”和“编辑器”页只做内容库预览、内存草稿、字段校验和导出文本，不提供写盘 API；正式写入必须经过内容校验和代码审查。
7. `PointDef.contentRefs` 与 `LargeAreaDef.coverage.*Ids` 只负责索引和摘要；`GatherNodeDef`、`CraftRecipeDef`、`TraitDef`、`EncounterDef` 的掉落、产物、失败处理、敌群、奖励和美术占位字段必须留在各自内容库。

编辑器导出后必须通过校验：Schema 必填字段、稳定 ID 唯一性、跨文件引用、本地化 key、任务可达性、大区连接图、资源 manifest 引用、四维反馈目标和废弃对象引用。复杂渲染或 3D 预览可以作为编辑器能力，但不能替代文本化数据源。

### `source/`

未来 Axmol/C++ 接入入口。长期分层建议：

- `app/`：应用入口、平台启动和主循环接入。
- `framework/`：渲染、输入、音频、平台与资源接口适配。
- `runtime/`：固定逻辑帧、实体、碰撞、战斗、技能、工艺、NPC、任务与世界状态。
- `game/`：具体游戏装配、地区加载和表现层 glue code。

### `tools/`

当前包含静态开发服务器和 Python 内容校验。工具可以服务原型，但输出和校验规则应尽量沉淀为可重复命令。

### `third_party/`

第三方依赖占位。Axmol 等大型依赖不要直接复制进仓库；正式接入前先在 `doc/adr/` 写明来源、版本、许可、构建链路和复审条件。

## 架构规则

### 核心规则不写死在场景回调

**适用范围**：战斗、技能、任务、NPC、工艺、地区四维和世界状态。

**规则**：
1. 规则层应由 Runtime 或数据驱动对象承载。
2. 表现层只负责动画、镜头、特效、音频和 UI 呈现。
3. 原型阶段可以先写在 JavaScript 中，但需要能迁移为数据或 Runtime 规则。

**原因**：避免首屏 demo 变成不可复用的一次性脚本。

**验收**：
- 新增玩法能说明它对应的内容数据、Runtime 规则或表现层职责。

### 内容文件是公开 API

**适用范围**：`content/**/*.json` 和未来 CSV / Markdown 内容资产。

**规则**：
1. 所有内容对象有稳定 ID 和 Schema 版本。
2. 交叉引用使用 ID，不使用文件路径或中文显示名。
3. 显示文本进入本地化表。
4. 修改内容后运行 `npm run validate:content`。

**原因**：保证内容可批量生成、审查、diff 和校验。

**验收**：
- 新增或修改 JSON 通过内容校验，并能被测试覆盖到基本结构。

### RegionDef 是地区加载根

**适用范围**：地区包、任务、子地区、NPC、Boss、建筑、交互、资源和音频。

**规则**：
1. 地区入口必须通过 `entrySubregionId`、`subregionIds`、`questIds`、`npcIds`、`enemySetIds`、`bossIds`、`buildingSetIds`、`interactionSetIds`、`largeAreaSetIds`、`gatherNodeSetIds`、`craftRecipeSetIds`、`encounterSetIds`、`assetPackIds` 和 `audioPackIds` 建立索引。
2. 新增地区内容时，先补 RegionDef 索引，再补集合文件；不要只在原型脚本里增加孤立对象。
3. `coreSubregionIds` 表示当前阶段最小可玩闭环，不能混入纯远期占位。
4. `07` 纵切/M1 阶段的江南核心子地区为水路入镇、水巷主路、柳伞匠作坊、竹林小径、夜市回廊、旧桥下层和断桥雨幕；屋檐上层作为借风回访验证节点。
5. `13` 大地图阶段应在 RegionDef 或后续 RegionPackage 中增加 `largeAreaIds`、`pointIds`、`largeAreaSetIds`、`gatherNodeSetIds`、`craftRecipeSetIds`、`encounterSetIds` 和 `selfSupplyLoopIds`，用来承载烟雨竹巷、河埠水市、青篁湿林、染纸晒场、风铃旧桥、湖田外圩六大区。

**原因**：未来 Axmol Runtime、内容校验器和调试面板都应能从一个地区入口找到完整内容包。

**验收**：
- `tests/content-m1.test.mjs` 能确认 RegionDef 索引到 M1 内容包和 7 个核心子地区。
- 迁移到 `13` 时，测试或校验器能确认 `07` 的核心节点均被映射到六大区内的点位，且没有孤立 NPC、建筑、交互或资源引用。

### 内容集合按职责拆分

**适用范围**：江南 M1 及后续地区包。

**规则**：
1. 子地区只描述空间节点、连接、生产角色和主要引用。
2. 建筑只描述空间功能和承载对象，不直接写具体触发逻辑。
3. 交互负责触发、前置、效果、反馈目标和四维变化。
4. 任务负责步骤、完成条件、解锁和奖励，不直接承担战斗数值。
5. NPC 至少包含功能身份、情感身份、行程和四维倾向；NPC 不能只是功能按钮。
6. Boss 阶段使用 `bossphase.*` ID，阶段切换条件和归位结果必须独立于美术表现。
7. 资产和音频以资源组或音频包描述需求，真实文件路径留给后续 manifest。
8. `assetGroup.referenceIds` 必须指向子地区、建筑、NPC、敌人、Boss、技能、UI 或结局等稳定内容 ID；`audio.usageIds` 和 `mixRules.from/to` 必须能回连到技能、机关、子地区、Boss 或其他音频 ID。
9. Boss、技能、机关和地区状态可以声明所需 VFX/SFX/BGM ID，但表现层只消费映射，不把阶段逻辑或任务推进写进素材对象。
10. 当前 Web demo 的资源事件层只记录“应该触发的资源 ID”，不播放不存在的正式资源；接入真实音频和高精度渲染时应替换为资源服务或 manifest 查询，而不是改动任务和战斗规则。

**原因**：大型端游项目需要让策划、程序、美术、音频和 QA 能并行修改不同职责面，减少同文件冲突和口径漂移。

**验收**：
- M1 + `13` 首轮扩展内容对象能覆盖 11 个子地区、6 个大区、36 个点位、18 名 NPC、11 个主线步骤、9 条支线、12 个建筑、至少 8 类交互、19 个采集点、11 条工艺配方、5 个伞具特质、12 个遭遇、9 类敌人、1 个 Boss、4 个 Boss 阶段、3 个归位结果。
- Web demo 的“江南M1”面板能展示资产组、音频项、Boss 阶段 requiredState、白盒接入状态、采集/配方/遭遇数量和当前资源事件 ID。

### 四维选择必须有回声目标

**适用范围**：`choice.*`、`interaction.*.market_choice`、Boss 归位和地区结算。

**规则**：
1. 四维变化只写在明确选择或任务效果中，不能散落在表现层。
2. 每个关键选择至少列出 3 个反馈目标，例如师傅评价、居民台词、归位视觉状态。
3. 折中路线等条件选择必须写清楚前置任务或特质。
4. Boss 归位后的结果用 `outcome.*` 对象表达，不把结局分支写死在 Boss 阶段脚本里。

**原因**：选择后果需要被 UI、剧情、关卡和 QA 同时追踪。

**验收**：
- 夜市选择包含 `choice.rush_order`、`choice.full_process`、`choice.material_compromise`，并在交互定义中列出四维变化和 3 个反馈目标。

### 重大技术取舍写 ADR

**适用范围**：引擎、构建链路、第三方依赖、内容 Schema、存档、资源包、服务器协议等长期决策。

**规则**：
1. 决策前记录背景、备选方案、最终决策、影响和复审条件。
2. 已接受的 ADR 不随意改写历史；需要变更时新增替代 ADR。

**原因**：让长期维护者知道为什么做出当前选择。

**验收**：
- `doc/adr/` 中能找到对应决策记录。

## 调试与排障

- 页面无法打开：确认 `npm run dev` 正在运行，并访问 `http://127.0.0.1:4174/`。
- 内容校验失败：先看报错文件，再检查是否缺少必填字段或 JSON 语法错误。
- 原型交互无响应：确认页面获得焦点，检查键位是否为 `WASD`、方向键、`Shift`、`Space`、`Q`、`E`、`R`、`F`。
- 文档口径冲突：入口以 `README.md` 和 `doc/README.md` 为导航准绳；已完成事实以 `CHANGELOG.md` 为准；未来任务以 `02_项目规划与验收.md` 为准。

## 发布前开发检查

1. 运行 `npm test`。
2. 运行 `npm run validate:content`。
3. 启动 `npm run dev` 并进行浏览器人工冒烟。
4. 更新 `CHANGELOG.md`，记录完成内容、验证证据和剩余风险。
5. 如果改动了长期技术边界，补充或更新 `doc/adr/`。
