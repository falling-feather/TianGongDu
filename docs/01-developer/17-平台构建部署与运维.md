# 17｜平台构建、部署与运维

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Windows-first Baseline；系统要求由 F1 硬件实验锁定
> 最后更新：2026-07-10

## 1. 首发平台

1.0 首发目标为 Windows x64 原生桌面。最低 Windows 版本、CPU/GPU/内存和安装包格式在 F1 硬件实验后写死；在此之前不对外公布猜测配置。

Linux/macOS 保持纯核心可编译和 Godot 可导出意识，但不进入 1.0 硬门。Web 不作为游戏发行平台。

## 2. 构建配置

| 配置 | 用途 | 特性 |
| --- | --- | --- |
| EditorDebug | 日常开发 | 编辑器工具、断言、详细诊断、热重载 |
| GameDebug | 独立调试 | 调试 HUD、可选开发命令、未优化符号 |
| Profile | 性能实验 | 近 Release 优化、性能标记、内部符号 |
| Release | 候选/正式 | 剔除开发入口、签名、受控日志、完整优化 |
| ServerTools | 控制面/离线工具 | 不包含 Godot 客户端资产 |

Release 构建不能仅靠 UI 隐藏调试能力；调试程序集、命令和资源应不进入产物或需要内部签名授权。

## 3. 版本与可追溯性

每个构建嵌入：语义版本、Git commit、dirty 标记、构建 ID、UTC 时间、渠道、Godot 版本、.NET SDK/TFM、内容 API、核心包哈希和签名 key ID。

玩家设置页和诊断包可复制这些信息。可执行版本与内容包版本分离，见 12。

## 4. 可重现构建

- 引擎二进制、导出模板、.NET SDK、NuGet、Node/Python 工具均锁版本和校验和。
- 构建机从干净 checkout 和空输出开始；缓存只是优化。
- 依赖恢复使用锁文件和受控源，禁止构建中下载未锁的“最新”。
- 烘焙产物规范化时间戳和路径；相同输入的内容清单/数据哈希一致。
- 发布在 GitHub Actions 或受控 runner 完成，不从开发者编辑器直接上传。

## 5. Windows 产物布局

```text
TianGongDu/
├── TianGongDu.exe
├── runtime/                 Godot/.NET 运行依赖
├── packages/                签名核心与地区包
├── licenses/                第三方通知和许可
├── tools/                   仅必要诊断/修复工具
└── version.json
```

可写数据不放安装目录：

```text
%LOCALAPPDATA%/TianGongDu/
├── profiles/
├── saves/
├── mods/
├── cache/
├── logs/
├── crashes/
└── updates/
```

实际 Godot `user://` 映射在 F1 固定并测试便携/多用户/只读安装目录。

## 6. 构建流水线

1. 验证来源 commit/tag 和工作树。
2. 恢复锁定工具与依赖，验证校验和。
3. 编译纯核心、工具、Godot C# 项目并运行快速测试。
4. 全量校验和确定性烘焙内容/资产。
5. Godot Headless 导入，导出 Windows 包。
6. 执行场景、存档和性能冒烟。
7. 生成 SBOM、许可证、版本清单和符号包。
8. 签名可执行、安装包、更新清单和内容包。
9. 在隔离 Windows VM 安装、启动、游玩短路径、升级、回滚、卸载。
10. 发布到目标渠道并保存不可变产物/哈希。

任何失败使整个发布无效；不能手工替换某个文件后继续沿用旧签名/清单。

## 7. 安装与卸载

安装器检查架构、磁盘空间、写权限和必要运行库。支持修复安装、无静默捆绑、清楚显示版本与许可。安装路径可选，但不要求管理员权限是优先目标，最终由签名/渠道实验确定。

卸载默认保留存档、设置和模组，并明确询问是否删除用户数据。卸载不能删除安装目录外无法验证归属的文件。

## 8. 启动与安全模式

启动参数最小且文档化：日志级别、窗口/渲染安全设置、禁用可选包/模组、指定诊断输出。Release 不提供任意内容跳关、写状态或加载程序集参数。

连续启动崩溃触发安全模式建议：低图形档、禁用模组和可选包、重建缓存、保留原设置备份。安全模式不自动保存覆盖正常档。

## 9. 日志与崩溃

日志按大小/天数轮换，默认不包含个人文件路径、输入文本和完整存档。崩溃包含构建信息、脱敏堆栈、最近结构化事件、性能摘要和包列表；玩家可在发送前查看类别并选择。

符号文件与公开产物分开受控保存，按构建 ID 可定位。正式版本的符号和发布产物保留周期覆盖支持生命周期。

## 10. 发布渠道

- **internal**：团队每日构建，允许开发诊断。
- **nightly**：自动全量，通过基础门禁，不承诺存档兼容。
- **preview**：受邀测试，明确版本和存档政策。
- **stable**：签名正式版，严格兼容和回滚。

不同渠道使用独立清单和签名策略；Preview 不能覆盖 Stable 安装/存档而不确认。渠道切换生成新 Profile 或显式迁移。

## 11. 发布与回滚

发布前创建不可变 Release Record：commit、工作流、依赖、包、哈希、签名、测试、迁移、已知问题和负责人。上线先小范围/人工验证清单，再扩大。

回滚包括撤回更新清单、恢复上一可执行/包集合、检查存档兼容、发布说明和诊断。若安全问题要求强制更新，离线仍能启动的边界由安全评审决定并透明说明。

## 12. 数据备份

本地存档保留轮换与关键里程碑。云存档若未来加入，不替代本地备份。发布团队保存：Git tag、构建定义、工具镜像、源资产、LFS 对象、发布包、符号、签名元数据（不含私钥）和迁移黄金档。

定期做“从空机器恢复并重建旧正式版”演练，防止只有当时那台机器能构建。

## 13. 平台适配边界

平台 API 通过接口：文件路径、窗口、输入设备、成就、云存档、商店、崩溃、URL。Gameplay 不知道 Steam/Epic/Windows 等渠道 SDK。首个渠道实现也不能成为全局单例散落在代码中。

引入渠道 SDK 必须通过 18 的许可/安全审查、离线降级和 CI Stub。

## 14. 运维指标

发布后关注更新成功/回滚、启动失败、崩溃率、存档迁移失败、包哈希错误、性能档分布和支持请求类别。只有经同意数据进入服务；没有遥测时通过测试渠道与用户主动诊断收集。

指标触发阈值、值班和事故流程在服务实际建立时加入运维 runbook，不提前制造无人维护的复杂平台。
