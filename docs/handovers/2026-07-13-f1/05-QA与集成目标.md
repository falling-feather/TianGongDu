# QA 与集成目标｜奖励事务阶段门和 F1 证据骨架

> 任务：`F1-QA-01A`；Owner：QA/发布负责人；必审：技术负责人、Web、Core、Gameplay、UX、内容。
> 分支：在主开发交付奖励里程碑 commit 后，从该 commit 建独立 worktree `codex/f1-qa-evidence`。
> 写入边界：测试/证据脚本、16 测试门禁和验证记录；发现产品缺陷先报告，不顺手修改产品代码。
> 最后更新：2026-07-13。

## 目标

为奖励原子事务建立可重复的本地阶段门，并把现有分散的 Native、Node、Web、浏览器和视觉证据整理成可由新对话复现的 F1 证据骨架。遵循“模块本地快速测试 → 阶段候选提交 → 集中三浏览器/GitHub 调试”，不重新把每个微提交都拖进远端 CI。

## 启动前置

主开发必须提供：候选 commit、干净暂存范围、构建命令、两种奖励 Stable ID、预期 Operation 数、QA 只读导出和已知限制。若仍只有当前未提交 WIP，QA 可先做只读测试计划，但不得把该状态记录为阶段证据。

## 第一阶段矩阵

1. 合同/Native：Profile Progress、Storage Coordinator、F1 结算收据；正常、重复、非法、冲突、失败和重试。
2. Web Single：干净构建、Guest 新档、两结算 claim、原子提交、刷新恢复、重复路线不重复权益。
3. IndexedDB：慢写/失败、配额、Head CAS、离线、损坏导出；UI 只在事务确认后显示已保存。
4. 生命周期：隐藏/恢复、失焦、浏览器重启、WebGL context restore；不得导致重复 claim 或粘键。
5. 浏览器：先本地 Chrome 冒烟，候选稳定后一次性跑 Chrome/Edge/Firefox；记录页面错误、控制台错误、请求失败和精确 build/commit。
6. 回归：现有 7 Beat 两路线、Boss 四阶段、失败重试、选择驱动入口/增援/动作、10,000 Tick 回放校验保持。
7. UI/可访问：保存状态文案、键盘路径、200% UI、DOM/Canvas 对齐；隔离 UI 原型只做自身证据，不代替正式宿主。

## 证据标准

- 每份报告绑定 commit、build ID、工具链、浏览器/OS、命令、结果、截图/trace 和已知例外。
- 从标题/Guest 入口复现，不使用未公开调试按钮跳过关键路径；QA 导出只观察状态。
- 截图检查画面尺寸、透明/黑屏、HUD 遮挡和关键状态；DOM 断言不能替代 WebGL 视觉检查。
- 失败证据记录最小复现、期望/实际、严重度和 Owner；没有产品 Owner 批准不在 QA 分支修业务代码。
- GitHub 失败先判断产品回归、环境漂移或证据脚本问题；阶段门以外不做大规模远端追逐。

## 交付

- 一键或分层可运行的本地测试清单/脚本，以及奖励事务报告模板。
- 候选提交的验证报告和缺陷列表；能通过则更新 16/02，完整事实再进入 03。
- 给技术负责人的放行建议：`pass`、`pass with bounded exception` 或 `fail`，附精确原因与回退点。
- 为后续 60 分钟真人盲测准备观察表：逐 Beat 合格/闲置/失败重试、首次理解点、卡点、输入设备和内容重复感。

## 非目标

不在本任务建设正式 CD、监控、云同步或全平台发布；不因远端 runner 波动阻塞主开发的本地模块实现，也不把一次性 Phaser 样例的测试结果算入正式 F1 放行。
