# ADR-0002｜分层 C# 核心与混合组件模型

> 状态：Accepted
> 日期：2026-07-10
> 决策者：项目技术基线

## 背景与问题

长期更新会产生大量 NPC、敌人、Boss、器物、任务和地区状态。如果按 Godot 场景脚本或深继承树实现，角色生命周期、战斗、存档、编辑器和表现会互相耦合；如果直接采用全量高性能 ECS，又会提前承担 Archetype、查询、调试和复杂领域状态机的成本。

需要一种既能通过模板组合内容，又能让复杂 Boss/任务保持可读，并能脱离 Godot 测试和迁移的模型。

## 决策驱动因素

- 新角色无需专属基类/脚本即可组合日程、关系、任务、战斗和职业。
- 固定 60 Hz、确定性测试、回放和存档。
- Gameplay 不依赖 Godot Node/Resource。
- 高频战斗数据可高效处理，低频复杂状态可清楚建模。
- 编辑器能从组件/模板自动生成表单和预览。
- 未来替换表现宿主不会重写世界与玩法真相。

## 备选方案

### A. Godot Node/Scene 为权威

制作直接，但存档、无表现实体、固定模拟和单元测试困难；内容脚本容易散落。拒绝。

### B. 深继承 OOP

早期简单，组合爆炸后会出现 `CombatMerchantQuestNpc` 等类型和大量覆盖。拒绝。

### C. 全量 Archetype ECS

高性能、数据导向，但项目实体规模并非主要瓶颈，复杂任务/Boss/关系需要额外抽象，工具与团队成本高。暂不采用；局部数据布局可借鉴。

### D. 混合组件 + 顺序系统 + 领域状态机

组合灵活，可按热点优化，同时保留复杂逻辑可读性。选择。

## 决策

建立 `Tgd.Contracts ← Tgd.Runtime ← Tgd.Gameplay ← Tgd.Presentation.Godot` 的单向分层。Runtime 实体由稳定身份和可选数据组件组成，系统按固定阶段执行；任务、Boss 阶段、内容包和存档迁移使用显式领域对象/状态机。Godot Node 是可重建视图。

跨系统变化通过命令、领域事件和只读快照。物理/导航等引擎能力通过接口适配，结果规范化后提交模拟。

## 正面后果

- 同一角色模板可以组合多种功能，无需新继承类型。
- 纯逻辑可快速测试、回放和在 Headless 工具中运行。
- Presentation 崩溃/卸载可重建，不污染权威状态。
- 高频组件未来可改成连续存储，无需改变内容契约。
- 复杂 Boss 和任务仍能用清楚的图与状态机表达。

## 负面后果与风险

- 需要维护实体/视图绑定和物理适配，代码比直接写 Node 多。
- 命令/事件边界若设计差会产生样板和延迟调试。
- “混合”可能被滥用成无规则折中，需依赖检查和系统阶段守卫。
- 确定性不自动成立，浮点、物理排序、时间和随机仍需约束。

## 验证与放行门

- 同一战斗回放在 30–144 FPS 得到同一权威校验和。
- 从模板创建 NPC，不新增 C# 类即可拥有日程、关系、任务和专业交互。
- 一个多阶段 Boss 完全由通用系统 + 受控扩展构成，无 ID 特判。
- Cell 卸载/重建后持久实体状态和表现一致。
- 依赖分析器阻止 Runtime/GamePlay 引用 Godot。

## 迁移与回滚

F1 先实现最小实体仓、命令/事件/Tick 和表现绑定。若性能不足，优先优化热点组件存储和查询，而不是推翻 API。只有明确实体规模与 CPU 证据显示 Archetype ECS 必要时，才新建 ADR 迁移内部存储；内容和系统接口保持。
