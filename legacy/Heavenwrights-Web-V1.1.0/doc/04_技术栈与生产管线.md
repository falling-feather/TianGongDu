# 《天工渡 / Heavenwrights》技术栈与生产管线

> 版本：0.5  
> 日期：2026-07-01  
> 状态：当前有效技术栈  
> 决策：C++ 为主力语言，Axmol 为客户端唯一主框架，WebAssembly / WebGL2 为首要发布目标。

## 1. 最终选择

```text
C++20/23
+ Axmol Engine
+ Heavenwrights Runtime 自研游戏层
+ Python 内容工具链
+ 文本化内容数据
+ 轻量调试与校验工具
+ WebAssembly / WebGL2 浏览器发布
```

一句话：

> 用 Axmol 解决成熟 2D 框架、跨平台、WebAssembly/WebGL2、渲染、输入、音频和资源基础；用自研 Heavenwrights Runtime 承载动作 RPG 规则；用 Python 工具链承载 AI 友好的内容生产。

## 2. 为什么定 Axmol

Axmol 是当前最符合《天工渡》长期利益的 C++ 框架。

- C++ 主体，符合团队熟悉度和性能诉求。
- 面向 2D 游戏，比 SDL 这类底层库更接近可直接做游戏的起点。
- 不依赖大型图形化编辑器，适合代码优先和 AI 协作。
- 继承 Cocos2d-x 生态经验，适合 2D 动作、像素、关卡、UI、特效和资源管理。
- 官方支持桌面、移动、Xbox UWP，并提供 WebAssembly 预览方向，底层 Web 渲染走 WebGL2。
- 能让团队把精力放在动作、工艺、门径、NPC 和内容管线，而不是从零搭窗口、渲染、音频和基础场景系统。

不再把 MonoGame、SDL3、SFML、raylib、Unity、Godot、UE 作为当前主线候选。

## 3. Axmol 使用边界

Axmol 是基础框架，不是游戏规则层。

Axmol 负责：

- 应用启动。
- 主循环接入。
- 窗口与平台基础。
- 输入与手柄基础。
- 2D 渲染与 WebGL2 发布基础。
- 纹理、字体、音频、资源加载。
- 场景壳与 UI 基础。
- 粒子、shader、动画表现的底层支持。

自研 Heavenwrights Runtime 负责：

- 固定逻辑帧。
- 2D / 类 2.5D 角色控制。
- 攻击盒 / 受击盒。
- 无敌帧 / 取消帧 / 弹反窗口。
- 战斗结算。
- 门径与技能系统。
- 工艺系统。
- 敌人与 Boss AI。
- NPC、任务、对话。
- 地区四维与世界状态。
- 存档与云存档数据结构。
- 内容数据加载、热重载和校验。

硬规则：

> 不把核心战斗、技能、任务、NPC、工艺、四维状态写死在 Axmol `Node` 或具体场景回调中。

## 4. 架构分层

```text
App Layer
  Axmol app entry / platform / window / loop

Framework Adapter
  tgd_platform / tgd_render / tgd_audio / tgd_input

Heavenwrights Runtime
  core / entity / animation / collision / combat / ability
  craft / npc / quest / world_state / save / ui

Content Layer
  JSON / CSV / Markdown / packed assets

Tools Layer
  Python validators / importers / packagers / generators
```

原则：

- Runtime 不直接依赖具体场景文件。
- Simulation 状态可序列化、可测试、可回放。
- Presentation 只负责动画、镜头、特效、音频和 UI 表现。
- Content 文件是公开 API，必须可被 AI 读写和校验。

## 6. 主循环

采用固定逻辑帧，渲染可插值。

```text
AxmolPollPlatform
MapInputActions
AccumulateFixedTime
FixedUpdateSimulation
  UpdatePlayerIntent
  UpdateAI
  UpdateAbilities
  UpdateMovement
  ResolveCollision
  ResolveCombat
  EmitSimulationEvents
UpdatePresentation
  UpdateAnimation
  UpdateCamera
  UpdateParticles
  UpdateAudioEvents
RenderFrame
```

规则：

- 战斗判定只在固定逻辑帧中发生。
- 输入记录可以保存并回放。
- 存档保存 Simulation 状态，不保存 Axmol 节点。
- 渲染对象可以随时重建，不影响游戏真状态。

## 7. 内容数据

开发期：

- JSON：地区、子地区、任务、建筑、交互、技能、敌人、Boss、Boss 阶段、NPC、工艺、地区状态、音频和过场引用。
- CSV：批量数值、材料、掉落、成长曲线。
- Markdown：策划说明、剧情草案、审查记录。

发行期：

- Python 工具将内容打包为压缩资源包。
- 保留 manifest 用于版本校验和资源更新。

所有内容对象必须有 `schemaVersion`、`id`、`displayNameKey`、`tags`、`designerNote`。显示文本进入本地化表，不写死在逻辑数据里。

## 8. Python 工具链

Python 是官方辅助语言，用于内容生产和自动化，不进入核心战斗运行时。

| 工具 | 作用 |
| --- | --- |
| `content validate` | 校验所有内容数据 |
| `content diff` | 对比内容和数值改动 |
| `content pack` | 打包发行资源 |
| `stage import` | 导入 Tiled / LDtk 或自研关卡数据 |
| `hitbox preview` | 预览攻击盒、受击盒、动画帧 |
| `dialogue preview` | 预览对话分支 |
| `replay verify` | 回放输入，复现战斗问题 |
| `schema docs` | 生成内容 Schema 文档 |

## 9. 服务器体系

客户端主力是 C++ / WebAssembly，第一阶段服务器不强行 C++。

建议：

- 运营服务：Python / FastAPI。
- 数据库：PostgreSQL 或 SQLite 起步，正式服再迁移。
- 缓存：Redis 后置。
- 协议：HTTP + JSON 起步。
- 实时联机：后置，必要时单独评估 C++ 服务。

第一阶段服务器功能：

- 账号登录。
- 公告。
- 版本检查。
- 资源 manifest。
- 云存档。
- 兑换码。
- 活动配置。
- 排行榜。
- 日志与崩溃上报。

## 10. WebAssembly 发布边界

《天工渡》当前改为 Web 端优先，但 Axmol 的 WebAssembly 能力仍应被视为必须先验证的发布链路。官方资料显示 Axmol 支持 WebAssembly 预览，渲染后端包含 WebGL2；因此第一阶段不能跳过浏览器技术冒烟。

Web 端必须优先验证：

- 浏览器加载首包。
- WebAssembly 初始化耗时。
- WebGL2 渲染、像素缩放、shader、粒子。
- 键鼠、手柄和触屏可选输入。
- 浏览器音频解锁与音量控制。
- 存档：LocalStorage / IndexedDB / 云存档同步。
- 资源包下载、缓存、版本 manifest。
- CDN 部署与跨域策略。
- 崩溃、日志和性能上报。

架构规则：

- 平台能力通过 `framework/platform` 封装。
- 文件系统通过资源接口访问，不直接依赖本地文件路径。
- 存档通过统一 Save API。
- 网络通过统一 Client API。
- 渲染、输入、音频不在 Runtime 业务层直接调用。
- 桌面端构建保留为开发调试、线下展示和未来封装方向，但不是第一发布目标。

M0 技术冒烟必须包含一个浏览器可运行版本。若 WebAssembly 路线在加载、音频、资源包或性能上达不到最低可玩标准，应先解决技术链路，不继续扩展内容。

## 11. AI 接手规范
