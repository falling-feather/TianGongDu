# 10｜技术架构与依赖规则

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Accepted Baseline；F1 通过兼容性实验锁定补丁版本
> 最后更新：2026-07-10

## 1. 架构目标

技术架构服务于三个十年尺度问题：能否持续增加地区而不复制核心代码；能否让设计者和 AI 通过模板安全生产；能否升级引擎、内容和存档而不整体报废。

优先级依次为：正确边界 → 可观测/可测试 → 内容吞吐 → 性能 → 局部便利。短期少写几行代码不能交换长期不可迁移的耦合。

## 2. 技术基线

| 层 | 选择 | 版本/策略 | 原因 |
| --- | --- | --- | --- |
| 客户端/编辑器宿主 | Godot .NET | 4.7 stable；补丁经 Bake 升级 | 原生 Windows、开放源码、C# 支持、编辑器插件与 PCK 能力 |
| 核心语言 | C# | .NET 10 LTS 工具链候选；引擎 TFM 由 F1 实验锁定 | 强类型契约、成熟测试/分析器、适合工具与玩法共享 |
| 渲染 | Godot Forward+ | Windows Vulkan 主路径，质量档回退 | 2.5D 场景、动态灯光与桌面性能目标 |
| 内容源 | JSON + JSON Schema | Draft 2020-12 | 可审查、可生成编辑器、可独立校验和迁移 |
| 表现资源 | `.tscn`/`.tres` + 导入产物 | 只保存表现组合和资产引用 | 利用 Godot 编辑能力，不让其成为业务契约源 |
| 离线工具 | Python + Node.js | 版本在 F1 锁文件固定 | DCC/批处理与轻量 CI 校验各取所长 |
| 资产版本 | Git LFS | F1 仓库烘焙后启用并锁客户端版本 | 避免大型二进制污染 Git 对象库 |
| CI | GitHub Actions | Windows 为发布主门，跨平台跑纯逻辑 | 与远端仓库、PR 和发布流程一致 |

Godot 4.7 stable 官方提供 Windows .NET 构建与 .NET 导出模板；C# 官方文档覆盖桌面平台。版本事实来源见 09。

## 3. 解决方案与程序集

目标解决方案：

```text
TianGongDu.slnx
├── Tgd.Contracts              net10.0 或经 Bake 确认的共享 TFM
├── Tgd.Runtime                仅依赖 Contracts
├── Tgd.Gameplay               依赖 Runtime + Contracts
├── Tgd.Presentation.Godot     依赖 Gameplay + Godot
├── Tgd.Editor.Godot           依赖 Contracts/校验器 + Godot editor API
├── Tgd.Tools.Content          离线内容工具
├── Tgd.Server.Contracts       可选控制面 DTO
└── *.Tests                    按层镜像
```

若 Godot 4.7 的引擎项目不能直接采用 .NET 10 TFM，允许引擎面对的程序集使用 Godot 官方支持 TFM，纯核心程序集多目标构建。F1 必须以实际 Windows 导出、热重载、编辑器插件、AOT/裁剪风险和 NuGet 兼容性决定，不凭版本号猜测。

## 4. 依赖规则

### 4.1 允许

```text
Contracts ← Runtime ← Gameplay ← Presentation.Godot ← Client
      ▲          ▲          ▲
      └──────── Editor/Tools ┘（只通过公开契约和测试工具）
```

- Contracts 只依赖 BCL 与极少量经审计的基础包。
- Runtime 可定义引擎无关的时间、事件、存档、包、实体和世界服务。
- Gameplay 可定义具体规则，但不知道 Node、AnimationTree、Control 或场景路径。
- Presentation 把快照和事件映射到 Godot，并把玩家输入转换成命令。
- Editor 使用与构建相同的 Schema 和校验器。

### 4.2 禁止

- Runtime/GamePlay 引用 Godot 类型或加载 `res://` 路径。
- Presentation 直接改生命、任务或关系状态。
- 地区内容程序集反向修改核心系统内部字段。
- Editor 为了方便维护第二套字段默认值或验证规则。
- 服务端包被单机玩法强制依赖。
- 跨层使用全局静态 Service Locator 隐藏依赖。

CI 使用项目引用图检查和禁用命名空间分析器守住边界。

## 5. Godot 宿主职责

Godot 负责：进程/窗口、平台输入、音频、渲染、物理查询适配、场景流式宿主、UI、动画、资源导入、编辑器插件和导出。

Godot 不负责：权威战斗状态、任务真相、关系数值、内容版本、存档身份、跨包依赖和随机种子。Godot 节点可以拥有表现生命周期，但实体即使暂时没有可见节点也可存在。

### 5.1 场景树建议

```text
AppRoot
├── PlatformServices
├── GameSessionHost
│   ├── WorldPresentation
│   ├── EntityViews
│   ├── AudioWorld
│   └── CameraRig
├── UiRoot
│   ├── HudLayer
│   ├── PanelLayer
│   ├── ModalLayer
│   └── AccessibilityLayer
└── DebugRoot (dev only)
```

每个 `EntityView` 通过 EntityId 绑定快照，不把 Node 引用传回核心。表现丢失可重建，权威状态不丢失。

## 6. 启动顺序

1. 读取可执行版本、渠道与启动参数。
2. 初始化日志、崩溃目录和平台服务。
3. 验证核心包清单、签名、哈希和兼容范围。
4. 加载本地化与设置；安全模式可禁用可选内容包。
5. 构建 Contracts/Runtime 服务图，注册确定性时钟和随机源。
6. 读取存档头并执行迁移计划；失败时不覆盖原档。
7. 创建 GameSession，载入初始世界状态和场景 Cell。
8. Presentation 绑定快照，UI 进入可操作状态。

每一步有结构化耗时、错误代码和恢复建议。Release 启动失败必须能进入诊断/安全模式，而不是黑屏。

## 7. 时间、线程与所有权

- 权威模拟固定 60 Hz，在单一逻辑所有权线程顺序执行。
- 渲染按显示帧率插值，不改变规则时间。
- 内容 I/O、解压、哈希、导航烘焙和资产准备可后台执行；提交到世界状态时回到安全同步点。
- 不在后台线程触碰 Godot 场景树，除非官方 API 明确线程安全。
- 长任务通过取消令牌、进度和超时管理；退出/切档时必须可收束。
- 事件队列设上限和背压；不能因一个坏内容无限增长。

## 8. 状态与服务

核心服务通过构造注入：`IGameClock`, `IRandomSource`, `IContentRegistry`, `IEventBus`, `ISaveStore`, `IWorldStreamer`, `ILogger`。Session 范围服务随存档/新游戏销毁；App 范围只保留平台与静态缓存。

不使用可写全局单例承载游戏状态。Godot Autoload 只做宿主桥接，并持有一个明确可替换的 App 容器。

## 9. 错误策略

- 内容作者错误在导入/CI 阶段硬失败，并定位文件、JSON Pointer、ID 与建议。
- 可选包损坏时隔离该包并进入安全模式；核心包损坏时拒绝进入游戏。
- 运行时不变量违反在开发构建抛出，在 Release 记录错误并尝试最近安全回退；不能静默继续污染存档。
- 存档迁移先复制、后迁移、再原子替换；失败保留原件和诊断包。
- 玩家错误与开发错误使用不同提示，不把堆栈暴露给玩家。

## 10. 可观测性

结构化日志至少包含 build、session、save、package、region、subregion、entity 和 correlation ID。性能标记覆盖启动、包加载、Cell 流式、固定 Tick、AI、战斗、UI、存档和 GC。

开发 HUD 显示 CPU/GPU 帧、Tick、实体数、活动 AI、流式队列、内存/显存水位、事件队列和当前内容版本。遥测对外发送必须遵守 15/18 的同意与最小化规则。

## 11. F1 技术烘焙矩阵

| 实验 | 成功条件 | 失败后的决策 |
| --- | --- | --- |
| .NET/导出 | 编辑器、Debug/Release 导出、测试和插件稳定 | 调整 TFM 或 Godot 补丁；重大失败重新评估宿主 |
| 2.5D 渲染 | 目标场景 60 FPS、正确排序/灯光/相机 | 调整表现组合或质量目标 |
| 场景流式 | Cell 后台准备、切换无长卡顿、内存回落 | 缩小 Cell、改加载策略或资源格式 |
| 编辑器插件 | 角色/Boss 模板可创建、校验、预览、保存 | 将复杂编辑移到外部工具但保留同一契约 |
| PCK 内容包 | 哈希/签名、依赖、加载、回滚工作 | 使用自定义包容器或整包更新 |
| 固定模拟 | 30–144 FPS 结果一致、回放可复现 | 收紧物理适配与随机/时间边界 |

F1 结束前不批量生产最终资产。

## 12. 替代方案边界

旧 Axmol + C++/WASM 路线已被主线否决：它证明了 Web 原型，却没有形成适合大型端游的成熟编辑器、内容预览和长期生产闭环。Unity、Unreal、自研引擎并非“差”，但当前团队规模、开放源码需求、C# 生产效率和 2.5D 目标下，Godot 是更合适的可验证起点。

若 F1 证据表明 Godot 在流式、工具、平台或性能上无法达到硬门禁，团队可以在内容规模扩大前重开 ADR。引擎选择是经验证的决策，不是身份认同。
