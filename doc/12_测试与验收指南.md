```bash
npm test
```

## 内容校验

运行：

```bash
npm run validate:content
```

当前最低要求：

- 每个 JSON 文件能被解析。
- 每个内容对象包含 `schemaVersion`、`id`、`displayNameKey`、`tags`、`designerNote`。
- `schemaVersion` 为整数。
- `tags` 为数组。

后续应补充：

- 跨文件 ID 引用检查；当前 Node 测试已要求 `gather.*`、`recipe.*`、`trait.*`、`encounter.*` 命中正式内容库，Python 校验器后续应吸收同等规则。
- 本地化 key 缺失检查已在 Node 测试中覆盖，后续需要并入 Python 内容校验器。
- 数值上下限检查。
- 循环引用检查。
- 关卡可达性检查。
- 文档和内容 JSON 的中文字段可读性检查，防止误存为乱码编码。
- 编辑器草稿协议检查：模板必须声明 `draftContract`、`draftFields`、`draftSeed`，导出的草稿应带 `editorValidation`，写入内容库前必须经过测试和代码审查。
- 资源契约检查：`requiredStates`、`referenceIds` 命名空间、`variants[].sourceQuestId`、资源占位白名单和 `image_example/` 不被运行时代码直接引用。

## 浏览器人工冒烟

启动：

```bash
npm run dev
```

打开：

```text
http://127.0.0.1:4174/
```

最低检查路径：

1. 页面加载成功，Canvas 画面非空，右侧 HUD 和底部技能栏可见。
2. `WASD` 或方向键能移动角色。
3. `Shift` 或 `Space` 能冲刺并消耗气力。
4. `Q` 能架伞，风息能随架伞积累。
5. `E` 能触发收风，影响附近敌人。
6. `R` 能借风越行，靠近风口时能改变位置和记录。
7. `F` 能执行上下文交互，包括检视雨线、请教师傅、检校风口和归位。
8. “接急单”和“完整工序”能改变工牒四目数值与事件记录。
9. “重置试炼”能恢复初始状态。
10. “江南M1”面板能展示美术资源组、音频资源、Boss 阶段 requiredState、白盒接入状态和当前资源事件 ID。
11. “大地图”面板能展示 6 个大区、48 个点位、8 条跨区路线、`07` 迁移说明、自给循环和“内容覆盖”标签；地图中应出现节点地图、路线线条和路线明细。点击湖田外圩后能看到巡圩望棚、雨粮垛、水闸、舟修岸、芦苇、舟修材料、外圩潮锚魇等正式对象名；点击河埠水市或烟雨竹巷后能看到潮尺埠、船灯排、雨线试伞场、巷口委托板等新增点位，说明扩容内容已经进入 demo 呈现。
12. “编辑器”面板能展示 8 类编辑器模板、字段表单、校验结果和 JSON 草稿；采集点、工艺配方、遭遇模板的 `outputTargets` 指向正式内容库，修改字段后预览同步，切换模板后草稿按模板保留，导出文本包含 `editorValidation`，重置按钮只影响当前模板的内存草稿，不写回磁盘。
13. 使用架伞、纸刃、伞旋、收风、借风、风铃铆、Boss 显形和 Boss 归位时，资源事件 ID 会随操作更新。
14. 推进 M1 至“竹林选材”后，地图能显示并进入“染纸晒场”；玩家靠近晒场工艺热点时，`F` 上下文交互能切到背包工艺面板并触发 `interaction.jiangnan_rain_alley.craft_paper_oil` 资源事件。
15. 在工艺面板选择“蓝染稳色”或其他可用配方后，材料数量减少，`quest_step.jiangnan_rain_alley.paper_and_oil` 完成，`wind_bell_lane` 解锁，工艺四项指标、NPC 评价、四维数值、伞面颜色和当前资源事件同步变化。
16. 桌面宽屏和窄屏视口下，HUD、技能栏、操作提示、工艺面板、大地图、编辑器和资源生产面板不互相遮挡到不可读。
17. 性能冒烟：连续停留在“大地图”“编辑器”“江南M1”任一重面板时，控制台无错误，`window.__heavenwrightsDemo.snapshot().performance.uiMaxHz` 不高于 12，`renderDpr` 不高于 1.5；移动角色时 Canvas 仍然连续绘制，但非当前抽屉面板不应被逐帧重建。

浏览器自动化可调用 `window.__heavenwrightsDemo.snapshot()`、`focusLargeArea()`、`selectEditorTemplate()` 和草稿字段调试方法做稳定断言。重点断言包括 `contentCounts.gatherNodes/craftRecipes/encounters` 非零、`focusLargeArea("large_area.jiangnan.lake_field_outer_dike")` 后详情包含正式对象名、同一编辑器草稿连续导出文本稳定，以及无效正式引用能被 `editorValidation` 报错。该 QA bridge 只操作 Web demo 内存态，不提供内容写盘能力，不能替代内容库审查和 Git 提交。

## 纵切验收口径

`07_江南雨巷纵切规格.md` 中的完整纵切验收仍是后续目标。当前 M0 只要求验证首屏表达和基础交互。进入下一阶段时，验收应至少覆盖：

- 从新游戏到短闭环结束可连续游玩。
- 不依赖开发者口头解释也能理解当前目标。
- 伞师门径与普通近战有明显差异。
- 工艺选择至少影响一个战斗参数和一个视觉或叙事反馈。
- 四维选择至少影响三个后续反馈点。
- Boss 或精英战能验证格挡、弹反、收风、借风越行中的核心组合。
