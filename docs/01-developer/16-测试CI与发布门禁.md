## 1. 目标

自动化必须同时守住 C++ 正确性、WASM/浏览器生命周期、内容契约、本地存档、云同步和发布缓存。原生测试快不等于网页可玩；浏览器能跑一遍也不等于跨设备不会丢档。

## 3. C++ 静态与单元门

- Clang/MSVC 至少两套编译器在支持矩阵构建纯核心。
- 警告视为错误；例外按规则编号和范围抑制。
- clang-tidy/自定义检查禁止跨层 include、裸 owning pointer、平台宏进 Gameplay。
- ASan/UBSan 原生夜间运行；适用时 MSan/TSan 独立实验。
- 单元测试注入时钟、随机、存储、网络和物理；不访问真实浏览器/数据库。
- 属性测试覆盖效果叠加、任务幂等、Operation 重放、合并交换/结合条件和迁移。
- 模糊测试内容包、快照、操作和 API payload 解析器。

## 5. 浏览器自动化

真实浏览器驱动完成：

- 冷缓存/热缓存启动、能力探测和加载进度。
- 键鼠、手柄模拟、DOM 焦点、全屏、DPR/缩放和窗口尺寸。
- `visibilitychange`, `pagehide`, blur/focus, reload, WebGL context lost/restored。
- IndexedDB 保存、事务失败、配额、旧 Schema、多标签写锁。
- Service Worker 首装、等待、激活、旧标签、新版本、离线和回滚。
- Web 单线程必跑；线程版只在正确头部环境跑并检查选择器。
- 控制台 error、unhandled rejection、CSP/COEP 违规和长任务为测试产物。

Chrome/Edge/Firefox 为主要 CI 矩阵；Safari 需要 macOS runner/人工门。不能用一个 Chromium 结果代表所有浏览器。

### 5.0 F1 真实移动的 QA 尝试预算

`tests/browser/web-host-lifecycle.mjs` 到达各世界交互时，必须继续通过 Playwright 键盘按键驱动真实移动；不得传送、直接改姿态、提交 Objective 或按浏览器名称放宽判定。到达容差、碰撞和 Gameplay 状态仍由现有运行时真相决定。

移动驱动不能再以一段固定墙钟作为全路线的进度真相。它在第一次权威玩家快照上计算扣除到达容差后的 X/Y 轴剩余距离，以 80 mm 保守进展量化，并按“量化格数 + 8”生成 10–256 次的有界输入尝试预算。360 mm 的短距离只获得 13 次尝试；旧 CI 从结算选择点返回 `(-10500, -600)` 的约 17,240 mm 剩余距离获得 224 次，不会因 Firefox 单次 Playwright 调用较慢而被固定 10 秒提前截断。

连续 12 次输入未刷新最佳剩余距离时必须以 `no-progress` 失败；总尝试数耗尽则以 `attempt-budget-exhausted` 失败。两类错误都要报告 `attempts / distanceQuanta / remaining / best / finalState`，既避免无限等待，也保留碰撞或路线阻塞的可审计现场。预算合同随浏览器脚本启动执行轻量断言，也可由 `node tests/browser/web-host-lifecycle.mjs --movement-budget-self-test` 独立执行；它锁定短距离不放大、旧 CI 长距离的 224 次预算和极端距离仍受总上限约束。每次真实移动还要把 `attemptsUsed / attemptBudget / minimumPositiveProgress` 写入浏览器报告，用于继续校准而非依赖一次偶然通过。

GitHub Actions run `29268579868` / job `86880234701` 的历史失败继续保留：Chrome、Edge 与此前全部构建门通过，Firefox 151 在第 6 Beat、任务尚未结算且 Operation 数为 0 时被旧 10 秒 QA 墙钟截断。此修复只消除 QA 驱动与宿主调用速度的耦合；在 exact-head 三浏览器远端证据出现前，不得据此把 G1、训练恢复、玩法时长或 F1 写成 Verified。

### 5.1 F1 authored choice 路线合同

F1 全流程浏览器脚本对 Definition 已作者化 `choice_available` cue 执行以下固定顺序，禁止以 QA hook、直接完成 Objective 或 fixture flow 缩短：

1. 移动到真实交互并按 F；等待正式 DOM choice 可见，同时断言 Objective 完成数与 selection 数均未改变。
2. 核对按钮数量以及每项 DOM `interaction/selection` 顺序与权威 `projection.choice_options` 完全一致；玩家可见 label/hint 非空且不泄露 Stable ID。
3. 从第一个真实按钮焦点开始，以 Arrow 键移动到路线目标，再用 Enter 触发正式 Action intent；不得调用 `__tgdQuestUi.dispatchAction` 代替玩家输入。
4. 只有同 generation、同 projection sequence/checksum 且 wire `messageSequence` 更大的 close，或 sequence/messageSequence 均更大的非 choice replace，才能结束 choice；随后断言 Canvas 恢复焦点、Objective/selection 各只增加 1，短等待不产生第二次推进。
5. 对没有 `choice_available` cue 的后续路线、调校和结算选择做反向断言：具体世界 F 继续推进，期间不得发布 choice event 或转移到 DOM 菜单。

每次 authored choice 保存一张包含 Canvas 与 DOM 的全页截图，并把 open/authority identity、close/replace disposition、前后 Objective/selection 和焦点状态写入报告。该脚本是自动化快跑：`beatTargetsMet=0`、`playableTargetMet=false` 必须保留，Web Host 标题、英文开发 HUD、Profile 修复按钮或 lifecycle QA 控件存在时不得标为无开发界面 player mode 或 P01–P05 真人证据。

### 5.2 F1 实际任务反馈路线合同

同一两条全 7 Beat 浏览器路线还必须从真实世界/战斗输入收集当前 1.6 的非 choice raw signal，禁止调用 QA 发布、直接构造 projection、复制 UI fixture 或以 classification 选择事件。每个检查点至少保存 wire/projection identity、source、focus Objective、polarity、Definition 派生 classification、primary/secondary 的 result owner + status + reason，以及 Active/Defeated Actor 集合。

- 已提交线索的另一作者化交互必须得到 `ignored_repeat/selection_already_committed` 且 Objective/selection/checksum 不增加。
- 两路线分别覆盖 cross-belay accepted-positive 与 quick-hitch accepted-negative；后者的负向只能来自 Definition selector，不能由 App ID 分支推导。
- 未读码鸣铃为 rejected/prerequisite_incomplete，读码后同一世界交互 accepted；拒绝尝试零推进，accepted 只跟随一次 Quest command。
- take-mark/cross-flower-line 完成后的 phase 必须来自真实相邻 Objective，并在 Encounter 激活后分别看到正确训练 Actor；不能扫描 cue/rule 推断阶段。
- 檐守反制为 accepted trigger + secondary N/A；翻花重击打错目标为 accepted trigger + rejected wrong-target outcome，且发布时新证明靶已 Active、旧训练架已移出 Active snapshot。

两条路线都要继续走过后五 Beat cue-less world choice、四阶段 Boss、双结算、奖励持久化和 replay probe。它们是自动化功能证据，不替代 Edge/Firefox 远端矩阵、200%/辅助完整矩阵、无开发界面 player mode、P01–P05 或真实分钟；当前内容恢复规则未成对时，也不能用两个独立 recovery Native 样本冒充整段失败路线。

## 6. DOM/可访问性

- 语义/ARIA/焦点图、隐藏控件、按键提示、文本溢出和本地化。
- Canvas 关键状态有 DOM 等价摘要。
- 键盘、手柄、触控布局切换不产生双输入。
- 深层菜单暂停 GameCore，关闭恢复正确 Action Context。
- 200% 浏览器/UI 缩放、无色、静音、减少动态和屏幕阅读菜单路径。
- DOM 更新频率/节点数和 layout/paint 纳入性能。

## 7. 存档测试

- IndexedDB 事务中断、刷新、浏览器崩溃、配额、存储清理提示和导出。
- 最近 N 个 SaveVersion 黄金档在 Web/Windows 读写/迁移。
- Boss 阶段、任务节点、地区归位、包缺失和内容升级往返。
- Native 导出→Web 导入、Web→Native 导入语义一致。
- 1000 次事务和 100 次页面重载后无 head/快照孤儿增长。
- 浏览器多标签只有一个写入者；接管流程不丢 Operation。

## 8. 同步服务测试

### 8.1 单元/集成

- request/operation 幂等、base revision、事务回滚和错误码。
- OIDC subject 映射、Profile 授权、设备撤销和 token 过期。
- PostgreSQL 并发更新、死锁重试、唯一约束和迁移。
- Snapshot/Operation 版本、哈希、压缩/大小限制。
- 领域合并：集合、进度、库存 claim、任务偏序、关系和叙事冲突。

### 8.2 多设备场景

模拟 A/B 从 revision R 离线：

- 不同解锁并集。
- 同一唯一奖励重复领取只生效一次。
- 一端消费、一端获得库存按账本正确。
- 两端推进可兼容任务得到共同后继。
- 互斥叙事选择返回冲突，双方快照保留。
- 冲突选择/复制 Profile 后产生新 revision。
- 网络超时后请求重放不重复应用。

## 9. 回放一致性

同一初始快照、内容包、随机种子和命令流在 Native、Web 单线程、Web Pthreads 运行。比较离散权威状态、事件/Operation 序列和容差规范，不比较 Sprite 像素。

出现差异时报告首个 divergent Tick、组件、浮点量化、容器顺序和平台信息。不得用扩大容差掩盖任务/奖励差异。

## 10. 内容与编辑器

- Native CLI 与 ContentCore WASM 对黄金输入输出同一规范模型/错误码。
- 模板创建、迁移、引用图、图可达、预算和本地化。
- 编辑器刷新/崩溃恢复、Undo、外部修改、三方合并和大项目性能。
- 预览与正式 Runtime 读取同一 `.tgdpack`。
- 当前 `npm run validate:design` 持续守住文档、内容规模、模板和技术基线。

## 11. 性能与网络

固定硬件、浏览器、DPR 和网络整形。报告 P50/P95/P99、长帧、WASM/JS/GPU 内存、GC、下载、解压、纹理上传、IndexedDB、DOM layout 和同步延迟。

每地区路线：冷缓存、热缓存、弱网、离线、最密战斗、最快穿越、30–60 分钟内存回环。趋势回退超过阈值即失败，即使未越绝对硬门。

## 12. CI 流水线

### F1-DEV-01～02 已实现门

`.github/workflows/f1-clean-build.yml` 已在 Windows 2022 托管 runner 上实现干净构建：固定 action commit 与 Node 20.18.0，恢复后重新校验全部工具缓存和 LLVM 官方签名，执行 MSVC/Clang Debug/Release（每配置含 Native 命令回放）、Axmol Web Single Debug/Release，再以 Chrome/Edge/Firefox 验证八个生命周期/WebGL2 context restore 检查点和 30/60/144 Hz、10,000 Tick WASM 黄金回放。无 GPU CI 显式使用软件 WebGL，并把 graphics mode/renderer、量化姿态、校验值与宿主/探针 BuildIdentity 写入报告；该结果只用于功能可复现性，不替代真实硬件性能门。

## 13. Flaky 政策

一次受控重跑只收集诊断，原失败仍记录。Flaky 标记负责人、首次日期、原因、到期日；存档、同步、发布和核心浏览器测试不得长期隔离。

测试禁止依赖真实墙钟、公共互联网、执行顺序、作者目录或未固定第三方账号。OIDC/CDN 使用契约模拟；少量 staging 真实集成在候选门执行。

## 14. 缺陷严重度

| 级别 | 示例 | 发布行为 |
| --- | --- | --- |
| Blocker | 无法启动/通关、存档丢失、同步覆盖、安全严重问题 | 禁止合并/发布 |
| Critical | 稳定崩溃、Boss/任务阻断、主要浏览器不可用 | 候选版前为 0 |
| Major | 明显玩法、性能、缓存、辅助或冲突 UX 错误 | 逐项批准，不能批量忽略 |
| Minor | 不阻断局部瑕疵 | 记录负责人和版本 |

## 15. 人工游玩

每地区做灰盒理解、内容节奏、近最终盲测。额外覆盖：首次网页用户是否理解加载/保存/云状态；刷新/离线是否安心；冲突 UI 是否能在不懂“revision”的情况下做决定；键鼠/手柄切换是否顺畅。

测试主持人不现场解释。每地区 5 小时按首玩中位数；下载等待不计入。

## 16. 发布清单

- Blocker/Critical 为 0。
- Web 单线程完整；线程版仅在支持环境启用。
- 内容/可执行/Web Shell/存档/同步兼容矩阵确认。

## 17. 验证工作的 Definition of Ready

测试项进入实现前必须具备：目标合同/行为、可创建初始状态、稳定断言、预期错误、环境/版本、Owner、运行频率和失败诊断。性能测试还要固定硬件、浏览器、DPR、网络整形、预热/采样区间和基线 build；人工盲测要有招募条件、任务脚本、主持人禁说事项和计时口径。

没有可测试合同的任务退回设计；没有自动夹具但理由成立的人工项必须写清何时补自动化。测试不得从实现细节猜预期后再自证正确。

## 20. 机器 Schema 验证成熟度

当前 `npm run validate:design` 会解析 Schema 文件，并以项目语义断言检查数量、ID、引用、文档层级和技术基线；它尚未执行完整 JSON Schema Draft 2020-12 关键字集合。这一限制不得在交接时表述成“所有 Schema 已被标准验证器证明”。

G1/G3 工具链必须引入锁定的 Draft 2020-12 validator，提供：

- 全部现有 Schema 的正向/负向 fixture；错误字段、缺字段、重复平台/variant、非法 ID、错误版本必须失败。
- 内容工作台、Native CLI 与 CI 的同一诊断 code/JSON Pointer。
- pack/save/operation/sync IDL 的机器合同和生成类型。
- Schema 自身 meta-schema 检查、版本迁移和兼容 fixture。

在引入完整 validator 前，手写语义守卫仍保留；二者解决的问题不同，不能用“Schema 通过”替代跨引用、图可达、预算和领域不变量。

## 21. 门禁例外

Blocker/Critical、安全、丢档、重复权益、不可回滚发布和必选浏览器启动失败没有常规例外。Major/性能趋势的临时例外必须包含：具体测试、影响用户、范围、Owner、批准者、到期日、修复里程碑、监控和回滚；到期自动恢复阻断。

测试重跑不能把红变绿后删除第一次失败。调整阈值、黄金档或快照必须在差异评审中说明玩家语义未变；禁止用批量更新黄金文件掩盖回归。
