# 00｜《天工渡》项目总纲

> 状态：Active
> 适用范围：产品、世界、技术、生产、版本与协作的最高层约束
> 最后更新：2026-07-11
> 维护者角色：制作人/游戏总监；改变本文件需产品、技术、内容三方评审

## 1. 项目定义

《天工渡》是一款浏览器首发、可向原生多端扩展的横版动作角色扮演游戏。它借鉴《造梦西游》一类“打开网页即可进入、横版战斗清晰、长期养成持续更新”的产品形态，但不是用少量事件包装的轻量页游或短篇技术 Demo。

玩家扮演年轻的“执灯渡人”，携一页残缺《天工录》巡行“九州百工”。每个地区都是有人生活、生产、交易、争论和传承的社会，不是只为刷怪而存在的关卡。玩家通过调查、交往、工艺、探索与战斗看见失序的根因，最后让器物、工序、责任与关系重新“归位”。

一句话体验承诺：

> 在温润繁盛的东方工艺世界里，以兵击破势、以工灯见因、以选择让百工重新归位。

## 2. 不可动摇的五根支柱

### 2.1 工艺必须成为玩法

工艺不能只存在于图鉴、背景文字或数值加成里。选材、工序、失误、用途与师承必须改变探索路径、战斗规则、角色关系或结局裁定。

### 2.2 战斗终点是归位

敌人可以被打倒，但完整胜利必须经历“破势—验因—归位”。Boss 的最后阶段要回答：何物失序、谁承担后果、怎样恢复用途、记录如何留下。

### 2.3 社会先于灾难

世界基调是平和、繁盛、有烟火气的日常。危机从真实运转的生产和关系中长出，不能用末世、全员阴谋或无差别黑暗替代复杂性。

### 2.4 系统先于一次性脚本

任何需要出现三次以上的内容，都必须先有稳定契约、模板、验证器和预览流程。新角色、新 Boss、新任务和新地区不允许每次从空白代码开始。

### 2.5 可验证的深度先于“做完”

功能完成不等于文件存在。它必须能被玩家触发、与其他系统联动、通过测试、有性能证据、有文档、有足够内容承载，并能在发布构建中复现。

## 3. 1.0 产品边界

1.0 是一个完整的单机章节合集，而非抢先体验占位符：

- 3 个地区：江南水路、龙泉山炉、徽州墨岭。
- 18 个大型子地区，每个地区 6 个；每个地区主线不少于 5 小时。
- 21 章主线、18 条成链支线、120 个以上兴趣点。
- 14 个具备多阶段机制和归位结局的 Boss。
- 24 个核心 NPC；每人至少 3 阶关系弧、2 类系统交互，并在 18 条成链支线中拥有独立或共同主角/关键参与弧。
- 两套互相联动的战斗系统：“护牒兵击”与“工灯定印”。
- 两条完整武器体系：伞器、剑器；各自包含架势、工艺、成长和宗师能力。
- 主线约 15.75 小时；主线加支线约 23.25 小时；全收集与精通目标约 30 小时。

精确目录和机器可验证数量以 [`content/design/v1-content-catalog.json`](../content/design/v1-content-catalog.json) 为准。

### 1.0 明确不做

- 不做无缝超大开放世界；采用高密度地区与可控流式子地区。
- 不做 MMO、强联网经济、PVP 或实时多人协作。
- 不承诺 Web、Windows、Android、iOS 同日首发；1.0 先让桌面浏览器达标，再逐端通过输入、性能和平台审核门。
- 不开放任意 C++/WebAssembly 代码模组；首版仅支持受限的数据与资源扩展。
- 不用 JavaScript 重新实现一套玩法；DOM/Service Worker 只承担网页外壳、可访问性、缓存和平台桥接。
- 不用随机生成代替手工关卡、角色关系和剧情因果。

## 4. 技术总决策

正式客户端采用 C++20 游戏核心，Axmol 2.11.x LTS 作为 2D 与跨平台宿主，Emscripten 将 Web 构建编译为 wasm32。桌面浏览器是 1.0 必选平台，WebGL2 为渲染基线；Windows/Android/iOS 通过平台适配复用同一 Gameplay，不建立平行代码线。

模拟固定 60 Hz，玩法状态与 Axmol 场景树、JavaScript 和 DOM 分离。Canvas 承载游戏画面，DOM 承载 HUD、菜单和辅助功能；IndexedDB 保存本地进度，Service Worker 与 CDN 负责版本化资源缓存。可选云同步使用“修订快照 + 幂等操作日志”，让访客离线游玩和跨设备接续同时成立。

内容以稳定 ID、JSON Schema Draft 2020-12、带哈希 `.tgdpack` 清单和显式迁移为契约源。Python/Node.js 只用于离线生产与校验，不进入关键帧循环。权威机器基线见 [`../content/design/technical-baseline.json`](../content/design/technical-baseline.json)。

详细决策见 [`01-开发者文档.md`](01-开发者文档.md) 和 [`01-developer/19-ADR索引.md`](01-developer/19-ADR索引.md)。

## 5. 产品层级

```text
游戏
├── 地区 Region（5–10 小时）
│   ├── 大型子地区 Subregion（通常 30–70 分钟；终战型可达 90–100 分钟并分段）
│   │   ├── 场景 Cell / Interior / Encounter Space
│   │   └── 兴趣点 POI / 事件 / NPC 日程
│   ├── 主线章 Chapter
│   ├── 支线链 Quest Chain
│   └── 地区归位状态 Region Resolution
├── 系统 System（战斗、关系、工艺、探索、经济）
└── 内容包 Package（版本、依赖、签名、迁移）
```

地区不是背景皮肤。每个地区必须拥有独立的工艺问题、探索语法、社会矛盾、Boss 机制、NPC 网络、武器或系统扩展，以及可感知的归位后变化。

| 05 | 世界观、叙事、组织、术语与正史边界 |

## 7. 决策与变更

- 产品支柱、1.0 边界或主技术路线的变化必须写 ADR，并说明替代方案、迁移、风险和回滚。
- 已发生事实写入 03；未来计划写入 02；二者禁止混写。
- 世界新设定先与 05 校验；游戏机制先与 04 校验；生产方法先与 06 校验。
- 内容目录的数量、ID 和引用必须通过 `npm run validate:design`。
- 任何人都可以提出变更，但没有证据的“感觉更好”不能越过发布门禁。

## 8. 质量红线

- 同一概念不允许多套正式 ID。
- 运行时代码不直接读取未经校验的任意 JSON。
- 表现层不拥有战斗真相，编辑器不反向依赖具体地区内容。
- 保存档不能依赖场景节点路径、数组下标或可变显示名。
- Boss 不得只有血量增长；NPC 不得只有站桩对话；地区不得只有换色地图。
- 自动化通过不能替代人工游玩；人工“感觉能玩”也不能替代自动化与性能证据。
- 不用短期赶工给长期系统制造永久例外。

## 9. 历史边界

旧版 Web/Canvas 项目已在 GitHub 分支 `codex/archive-legacy-web-v1` 归档。它保留早期世界构思、内容 JSON、江南雨巷可玩闭环与验证结果，可供研究，但不再定义新主线的架构、目录或发布目标。

主分支只向前演化。需要借鉴旧内容时，应把设计意图迁入新契约，而不是复制旧运行时代码。

## 10. 玩家承诺与成功判定

《天工渡》面向喜欢横版动作、东方幻想、角色关系和探索推理的玩家。它允许玩家用 20–45 分钟完成一段有明确收束的游玩，也允许连续数小时深入地区；浏览器“即开即玩”降低进入成本，不代表系统、叙事或关卡必须轻量。

1.0 成功不是只把目录中的数量做满，而是同时满足：

- 新玩家能在序章理解移动、兵击、工灯、证据和归位，并愿意继续进入江南。
- 熟练玩家能从两种武器、架势、部件、工印、路线与关系协作中形成多种有效解法。
- 每个地区至少提供五小时不靠等待、强制慢走、刷同组敌人或无意义往返填充的主线。
- 玩家在归位后能从空间、人物和记录看见自己的决定，而不是只收到不同奖励数字。
- 刷新、离线、设备切换、更新和失败重试不会让玩家怀疑存档是否安全。
- 新地区与新角色主要通过模板、内容包和既有系统扩展，长期更新不持续放大返工成本。

## 11. 长期更新模型

发布后的基本内容单元是“地区篇章”，不是无限追加数值等级。一个新地区应带来新的工艺问题、探索语法、社会网络、Boss 归位和至少一个可复用系统扩展；旧武器、旧关系和《天工录》记录要能在新地区产生组合价值。

- 1.x 优先修复、平衡、辅助、关系场景、挑战回访和章节型扩展，不随意破坏存档语义。
- 新武器只能在共同输入、命令、效果和工灯接口上扩展，不能复制整套战斗核心。
- 数值天花板缓慢增长，横向用途和组合优先；不能让新地区装备使旧地区全部失去意义。
- 正式内容包可以独立更新，但新增玩法语义必须升级可执行版本/Content API。
- 每个大版本声明可读/可写存档范围、迁移、回滚和停止支持窗口。
- 未完成地区不以“长期运营”为理由切碎到无法形成一段完整体验。

商业模式、付费点和外部发行合作尚未决定；在专门产品决策完成前，禁止把付费、抽取、体力、广告或联网权益写入 Gameplay。任何商业设计都不得破坏访客可玩、本地存档、完整叙事购买边界和单机可恢复性，且必须单独经过产品、法律、安全与平台评审。

## 12. 决策权与否决边界

| 决策 | 最终负责 | 必须会签 | 可否由单个开发者临时改变 |
| --- | --- | --- | --- |
| 产品支柱、1.0 范围、里程碑 Go/No-Go | 制作人 | 游戏总监、技术、内容、QA | 否 |
| 世界正史、角色核心与结局语义 | 叙事负责人 | 游戏总监、内容、相关设计 | 否 |
| 分层、平台、存档、同步和长期依赖 | 技术负责人 | 模块 Owner、安全、运维 | 否，需 ADR |
| 战斗手感、数值和关卡节奏 | 战斗/关卡负责人 | 游戏总监、动画、QA | 仅可调试实验，合入需评审 |
| 安全、隐私、许可和发布门 | 安全/发布负责人 | 技术、制作人 | 有权阻断发布 |
| 自动化与人工验收是否充分 | QA 负责人 | 功能 Owner、制作人 | 有权要求补证据 |

制作人不能以排期压力单方面覆盖存档、安全或 Blocker；技术负责人也不能以架构纯洁度单方面删减玩家承诺。争议先回到证据与本文件支柱，仍无法决定时保留 No-Go，并把取舍写入 ADR/02。
