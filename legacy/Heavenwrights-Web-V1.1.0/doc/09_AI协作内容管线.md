# 《天工渡 / Heavenwrights》AI 协作内容管线

> 版本：0.3  
> 日期：2026-07-01  
> 状态：当前有效生产方法  
> 目标：让 AI 能持续参与大型 Web 端游戏内容生产，而不是被大型编辑器配置阻断。

## 1. 核心判断

大型引擎编辑器的问题不在能力弱，而在内容生产容易变成：

- 手工点选。
- Inspector / Prefab / Scene 黑盒配置。
- 大量二进制资源难以 diff。
- AI 难以批量审查和改写。
- 自动化校验困难。

《天工渡》采用：

```text
文本化内容数据 + Python 代码生成 / 校验 + 轻量可视化工具 + Axmol WebAssembly/WebGL2 + C++ Runtime
```

## 2. 内容资产分层

| 层级 | 形式 | AI 参与方式 |
| --- | --- | --- |
| 设定层 | Markdown | 起草、改写、审查一致性 |
| 结构层 | JSON | 生成地区、任务、敌人、技能框架 |
| 数值层 | CSV | 批量调参、范围检查、对比版本 |
| 关卡层 | Tiled / LDtk JSON 或自研 Room JSON | 生成房间草图、刷怪点、触发器建议 |
| 脚本层 | JSON 条件树 | 生成任务条件、对话分支、事件结果 |
| 表现层 | PNG / Aseprite / 音频 | 生成占位需求单，最终由人工美术把关 |

## 4. Schema 原则

每个数据文件必须：

- 有稳定英文 `id`。
- 有 `schemaVersion`。
- 有 `tags`。
- 有 `designerNote`。
- 引用其他对象时只引用 ID。
- 中文显示文本进入本地化表。
- 可被命令行校验。

江南第一版投放起，内容对象采用稳定 ID 风格：

```text
<objectType>.<scope>.<name>
```

示例：

```text
subregion.jiangnan_rain_alley.water_entry
quest.jiangnan_rain_alley.mainline
building.jiangnan_rain_alley.umbrella_shop
interaction.jiangnan_rain_alley.market_choice
bossphase.rain_alley_broken_umbrella_obsession.p2
bgm.jiangnan_rain_alley.main_loop
cinematic.jiangnan_rain_alley.boss_outro
```

新增地区投放至少要先写 Markdown 设计书，再拆为 `RegionDef / SubregionDef / QuestDef / NpcDef / BuildingDef / InteractionDef / EnemyDef / BossDef / BossPhaseDef / AssetDef / BgmDef / CinematicDef`。中文显示名不参与引用，统一进入 `localization/`。

示例：

```json
{
  "schemaVersion": 1,
  "id": "skill.umbrella.gather_wind",
  "displayNameKey": "skill.umbrella.gather_wind.name",
  "tags": ["pathway.umbrella", "element.wind", "combat.control"],
  "cost": {
    "windBreath": 25
  },
  "effects": [
    {
      "type": "pull",
      "radius": 96,
      "force": 0.8,
      "targetTags": ["enemy.light"]
    },
    {
      "type": "break_damage",
      "amount": 18
    }
  ],
  "designerNote": "伞师基础控场技，用于拉近轻型敌人并吹散雨幕机关。"
}
```

## 5. AI 工作流

新增内容：

```text
策划提出意图
-> AI 生成 Markdown 设计草案
-> AI 生成 JSON / CSV 初稿
-> Python 校验器检查
-> C++ 沙盒运行
-> 人工试玩和审查
-> AI 根据反馈批量修正
```

修改内容：

```text
提出修改目标
-> AI 搜索受影响文件
-> AI 给出影响范围
-> AI 修改文本数据
-> 校验器比较差异
-> 回放 / 沙盒验证
-> 合并
```

## 6. 必要工具

| 工具 | 作用 |
| --- | --- |
| `content validate` | 校验所有内容数据 |
| `content diff` | 对比数值和文本改动 |
| `content pack` | 打包发行资源 |
| `stage preview` | 预览房间、碰撞、刷怪点 |
| `hitbox viewer` | 查看攻击盒、受击盒和动画帧 |
| `skill sandbox` | 快速测试技能和敌人 |
| `dialogue preview` | 预览 NPC 对话分支 |
| `record replay` | 记录输入并复现战斗问题 |

## 7. 防止 AI 写坏内容

必须建立：

- JSON Schema。
- ID 引用检查。
- 数值上下限检查。
- 循环引用检查。
- 未本地化文本检查。
- 关卡可达性检查。
- 技能帧事件完整性检查。
- 自动生成 changelog。

## 8. 为什么能支撑大型项目

项目能不能做大，不取决于是否用了大型编辑器，而取决于：

- 内容是否有稳定 Schema。
- 生产是否可批量化。
- 改动是否可 diff。
- 错误是否可自动发现。
- 运行时是否可热重载。
- 设计意图是否能追溯。
- AI 是否能读懂并参与修改。
