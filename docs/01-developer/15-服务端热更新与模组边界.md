# 15｜服务端、云同步、更新与模组边界

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Accepted Boundary；同步服务在 F1 Spike 验证
> 最后更新：2026-07-10

## 1. 产品边界

1.0 是浏览器首发的单人动作 RPG，不是 MMO。账号和云同步用于跨浏览器/设备接续、备份和未来原生多端，不进入每帧战斗权威。

承诺：访客无需注册即可开始；网络中断时已缓存内容和本地档继续；同步服务故障不回滚本地已保存进度。跨端接续是可选增强，但一旦开启必须可解释、可撤销、可导出。

## 2. 服务组成

```text
Static Web/CDN
├── HTML/JS/Service Worker/WASM
├── release manifests
└── hashed .tgdpack bundles

Identity Provider (OIDC)
└── authorization; no game passwords

Sync API (C++20/Drogon)
├── session exchange
├── profiles/devices
├── snapshots/operations/conflicts
├── export/delete
└── update manifest metadata (optional)

PostgreSQL
├── identity mapping/profile heads
├── revisions/operation dedupe
├── conflict/audit metadata
└── migration/version tables

Object Storage (when needed)
└── encrypted/compressed large snapshots and exports
```

Redis、消息队列和微服务不预装。只有压测、异步导出或运维证据证明需要时加入。

## 3. 身份与会话

- 本地 Guest Profile 由随机 `profileId/deviceId` 标识，不等于安全账号。
- Web Shell 使用外部 OIDC 提供方推荐的 Authorization Code + PKCE/安全 Cookie 方案；不自建密码数据库。
- Sync API 将外部 subject 映射项目 user，签发短期项目会话；刷新/撤销策略经安全评审。
- 多 Profile、设备重命名、设备撤销、账号退出和数据导出/删除是正式功能。
- 链接账号前显示本地/云档摘要；双方有进度时先合并预览，不自动覆盖。
- Token 不写普通游戏存档，不记录日志，不放可被任意脚本读取的长期 `localStorage`。

## 4. 同步 API 草案

```text
POST /v1/session/exchange
GET  /v1/profiles
POST /v1/profiles/{profileId}/claim
GET  /v1/profiles/{profileId}/head
POST /v1/profiles/{profileId}/sync
GET  /v1/profiles/{profileId}/snapshot/{revision}
POST /v1/profiles/{profileId}/conflicts/{conflictId}/resolve
POST /v1/profiles/{profileId}/export
DELETE /v1/devices/{deviceId}
```

`/sync` 请求含 `requestId`, `deviceId`, `baseRevision`, `clientSchema`, `packageSet`, `snapshotHash`, `operations[]`。响应含新 revision、accepted/rejected operation IDs、server operations、conflicts、迁移/更新要求和配额状态。

所有写请求幂等；相同 request ID 返回相同语义结果。API 用稳定错误码，不让客户端解析英文消息。

## 5. PostgreSQL 事务模型

核心表概念：`users`, `profiles`, `devices`, `profile_revisions`, `operations`, `operation_claims`, `conflicts`, `snapshots`, `schema_migrations`, `audit_events`。

一次同步在事务中：锁定/比较 Profile head → 去重 request/operations → 验证版本与前置 → 应用领域合并 → 写新 revision/冲突 → 更新 head → 提交。Snapshot Blob 可事务写临时对象后再以哈希引用，失败不产生悬空 head。

Operation 表按 profile/revision 分区或归档策略需压测后确定。审计不保存不必要的完整叙事文本/个人信息。

## 6. 领域合并

详细规则见 12。服务端不得只做 JSON merge 或 Last-Write-Wins。

- 集合解锁：并集 + 前置验证。
- 库存/货币：唯一 Operation/claim 账本，不能比较两个最终数字。
- 任务：按状态机偏序寻找合法共同后继。
- 关系：按来源 Operation 重放；互斥里程碑转冲突。
- 地区归位、关键叙事：不自动覆盖，返回双方分支摘要。
- 冲突解决可选择 A、B 或复制为两个 Profile；不会删除未选分支直到保留期结束。

客户端时间只用于展示，不能决定胜者。服务端 revision 和 Operation 因果前置决定顺序。

## 7. 本地优先同步流程

1. Gameplay 完成持久事务并生成 Operation。
2. IndexedDB 原子提交 Snapshot/Operation/Head。
3. UI 显示“已保存在此设备”；云状态单独显示。
4. 网络可用且已登录时批量同步。
5. 成功后本地更新 cloud head；失败保留队列并退避。
6. 409/conflict 下载差异，在 DOM 面板解释来源、设备和影响。
7. 解决操作先本地提交，再上传；任何时候可导出完整本地档。

不能在网络请求返回前阻塞 Boss 结算/关卡退出，也不能显示一个模糊云图标让玩家猜是否已保存。

## 8. 内容发布与热更新

Web 代码天然通过站点部署更新，但仍需要版本纪律：

- HTML/Service Worker 短缓存并检查新 release manifest。
- 哈希 JS/WASM/`.tgdpack` 使用不可变长缓存。
- Service Worker 先完整下载/校验新最小集合，等待安全激活。
- 已运行 Session 不热换 GameCore/Gameplay/地区规则；标题页或明确重启切版本。
- 同一存档记录 Executable/WebShell/ContentApi/PackageSet，旧标签页不会写入新格式而不迁移。
- CDN 回滚指向上一签名 manifest；旧资源保留到回滚窗口结束。

内容包无 C++/WASM/JavaScript 可执行代码。新增玩法代码发布新 WASM/Native 可执行版本。

## 9. PWA 与离线

Service Worker 预缓存壳、WASM、Boot Pack；地区包在进入/空闲时缓存。离线界面列出已可用地区和云同步待处理数量。

Service Worker 可能在任意时刻终止，所有缓存操作用完整 Promise/`waitUntil`，不把内存变量当事务。旧 Service Worker、多个标签、刷新中更新和 Cache Storage 配额必须测试。

PWA 安装不是强制；普通标签页拥有同等游戏内容。卸载/清站点数据的风险在设置和同步页清楚说明。

## 10. 多端计划

- Web Desktop：1.0 权威发布目标。
- Windows Native：F1N 对照，M1 后 Beta；复用 `.tgdpack`、存档、同步和 Gameplay。
- Android：触控、内存、后台、商店和账号门通过后 Beta。
- iOS：同上并完成平台/支付/浏览器引擎政策审查。

每端是同一 Profile 的设备，不复制账号。设备能力/设置本地保存；进度通过领域 Operation 同步。

## 11. 模组边界

Web 1.0 默认不提供任意本地模组执行。未来允许的扩展：

- 独立命名空间的数据、对话、任务、外观和受预算资源。
- 公开能力/效果/任务节点和 UI 插槽。
- 仅本地 Profile 或明确标记的 Modded Profile。

禁止：

- 加载任意 C++/WASM/JavaScript、原生库或脚本解释器。
- 覆盖官方 ID、WASM、Service Worker、签名和 Profile head。
- 网络访问、任意文件/IndexedDB origin 访问、收集个人信息。
- 将 Modded Operation 上传普通官方 Profile。
- 绕过资源/图深度/解压预算。

原生端也不能因为能访问文件系统就放宽代码执行。未来代码模组需独立沙箱 ADR。

## 12. 安全与滥用

- 所有请求鉴权、授权、大小/速率/复杂度限制。
- Operation payload 按版本/领域白名单解析；不反序列化任意 C++ 类型。
- 服务端验证任务/奖励/库存基本不变量；异常 Profile 隔离，不直接封禁或删除。
- 快照压缩防炸弹，Blob 哈希、配额和保留策略。
- OIDC token audience/issuer/nonce/PKCE 校验由成熟库/提供方流程完成。
- CORS 只允许正式 origin；CSRF/Cookie 策略、CSP、COOP/COEP 共同评审。
- 用户可导出/删除数据；备份和审计的延迟删除政策透明。

## 13. 隐私

云同步最少存：外部身份映射、Profile/设备、游戏进度、版本和必要审计。不同意云同步的 Guest 不上传 Profile。遥测与云同步分别征得同意，不能捆绑。

禁止默认上传输入文本、完整日志、模组文件、个人路径和无关浏览器指纹。设备 ID 是项目随机 ID，不使用硬件序列号。

## 14. 运维与灾备

- PostgreSQL 自动备份、时间点恢复、定期恢复演练。
- Snapshot 对象存储版本/生命周期与数据库 head 一致性检查。
- 多版本服务滚动部署先向后兼容，数据库迁移 expand→migrate→contract。
- 监控同步成功/延迟/冲突/重复/迁移/配额，不记录敏感 payload。
- 服务故障时客户端进入离线队列；恢复后限速排空，避免惊群。
- 数据损坏先冻结相关 revision、保留证据和用户导出，再修复。

## 15. F1 Sync 验收

- Guest 完整游玩并刷新恢复，无账号强制。
- Guest 链接 OIDC，正确处理本地有档/云端有档/双方有档。
- 两设备从同一 revision 离线推进，再上线：可合并域自动合并，叙事冲突不丢分支。
- 同一请求/Operation 重放 100 次只生效一次。
- 服务 5xx/超时/429/断网不破坏本地档。
- 旧 Schema 客户端、新服务和数据库滚动迁移可共存。
- Profile 导出/删除、设备撤销和 token 过期可测试。
- 备份恢复到隔离环境后 revision/hash 一致。
