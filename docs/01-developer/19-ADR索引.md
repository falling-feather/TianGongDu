## 1. ADR 是什么

Architecture Decision Record 记录一个长期技术/结构决策当时的背景、选择、替代方案、后果、验证和回滚。ADR 解释“为什么”，10–18 说明“现在怎么做”。

ADR 接受后不可直接改写历史结论；新证据通过新 ADR Supersede。允许修正错字、链接和不改变语义的澄清，并保留 Git 历史。

## 4. 何时必须写 ADR

- 更换引擎、语言、平台、渲染路径或核心第三方框架。
- 改变程序集依赖、权威状态、实体模型、网络/离线边界。
- 改变内容源、Schema、包、存档兼容或模组安全模型。
- 引入不可逆的资产/构建/发布基础设施。
- 建立团队必须长期遵守且迁移成本高的例外。

以下通常不需要：局部重构、可逆参数、单一内容设计、已由现有 ADR 明确覆盖的实现。若局部决定开始被多个团队依赖，应升级为 ADR。

## 5. ADR 模板

```text
# ADR-NNNN｜标题

状态：Proposed / Accepted / Rejected / Superseded
日期：YYYY-MM-DD
决策者：角色/团队
关联：文档、Issue、实验、旧 ADR

## 背景与问题
## 决策驱动因素
## 备选方案
## 决策
## 正面后果
## 负面后果与风险
## 验证与放行门
## 迁移与回滚
## 后续行动
```

提案必须写至少两个真实替代方案和“不改变”的后果。决策不能只列优点。

## 6. 生命周期

1. 作者创建 Proposed ADR，绑定需要决策的里程碑。
2. 相关产品/技术/内容/安全负责人评审，先收集实验数据。
3. 决策者接受或拒绝，记录日期和未解决风险。
4. 接受后同步 00–18、代码守卫、模板和迁移。
5. 在验证门收集结果；不满足则回滚或新建 Superseding ADR。
6. 发布后的事实进入 03，ADR 继续保留原因。

## 7. 编号与文件

主线 ADR 位于 `docs/01-developer/adr/`，按四位数字递增，文件名为 `ADR-NNNN-简短标题.md`。编号一旦占用不复用；Rejected 也保留。索引更新与 ADR 同一提交。
