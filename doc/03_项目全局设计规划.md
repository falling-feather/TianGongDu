## 0. 规划原则

本文件是《天工渡》的全局设计准绳。后续开发应优先遵循本文件；已有专项文档作为细节支撑，除非本文件明确标注“待验证”或“初始参考”，否则不应随意改写项目方向。

本规划不是一次性 GDD 终稿。它负责锁定长期方向、内容结构、系统边界和初始数值区间；后续可以通过实机验证、资料核验和玩家反馈微调数值、扩充地区内容、细化剧情和资源清单，但不应轻易改变以下底层判断：

- 游戏是 Web 端优先的 2D / 类 2.5D 像素风动作 RPG。
- 核心体验是“工艺成为动作，师承成为成长，归位成为战斗终点”。
- 工艺不是图鉴背景，也不是单独小游戏，而要进入移动、战斗、探索、装备、NPC 关系和地区后果。
- 内容生产必须文本化、可 diff、可校验、可被 AI 协作审查。
- 规则层、内容层和表现层必须分离；原型可以快，但长期实现不能把核心规则写死在场景回调里。

## 1. 产品定位

### 1.1 一句话定位

《天工渡 / Heavenwrights》是一款以架空九州百工社会为背景的 Web 端优先 2D / 类 2.5D 像素风动作 RPG。玩家作为持渡牒与工灯的年轻执灯渡人，在城镇、市井、工坊、商路、工事与祟局之间行脚，通过拜师、试艺、采集、修复、战斗与归位补录《天工录》，让器物、场所和人各得其位。

### 1.2 目标玩家

| 玩家类型 | 期待 | 设计回应 |
| --- | --- | --- |
| 动作 RPG 玩家 | 清晰操作、可练习战斗、Boss 机制和成长反馈 | 统一输入底盘、门径资源循环、破绽/归位机制、精英与 Boss 验收 |
| 国风/非遗题材玩家 | 可信的工艺、师徒关系、地方生活和文化尊重 | 真实工序转译、资料核验、NPC 师承、四维地区后果 |
| 轻度网页玩家 | 快速进入、键鼠友好、低安装门槛 | Web 端优先、短纵切、清晰 HUD、可重绑定输入 |
| 长期运营玩家 | 多地区、多门径、可回访与世界线索 | 地区包结构、无款工印暗线、门径池和地区回声 |

### 1.3 北极星

> 让“工艺”成为动作，让“师承”成为成长，让“归位”成为战斗终点。

判断内容是否合格时，按以下问题验收：

- 这个非遗或工艺内容是否进入了可操作行为，而不是只成为说明文本？
- 这个成长是否带来新动作、新解法、新关系或新资源循环，而不是只加攻击力？
- 这场战斗的终点是否完成了修复、净化、唤醒、调停或归位，而不是单纯击杀？
- 这个地区选择是否影响了师傅评价、居民反馈、商路、材料、敌人或后续回访？

## 2. 世界观总纲

### 2.1 世界基底

《天工渡》的世界是架空九州承平社会。它不是末世、乱世，也不是“工艺魔法世界”。大多数地区有稳定官造、行会、商帮、工坊、祠庙、节令、市场和师徒关系。正因为社会整体有序，地方工艺才有工评、订单、改良、传承、竞赛和创新空间。

朝廷设有管理官造、贡作、工籍、公共工事、工评档案和异事处置的机构，暂名“天工院”。天工院不是神秘超自然组织，而是官造档案、工程验收、工籍谱牒、祖师祭仪和异事应对交织出的庞大体系。

玩家所在的“天工渡”是天工院与地方行会共同承认的行脚制度。执灯渡人持有渡牒和工灯，可以在官局、行会、工坊、货栈、工地和祠社之间通行，协助地方处理器物、工事、商约、师承与祟局纠缠出的课题。

### 2.2 “渡”的含义

| 含义 | 世界内解释 | 玩法表达 |
| --- | --- | --- |
| 渡河过境 | 行脚九州，跨地区补录工艺 | 地区包、商路、回访 |
| 渡牒通行 | 进入工坊、仓库、工事和祠社的资格 | 门禁、调查、支线权限 |
| 渡人解结 | 在官、商、匠、民、祠之间调停 | 选择、关系、四维后果 |
| 渡器归位 | 让器物回到应有用途 | 战斗终结、修复、净化 |
| 渡艺传承 | 把将断的技艺重新接上人 | 师承、门径、技能树 |

### 2.3 三层空间

每个地区按三层组织，保证社交、经营、探索和战斗不是割裂模块。

| 层级 | 功能 | 典型内容 |
| --- | --- | --- |
| 市井层 | NPC、订单、交易、关系、民生日常 | 工坊、夜市、码头、货栈、祠庙、行会馆 |
| 工事层 | 类 2.5D 动作区域、采集、机关、试艺 | 竹林、旧桥、窑场、矿洞、染坊、船坞 |
| 祟局层 | Boss、强机制、被放大的地方课题 | 断桥雨幕、裂窑火场、错账幻境、风沙商路 |

设计顺序必须是：先有地方生活，再有局部课题；先有真实工序，再有玩法抽象；先有人的选择，再有 Boss 形态。

### 2.4 表达边界

必须避免：

- 直接说“工艺是魔法”或“世界靠工艺维持”。
- 把真实非遗项目做成怪物。
- 把所有地区写成同一种道德题。
- 把玩家写成无所不能的救世主。
- 把每个地区写成灾难现场，缺少日常欢欣与地方活力。
- 把无款工印写成俗套幕后黑手标记。
- 把《天工录》写成单纯收集图鉴。

必须坚持：

- 非遗是活态实践、知识、技能、器物和场景的组合；游戏里要转化为行为、师承、工序、器具、场所和反馈。
- 敌人是失序现象、器影、祟局、机关生命或执念投影，不是文化符号猎奇化。
- 打完以后必须回到现实生活和技艺精进。
- 每个地区都要给《天工录》增加一条可追溯记录。

## 3. 主线结构

### 3.1 长期主线

玩家是天工院巡灯司下的年轻执灯渡人，持临时渡牒与一盏尚未完全校准的工灯。师傅留下了一卷未完成的行脚录，里面有空白页、材料样本、未解释工印、祟局案例和对地方匠人的评价。玩家的第一目标不是拯救世界，而是走完师傅没走完的路，补录《天工录》，校准工灯，并获得正式渡牒。

长期主线由四条线交织：

| 线索 | 内容 | 玩法作用 |
| --- | --- | --- |
| 《天工录》续修 | 天工院重修百工档案，记录地方新工法、师承、器具与异象案例 | 地区包入口、成就、长期目标 |
| 师傅旧页 | 师傅遗留的空白页、材料、工印和评价 | 个人情感线、回访线索 |
| 九州工评 | 各地工坊、行会、官局定期举行工评、试艺、开窑、灯市、赛伞等盛事 | 区域事件和版本活动 |
| 无款工印 | 多地反复出现的共同痕迹，不立即定义为阴谋 | 暗线、跨地区谜题、Boss 终局线索 |

### 3.2 地区循环

每个地区包遵循同一骨架：

```text
获得《天工录》线索或工灯反应
-> 抵达地区，路途小战斗或护送
-> 拜访官局 / 行会 / 工坊 / 祠社
-> 理解当地订单、材料、师承或工事课题
-> 学习当地门径或工艺动作
-> 在工事层多次验证新能力
-> 地区选择影响四维
-> 进入祟局层 Boss
-> 归位、落款、入录
-> 获得下一处线索或回访旧地区
```

### 3.3 地区完成记录

每个地区完成后，《天工录》至少记录：

- 地区名与地貌。
- 代表工艺和关键工序。
- 关键 NPC 与师承关系。
- 玩家选择及四维变化。
- Boss 归位方式。
- 获得门径、器具或探索能力。
- 后续回访线索。
- 资料核验状态。

## 4. 地区包规划

地区包是长期版本主线，不以堆职业或堆副本为主。每个地区包至少包含：

- 一个城镇或据点。
- 一组工事层动作区域。
- 一条地方主线冲突。
- 一到两种代表工艺。
- 一组关键 NPC 与师承关系。
- 一条材料或工艺链。
- 普通敌人、精英敌人、Boss。
- 至少一个可回访支路。
- 一组四维选择与地区回声。

### 4.2 首个地区：江南雨巷

江南雨巷是首个纵切，也是全局设计的可验证样板。

| 模块 | 设计 |
| --- | --- |
| 主题工艺 | 纸伞：选竹、削骨、糊纸、上油、晾晒、绘制、试伞 |
| 地区课题 | 雨季订单暴增，年轻作坊想提高产量，老伞师坚持完整试伞规矩 |
| 核心玩法 | 水路护送、竹林采集、雨幕战斗、纸伞铺学艺、夜市选择、旧桥 Boss |
| 门径能力 | 架伞、完美格挡、弹反、收风、伞旋、借风越行 |
| 四维选择 | 接急单并省工、坚持完整工序、寻找替代材料折中 |
| 归位表达 | 旧桥雨幕平复，夜市灯牌恢复，师傅评价入录，工灯校准一枚工印 |

第一版投放在 `07_江南雨巷纵切规格.md` 中扩展为 11 个节点的地区包样板：水路入镇、镇口雨棚、水巷主路、柳伞匠作坊、竹林小径、染纸晒场、风铃巷、夜市回廊、旧桥下层、屋檐上层和断桥雨幕。M1 最小实现至少完成 7 个核心子地区，并保留其余节点的数据 ID 和回访空间。

江南雨巷必须证明：

- 工艺能进入动作，而不是只出现在说明文本。
- 师傅 NPC 能承担教学、情感和评价。
- 四维选择能影响后续反馈。
- Boss 能整合格挡、弹反、收风和借风越行。

### 4.3 后续地区优先级

短中期优先顺序建议：

1. 江南雨巷：纸伞门径与首个纵切。
2. 龙泉山坊：验证剑客/炉工，补足更主动的近战破招体验。
3. 徽州文房：验证墨工/书符，扩展远程标记和机关封印。
4. 巴蜀竹乡：验证木作/牵丝，强化地形和回访能力。
5. 京畿官造：验证天工院主线、账册、工牒、标准化冲突。

其余地区作为长期内容池，进入正式生产前必须补资料核验和地区包小纲。

## 5. 玩家核心动词与输入

### 5.1 核心动词

玩家从第一小时就应持续做这些事：

- 行脚：移动、穿越、护送、回访。
- 观察：看敌人前摇、机关提示、工序缺陷、NPC 反馈。
- 格挡与闪避：用统一动作底盘处理风险。
- 试艺：通过工艺挑战或战斗教学学习门径。
- 采集与修复：选择材料、处理缺陷、形成器具特质。
- 归位：让器物、场所和关系恢复应有位置。
- 落款：把地区选择与后果写入《天工录》。

### 5.2 推荐输入

| 操作 | 键鼠 | 手柄 | 设计说明 |
| --- | --- | --- | --- |
| 移动 | `WASD` / 方向键 | 左摇杆 / 十字键 | 斜俯视动作平面 |
| 闪避 / 取消 | `Shift` / `Space` | B / Circle | 可取消轻击后摇，不取消所有重击前摇 |
| 轻击 | 鼠标左键 | X / Square | 基础输出与连段入口 |
| 重击 | 鼠标右键 | Y / Triangle | 高消耗、破势、压制 |
| 格挡 / 架伞 | `Q` | LB / L1 | 防守反击和门径变体核心 |
| 技艺键 | `E` | RB / R1 | 门径通用技艺位，伞师为收风 |
| 门径技能 | `R` | RT / R2 | 消耗门径资源，伞师为借风越行 |
| 工灯 / 照形 | `Tab` / 鼠标中键 | LT / L2 | 弱点、机关、工印和资料提示 |
| 交互 / 归位 | `F` | A / Cross | 对话、采集、归位、调查 |
| 快捷 / 门径切换 | `1-4` | 方向键 | 道具或辅修门径 |
| 菜单 | `Esc` | Menu | 暂停、设置、天工录 |

所有键位必须可重绑定。字幕、字号、色盲提示、镜头晃动强度、长按/连点替代、弹反辅助都应进入辅助设置。

### 5.3 输入缓冲

| 行为 | 初始参考 |
| --- | ---: |
| 轻击连段缓冲 | 180ms |
| 重击派生缓冲 | 150ms |
| 闪避缓冲 | 120ms |
| 格挡缓冲 | 100ms |
| 弹反 | 不缓冲，要求时机 |
| 非战斗交互 | 200ms |

## 6. 战斗系统

### 6.1 战斗循环

```text
观察敌人动作
-> 闪避 / 格挡 / 弹反
-> 制造破绽
-> 使用门径技能或工艺器具
-> 积累资源
-> 爆发 / 控场 / 换位
-> 归位或进入下一轮
```

战斗必须强调动作判断，不依赖大随机暴击。玩家主要靠观察、站位、闪避、格挡、弹反、弱点命中和门径资源循环取胜。

### 6.2 格挡、弹反、破绽、归位

| 机制 | 规则 |
| --- | --- |
| 格挡 | 抵消部分伤害，消耗气力，部分门径可从格挡派生反击 |
| 完美格挡 | 在受击前短窗口架势成立，减少消耗并获得门径资源 |
| 弹反 | 更短窗口，造成大量破绽力，打开短反击窗口 |
| 破绽 | 短时间破绽积累超过阈值后敌人进入硬直或机制窗口 |
| 失序 | 相当于敌人的生命/归位进度，归零后进入归位流程 |
| 归位 | 非处决终结，表现为修复、净化、安放、唤醒、调停或落款 |

### 6.3 结算公式

初始参考：

```text
finalDamage =
  actionPower
  * pathwayModifier
  * artifactModifier
  * stateModifier
  * enemyVulnerability
  * difficultyModifier
```

```text
breakDamage =
  actionBreakPower
  * timingBonus
  * postureBonus
  * weakTagBonus
```

格挡：

```text
blockedDamage = incomingDamage * (1 - guardReduction)
staminaCost = max(0, incomingGuardPressure - guardPower)
```

这些公式是初始框架，不是最终平衡常量。正式数值必须通过纵切测试替换。

## 7. 门径体系

### 7.1 门径定义

“门径”不是开局固定职业，而是玩家通过地区师承、工艺试炼和器具精进获得的战斗/探索方法。玩家可以主修一种门径，辅修其他门径。所有门径共享基础输入，但改变动作效果、资源循环、位移方式、破绽处理和探索能力。

每个门径至少定义：

- `id`
- 中文名与显示键。
- 主题标签。
- 战斗定位。
- 探索定位。
- 门径资源。
- 核心动作。
- 技能节点。
- 师傅 NPC。
- 解锁条件。
- 协同标签。
- 难度评级。
- 辅助设置说明。

### 7.2 门径池

| 门径 | 玩法定位 | 资源 | 代表地区 | 上线阶段 |
| --- | --- | --- | --- | --- |

| 乐师 | 节奏增益、削弱、群控 | 音律 | 西域巴扎 / 京畿 | P2 |
| 修真 | 调息蓄势、术法底座、风险爆发 | 内息 | 多地区辅修 | P2 |
| 墨工 / 书符 | 远程标记、封印机关、弱点改写 | 气墨 | 徽州文房 | P1 |

### 7.3 伞师样板

| 技能 | 消耗 | 作用 |
| --- | ---: | --- |
| 架伞 | 气力持续消耗 | 正面防守，抵挡雨针、伞刺和部分远程 |
| 完美格挡 | 窗口 180-220ms | 减少气力消耗，获得风息 |

## 8. 基础数值参考

### 8.1 数值原则

- 动作优先，数值服务动作。
- 不做大随机波动，弱点和破绽优先于暴击。
- 成长以横向能力为主，小幅数值成长为辅。
- 工艺特质改变玩法倾向，不只增加面板。

推荐成长比例：

| 类型 | 占比 | 内容 |
| --- | ---: | --- |
| 横向成长 | 70% | 新招式、新探索能力、新工艺特质、新 NPC 关系 |
| 小幅数值成长 | 20% | 气力、定力、恢复效率、资源上限 |
| 装备强化 | 10% | 器具特质、缺陷修复、材料差异 |

### 8.2 玩家属性

| 字段 | 中文名 | 范围 | 作用 |
| --- | --- | ---: | --- |
| `vitality` | 生机 | 100 起 | 容错，归零后失败、撤退或救回 |
| `stamina` | 气力 | 100 起 | 闪避、重击、格挡维持、部分技艺消耗 |
| `poise` | 定力 | 60 起 | 抗打断能力 |
| `focus` | 专注 | 0-100 | 通用技艺资源 |
| `pathResource` | 门径资源 | 0-100 | 每个门径独立，伞师为风息 |
| `craftAffinity` | 匠心 | 0-5 阶 | 工艺试炼、器具返修和师傅评价 |

### 8.3 敌人属性

| 字段 | 中文表达 | 作用 |
| --- | --- | --- |
| `disorder` | 失序值 | 相当于生命，归零后进入归位流程 |
| `stability` | 稳定 / 定力 | 被打空后暴露破绽 |
| `breakThreshold` | 破绽阈值 | 短时间破绽力超过阈值时硬直 |
| `guard` | 护势 | 防御或结构保护 |
| `aggression` | 攻势倾向 | AI 行为频率 |
| `recovery` | 复稳速度 | 从破绽恢复的速度 |
| `weakTags` | 弱点标签 | 如 `wind`、`paper`、`fire`、`sound` |
| `resistTags` | 抗性标签 | 如 `rain`、`shadow`、`metal` |

### 8.4 江南纵切敌人预算

| 类型 | 失序值 | 稳定 | 目标时长 |
| --- | ---: | ---: | --- |
| 雨魇 | 45-60 | 30 | 8-15 秒 |
| 破伞影 | 70-90 | 45 | 15-25 秒 |
| 断骨伞灵 | 220-300 | 100 | 60-90 秒 |
| 雨巷破伞执念 Boss | 900-1200 | 分阶段 | 4-6 分钟 |

难度曲线初始参考：

- 教学敌人只教一个动作，不叠机制。
- 普通敌人每档 `disorder` 提升 10%-15%，攻势节奏提升 8%-12%。
- 精英敌人必须验证一个门径循环。
- Boss 必须整合本地区核心能力，不做纯血量木桩。

## 9. 工艺、器具与四维系统

### 9.1 工艺转译原则

每种工艺拆成：

```text
真实工序
-> 玩法抽象
-> 战斗 / 探索映射
-> 器具特质
-> NPC 评价
-> 地区四维后果
```

工艺失败不应简单重来，而是形成缺陷、返修空间、师傅评价和地区反馈。

### 9.2 器具字段

每件器具至少描述：

- `materialIds`
- `craftProcessIds`
- `makerNpcId`
- `qualityTier`
- `traits`
- `flaws`
- `visualVariant`
- `combatOverrides`
- `npcCommentIds`

### 9.3 特质样例

| 特质 | 效果 |
| --- | --- |
| 竹骨匀称 | 格挡气力消耗 -8% |
| 伞面吃风 | 完美格挡风息 +2 |
| 纸性偏脆 | 连续格挡第三次后守势下降 |
| 蓝染稳色 | 湿滞抗性 +15% |
| 炉火匀净 | 重击破绽力 +8%，但蓄力时间 +5% |
| 墨线入骨 | 标记弱点持续时间 +1.5 秒 |
| 牵丝有韧 | 束缚类技能持续 +10%，但气力恢复 -4% |

### 9.4 地区四维

四维底层为 0-100，对玩家可用等级、账册、居民反馈和视觉状态表达。

| 策划维度 | 世界内表达 | 含义 |
| --- | --- | --- |
| 传承 | 师谱 | 师徒、谱牒、工序、祖法 |
| 市场 | 货账 | 订单、价格、商路、货期 |
| 生活 | 民用 | 日用、修补、节令需求、居民方便 |
| 精神 | 礼信 | 祖师、祠社、禁忌、仪式意义 |

四维变化至少影响以下一项，重要选择应影响三项以上：

- NPC 台词。
- 商店和材料供给。
- 敌人失序强度。
- 可回访事件。
- Boss 终局视觉状态。
- 《天工录》落款。

## 10. 内容数据与 AI 协作

### 10.1 内容对象

长期最低内容类型：

- `RegionDef`
- `StageDef`
- `RoomDef`
- `NpcDef`
- `DialogueDef`
- `PathwayDef`
- `SkillDef`
- `EnemyDef`
- `BossDef`
- `CraftProcessDef`
- `MaterialDef`
- `ArtifactDef`
- `WorldEventDef`
- `LocalizationDef`

江南第一版投放起新增以下内容对象作为生产基线：

- `SubregionDef`
- `QuestStepDef`
- `BuildingDef`
- `InteractionDef`
- `BossPhaseDef`
- `AssetDef`
- `BgmDef`
- `CinematicDef`

所有内容对象必须有：

- `schemaVersion`
- `id`
- `displayNameKey`
- `tags`
- `designerNote`

引用其他对象只使用稳定 ID。中文显示文本进入本地化表，不能写死在逻辑数据里。

### 10.2 AI 协作流程

新增内容：

```text
策划提出意图
-> AI 生成 Markdown 设计草案
-> AI 生成 JSON / CSV 初稿
-> Python 校验器检查
-> 沙盒运行或预览验证
-> 人工试玩和文化审查
-> AI 根据反馈批量修正
-> 进入发布历史或规划待办
```

修改内容：

```text
提出修改目标
-> 搜索受影响文件
-> 列出影响范围
-> 修改文本数据
-> 校验器比较差异
-> 回放 / 沙盒验证
-> 文档同步
```

AI 不得绕过 Schema 直接改运行规则。每次内容批量写入必须留下验证命令和复核记录。

## 11. 工程与运行边界

### 11.1 分层

| 层级 | 职责 | 禁止 |
| --- | --- | --- |
| Runtime | 固定逻辑帧、实体、碰撞、战斗、技能、工艺、NPC、任务、世界状态、存档 | 直接依赖具体 Axmol 场景节点 |
| Content | JSON / CSV / Markdown / 资源 manifest | 用中文名或文件路径做公开引用 |
| Presentation | 动画、镜头、特效、音频、UI 表现 | 拥有游戏规则真值 |
| Tools | 校验、diff、打包、预览、回放 | 输出不可复现的手工状态 |
| Server | 账号、公告、版本、资源清单、云存档、活动、日志 | 承担客户端战斗规则 |

### 11.2 发布边界

Web 端优先，必须优先验证：

- 浏览器加载首包。
- WebAssembly 初始化耗时。
- WebGL2 渲染、像素缩放、shader、粒子。
- 键鼠、手柄和触屏可选输入。
- 浏览器音频解锁与音量控制。
- LocalStorage / IndexedDB / 云存档同步。
- 资源包下载、缓存、版本 manifest。
- CDN 部署与跨域策略。
- 崩溃、日志和性能上报。

若 WebAssembly 路线在加载、音频、资源包或性能上达不到最低可玩标准，应先解决技术链路，不继续扩展内容。

### 11.3 安全边界

- 内容输入必须通过 Schema 与 ID 引用检查。
- 资源包必须有 manifest 和版本校验，生产环境不得信任未签名或未校验资源。
- 存档保存 Simulation 状态，不保存渲染节点；云存档接口必须鉴权。
- 开发服务器仅用于本地调试，生产静态资源必须来自受控资源清单。
- 第三方依赖接入前必须写 ADR，记录来源、版本、许可、构建链路和复审条件。
- Runtime 与 Presentation 接口应白名单化，避免表现层反向污染规则层。

## 12. UI、HUD 与辅助功能

### 12.1 常驻 HUD

首个纵切最低 HUD：

- 生机。
- 气力。
- 门径资源，伞师为风息。
- 当前技艺 / 技能。
- 交互提示。
- Boss 归位进度。
- 工牒四目。
- 行脚记录。

HUD 要尽量清晰克制，不遮挡主 playfield。浏览器版本默认使用 DOM HUD，Canvas / WebGL 负责 playfield。

### 12.2 菜单

长期菜单结构：

- 天工录：地区记录、师承、器具、工印、回访线索。
- 门径：主修、辅修、技能节点、师傅评价。
- 器具：材料、工序、特质、缺陷、返修。
- 地区：四维、NPC、商路、订单、祠社状态。
- 设置：输入、音频、画面、辅助功能。

### 12.3 辅助功能

最低要求：

- 全键位重绑定。
- 字幕。
- 字号可调。
- 色盲友好提示。
- 弹反窗口辅助。
- 受击伤害辅助。
- 镜头晃动强度。
- 长按 / 连点替代。

辅助功能不得羞辱玩家或改变剧情评价。若影响挑战统计，应在成就或挑战榜单中单独标记。

## 14. 长期版本路线

### 14.1 版本阶段

| 阶段 | 目标 | 主要产物 |
| --- | --- | --- |
| M0 | 首屏原型和文档体系 | 浏览器 demo、内容样例、全局规划 |
| M1 | 江南雨巷短闭环 | 15-25 分钟纵切、可归位 Boss、内容 Schema v1 |
| M2 | Runtime 技术冒烟 | Axmol/C++/WASM 加载、输入、渲染、音频、存档 |
| M3 | 第二地区与第二门径 | 龙泉或徽州，验证非伞师玩法 |
| M4 | 多地区回访 | 至少 3 个地区包、门径辅修、四维回声 |
| Beta | 内容生产管线稳定 | 打包、diff、回放、发布清单、云存档边界 |

### 14.2 不做事项

直到 M1 验收前，不做：

- 全国大地图。
- MMO 或实时联机。
- 大型二进制编辑器工作流。
- 多门径全面铺开。
- 未经资料核验的真实地域或真实传承人强绑定。
- 把当前 browser demo 当作最终 Axmol 实现。

## 15. 资料核验与外部依据

### 15.1 已知依据

- 中国非物质文化遗产网提供国家级代表性项目名录、传承人、十大门类和地方非遗资源入口。该站公开数据可作为内容池和地区资料导航。
- UNESCO 对非物质文化遗产的通行理解强调实践、知识、技能及相关器物、物品和文化空间。游戏转译必须尊重“活态实践”而不是静态符号。
- Axmol 官方仓库说明其面向桌面、Xbox UWP、WebAssembly 和移动游戏，符合当前 C++ + WebAssembly/WebGL2 路线的技术预研入口。

### 15.2 必须核验

- 江南纸伞具体工序、材料、结构和常见缺陷。
- 江南水巷、夜市、灯市、桥、埠头、雨季空间的真实资料。
- 龙泉、徽州、赣鄱、巴蜀、岭南等地区工艺的实际工序和地方语境。
- 每个地区是否有真实地域、真实传承人或真实项目名的表达风险。
- 英文名 `Heavenwrights` 的商标、域名、Steam、社媒和发行区域可用性。

### 15.3 资料进入游戏的门槛

资料进入正式内容前必须经过：

1. 来源记录。
2. 二次核验。
3. 玩法转译说明。
4. 文化表达风险检查。
5. 内容 Schema 校验。
6. 人工复核。
