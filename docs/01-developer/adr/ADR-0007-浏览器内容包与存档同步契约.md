# ADR-0007｜浏览器内容包、IndexedDB 存档与同步 Operation

> 状态：Accepted
> 日期：2026-07-10
> 决策者：技术/内容/同步基线
> Supersedes：ADR-0003

## 背景与问题

JSON Schema、稳定 ID 和显式迁移仍正确，但 2.0 的桌面/PCK 存储模型不适合浏览器。Web 需要小首包、CDN/Service Worker 缓存、IndexedDB 事务、页面刷新恢复，以及能跨设备合并而非整档覆盖的持久操作。

## 驱动因素

- 首次打开不下载全部三地区。
- 内容可缓存、校验、回滚和跨 Web/Native 复用。
- 浏览器刷新/崩溃/配额失败不损坏唯一档。
- 本地档先可靠提交，云同步异步进行。
- 重复/乱序网络请求不重复奖励。
- 叙事冲突不被 Last-Write-Wins 静默覆盖。

## 备选方案

### A. Emscripten 单一 preload `.data`

简单，但首包巨大、地区更新和缓存粒度差。拒绝作为完整内容交付，只可用于最小 Boot。

### B. 自由 JSON 直接 Fetch/LocalStorage

开发快，但缺少类型、哈希、事务、容量和迁移能力。拒绝。

### C. IndexedDB 只保存最新全档，云端 Last-Write-Wins

本地可行但双端会丢进度，重复奖励/库存难合并。拒绝。

### D. `.tgdpack` + IndexedDB Snapshot/Operation + 显式版本

满足分包、本地事务和领域同步。选择。

## 决策

Authoring 继续使用 JSON Schema；C++ ContentCore 烘焙带清单、版本、依赖、哈希和签名的 `.tgdpack`。Web 以哈希 URL/CDN/Service Worker 缓存，Native 使用同一包。

本地 Profile 由版本化 Snapshot + 幂等 Operation Journal + Head 组成，Web 事务写 IndexedDB，Native 原子写文件。每个重要领域事务生成唯一 Operation；云端使用 base revision 检测分叉。

Executable、WebShell、ContentApi、SchemaSet、Package、Save、SyncProtocol 分开版本。

## 正面后果

- 小 Boot、按地区/语言/质量分包和稳定 CDN 缓存。
- 内容更新不覆盖当前可运行集合，易回滚。
- 浏览器/原生共享同一内容和存档字节模型。
- 本地保存不依赖网络；同步可重试/去重。
- 领域 Operation 支持合理自动合并和叙事冲突保留。

## 负面后果与风险

- 需要自建容器、索引、缓存管理和版本迁移。
- IndexedDB 异步/配额/多标签复杂，不能假设 POSIX 文件。
- Operation 模型必须精心划分领域，过细膨胀、过粗无法合并。
- Service Worker 旧版本与内容包清理容易制造隐蔽问题。
- 跨版本云端需同时支持新旧 Snapshot/Operation 窗口。

## 验证

- 首访/回访/弱网预算通过，三地区不进单一 preload。
- `.tgdpack` 篡改、缺包、依赖循环、回滚安全失败。
- IndexedDB 刷新/崩溃/配额/多标签/迁移保留旧 Head。
- Native↔Web 导出/导入一致。
- Operation 重放/重复/乱序只应用一次。
- 双端解锁、库存、任务、关系和叙事冲突按规则处理。
- Native/WASM ContentCore 产物/诊断一致。

## 迁移/回滚

2.0 设计目录继续使用，新容器从 Schema 重新烘焙，不迁移 PCK 假设。若自定义容器复杂度过高，可改为标准 ZIP/独立文件运输，但项目 manifest、稳定 ID、哈希、版本、IndexedDB 和 Operation 合同保持。
