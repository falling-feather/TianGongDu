# 17｜平台构建、部署与运维

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Web-first Baseline；Native 逐端放行
> 最后更新：2026-07-11
> 维护者角色：构建/运维负责人；Web、后端、安全、QA 与发布负责人会签

## 1. 发布顺序

1.0 必选平台是桌面现代浏览器，HTTPS 静态站点 + CDN + 可选同步 API。Windows x64 在 F1N 建立对照构建，M1 后可进入 Beta。Android/iOS 在触控、内存、后台、账号和平台审核通过后再加入。

“同一核心可编译”不等于“该平台已支持”。每端必须有性能、输入、存档、生命周期、部署和回滚证据。

## 2. CMake Preset

`F1-DEV-01` 已实现并由干净 CI 验证的 Preset：

Axmol 固定 tag/commit；Emscripten 经 emsdk 精确版本/官方镜像；CMake/Ninja、编译器、Python/Node 与支持工具都有锁与校验和。构建不运行 `latest` 安装。当前权威锁为 `locked-and-validated`，首个干净构建证据是 `g1-gh-29152364317-1`；尚未进入清单的内容工具、后端和移动依赖不得借此状态冒充已锁定。

## 3. Web 产物

```text
dist/web/<release-id>/
├── index.html
├── manifest.webmanifest
├── service-worker.<hash>.js
├── shell.<hash>.js
├── shell.<hash>.css
├── game-single.<hash>.js
├── game-single.<hash>.wasm
├── game-pthreads.<hash>.js/.wasm/.worker.js
├── release-manifest.<hash>.json
├── boot/<hash>.tgdpack
├── packages/<hash>.tgdpack
└── licenses/
```

Source map/原生符号上传受控错误平台或留档，不公开同路径分发。HTML/manifest 短缓存；哈希文件不可变长缓存。

F1 当前候选输出包括宿主 `tiangongdu-f1.html/.js/.wasm`、同目录 `service-worker.js` 与仅用于确定性证据的 `tgd-replay-probe.html/.js/.wasm`，分别位于 Debug/Release Single 的 `dist/web`。回放探针不是玩家入口或发布壳；F1 Worker 也只为隔离的 `prototype_f1/internal` origin 验证“最小启动壳离线可重开”，不缓存 Profile、内容生产包或账号数据。上面的哈希化发布目录、许可证、正式 release manifest 与可回滚生产 Worker 仍由 `F1-OPS-01` 完成。

## 4. Web 安全/线程头

基础响应至少考虑：

- HTTPS、HSTS（确认全域后）。
- `Content-Security-Policy` 限定脚本、连接、Worker、字体和资源来源。
- `X-Content-Type-Options: nosniff`。
- 适当 `Referrer-Policy`、`Permissions-Policy`。
- WASM 正确 MIME `application/wasm`。
- CORS/CORP 与 CDN 资源明确配置。

Pthreads 渠道需要 COOP/COEP/SharedArrayBuffer；因为隔离可能影响 OIDC popup、第三方 iframe 和 CDN，单线程站点/构建保持独立兼容路径。能力探测决定加载哪个产物，不从 URL 参数任意开启不兼容构建。

## 5. Service Worker 发布

1. 新 release 上传全部哈希文件和包。
2. 在 staging origin 完成哈希/浏览器/离线测试。
3. 最后发布签名 release manifest 与短缓存 HTML/Service Worker。
4. 新 Worker 安装时只缓存最小集合并验证；默认等待安全激活。
5. 运行中 Session 显示“新版本已准备”，保存后由用户重启切换。
6. 激活后保持上一 release 的关键文件到回滚窗口结束。

不能在新 Worker `activate` 时无差别删除所有旧 Cache；旧标签可能仍依赖它们。

隔离的 F1 自动化 Worker 当前允许 `skipWaiting + clients.claim`，只缓存 HTML/loader/WASM 三个启动文件并采用网络优先、离线回退，用于证明断网刷新不阻断 IndexedDB Profile。该行为不得直接提升到 preview/stable；正式渠道必须恢复“等待安全激活”、使用构建/manifest 版本化 Cache，并保留上一 release 到回滚窗口结束。

## 6. CDN 与内容包

`.tgdpack` 路径包含内容哈希，跨版本去重。Release manifest 声明包、依赖、大小、哈希、兼容和资源变体。CDN 不负责信任，客户端仍校验项目清单/哈希/签名。

分包上传后先抽样从多个边缘读取并校验。回滚只切 release manifest，不覆盖已有哈希对象。

## 7. 构建流水线

1. 验证受保护 commit/tag 和干净工作树。
2. 恢复锁定工具/依赖，验证哈希。
3. 构建/测试 Native GameCore、ContentCore 和 Sync Contracts。
4. 全量内容/资产确定性烘焙。
5. 始终构建 Web Single 与 Windows GameCore 对照；仅在线程通道被决策启用时构建 Web Pthreads，否则产出对应 Spike/Deferred 证据。
6. 运行浏览器、回放、存档、同步、性能和安全测试。
7. 生成 SBOM、许可证、版本清单、符号和包哈希。
8. 签名 release/content manifests。
9. 部署 staging，运行冷/热缓存、弱网、旧 Worker、回滚和 API/DB 测试。
10. 人工批准后灰度 Stable manifest，保存不可变产物。

任何手工替换文件都会使 release 无效，必须重新构建/签名。

## 8. 版本可追溯性

构建嵌入：语义版本、Git commit、build ID、渠道、Axmol/Emscripten/编译器、WASM variant、Content API、Web Shell、Sync Protocol、PackageSet、签名 key ID。

设置/诊断页可复制这些信息。Service Worker、WASM、内容包和同步服务各有版本，不能只显示一个“游戏版本”。

## 9. 渠道

- `internal`：开发构建、详细诊断、测试身份。
- `nightly`：自动全量，不承诺存档/云兼容。
- `preview`：受邀用户，独立 origin/Profile/数据库或明确隔离。
- `stable`：正式签名、严格迁移和回滚。

不同渠道的 Service Worker scope、IndexedDB name、Cache name、OIDC client、API/DB 必须隔离，防止 Preview 覆盖 Stable 档。

## 10. 灰度与回滚

静态站点可按 release manifest 分桶；Sync API 采用向后兼容滚动部署。灰度监控启动、包哈希、Service Worker、存档迁移、同步错误和冲突率。

回滚顺序：停止扩大 → manifest 指回上一兼容 release → 必要时回滚 API feature flag → 数据库使用向后兼容路径。已经写入新版存档/云 revision 时不得让旧客户端静默删除字段；可要求更新或只读导出。

## 11. Web 本地数据与支持

本地数据属于 origin。域名、scheme、port 或路径策略变化可能使 IndexedDB/Service Worker 看起来“丢失”，因此正式 origin 一旦发布视为持久 API；域名迁移需桥接页面、账号/云同步和导出方案。

支持页提供：存储额度、持久性、当前/旧 Cache、Profile/云 revision、待同步操作、导出、清缓存但保留档（若可）、完全重置。危险操作二次确认并先提供导出。

## 12. Windows Native

产物包含可执行、运行依赖、`.tgdpack` 和许可证；用户数据在平台应用数据目录，格式与 Web 相同。Native 仍使用 release manifest/同步 API，可从同 CDN 下载平台资源变体。

安装、修复、升级、卸载、签名、崩溃符号和商店 SDK在进入 Beta 前锁定。卸载默认保留存档并明确询问删除。

## 13. 移动端

Android/iOS Platform Adapter 处理触控、应用挂起/终止、安全存储、网络切换、商店/账号、下载目录和存储压力。后台不是无限保存时间；每次重大事务提前落盘。

移动包体/按需资源、平台隐私声明、审核、支付（若有）和账号删除需单独门。Web PWA 与原生移动不能共用未经验证的性能假设。

## 14. Sync Service 部署

容器化无状态 Drogon API，PostgreSQL 为权威数据。配置/秘密来自平台密钥系统，不烘焙进镜像。健康分 readiness/liveness；就绪必须包含迁移兼容但不做重查询。

发布遵循：数据库 expand migration → 部署向后兼容服务 → 迁移/回填 → 客户端采用新协议 → 观察窗口 → contract migration。备份和恢复演练是发布门。

## 15. 日志与诊断

Web 客户端本地结构化日志轮换/限额，默认不上传。崩溃/错误报告经同意，含脱敏 stack、build、浏览器、WASM variant、包/存档/同步版本和最近错误摘要。

服务器日志含 correlation/user pseudonymous/profile/revision/request ID，不含 token、完整 Operation payload 和叙事文本。指标聚合，不制造长期设备指纹。

## 16. 灾备

保存：Git tag、工具镜像、Axmol/Emscripten 源/二进制、LFS、Web 产物、内容包、签名元数据、符号、DB migration、备份和黄金存档/同步夹具。

定期演练：从空缓存重建旧正式版；旧 Service Worker/浏览器访问回滚；PostgreSQL 时间点恢复到隔离环境；Profile revision/hash 与对象存储快照一致。

## 17. 环境矩阵与隔离

| 环境 | Origin/Profile/DB | 数据/身份 | 用途 | 发布权限 |
| --- | --- | --- | --- | --- |
| Local | `localhost`/本机独立 DB 名 | mock OIDC、合成 Profile | 开发、契约和故障注入 | 开发者本机，无生产秘密 |
| CI | 每作业临时 origin/DB | 固定 fixture | 构建、浏览器、迁移、同步 | CI 服务账号最小权限 |
| Internal | 独立 internal origin/API/DB | 团队测试身份 | 每日可玩、详细诊断 | 自动部署，不访问 Stable 数据 |
| Preview | 独立 preview origin/API/DB | 受邀账号、明确非正式档 | 外部测试、兼容/性能/隐私 | 受保护环境批准 |
| Staging | 与生产拓扑等价，域名/密钥隔离 | 合成/脱敏数据 | 候选部署、旧 Worker、回滚/灾备 | 发布负责人批准 |
| Stable | 固定正式 origin/API/DB | 正式 Guest/账号/Profile | 玩家发布 | 双人/受保护环境批准、审计 |

任何环境不得通过一个 `ENV=prod` 开关复用同一 IndexedDB、Cache、OIDC client 或数据库。Production 数据不复制到 Local/CI；灾备恢复只进隔离环境并限制访问。

## 18. Single 与 Pthreads 部署拓扑

COOP/COEP 是页面响应/浏览上下文级能力，不能仅在选择 WASM 文件后临时打开。Stable 基线站点保持 Web Single、非跨源隔离，并提供完整 1.0 功能。Pthreads 只有实测收益且 CDN/字体/OIDC/Worker 全部满足时，使用独立受控 channel/origin（如 `threads`）或全站一致的隔离策略；不能在同一页面无刷新切换头部状态。

- 两渠道使用相同 GameCore/Content API/包/存档字节/同步协议；不同 origin 的 IndexedDB、Cache、Guest Profile 互不共享。
- Single 入口只能判断“可能支持”并在用户确认后导航；Threads 目标页在响应头生效后检查 `crossOriginIsolated`、SharedArrayBuffer 与 Worker，失败则解释并回到 Single。
- 已登录玩家可通过云同步在两个 origin 接续同一云 Profile，但两边各有本地镜像/设备 ID；纯本地 Guest 必须先导出/导入或在清楚预览后认领账号，禁止静默搬运。
- Threads 失败回到 Single 时，不改写 Profile 格式；若最新进度只在 Threads 本地，必须先成功同步/导出或明确保留在原 origin，不能承诺自动出现于 Single。
- OIDC redirect、popup/opener、第三方 iframe、CDN CORP/CORS 逐项在 staging 验证。
- 没有至少一个代表性重任务 >15% P95 改善，或隔离显著增加运维/身份风险时，不发布 Threads。

## 19. 服务目标、告警与灾备起始值

生产值在 G3/外部 Preview 压测后锁定；起始目标用于设计容量和演练：

| 能力 | 起始 SLO/目标 | 告警/动作 |
| --- | --- | --- |
| 静态启动/包下载 | CDN 成功率 ≥99.9%（月）；哈希失败接近 0 | 哈希/manifest 错立即停灰度并回滚 |
| Sync API | 成功请求可用性 ≥99.9%；P95 非冲突同步 <2s | 5 分钟错误率/延迟超预算停止发布；客户端本地队列 |
| 本地保存 | 受支持浏览器事务成功率 ≥99.99% | 任一版本丢档/重复奖励为 Blocker |
| 数据恢复 | PostgreSQL RPO ≤15 分钟，RTO ≤4 小时起始目标 | 备份/恢复演练失败阻断生产同步 |
| 静态发布回滚 | manifest 回滚 ≤15 分钟 | 旧资源/Worker 不可用则停止发布 |

指标只记录必要元数据：build/channel/browser family、错误码、延迟、包 ID/hash、revision 结果，不上传完整存档/叙事。每项告警有 Owner、值班/升级路径、运行手册和最近演练日期；团队规模未支持 24×7 值班前，不对玩家承诺超出能力的同步 SLO，离线本地玩法是首要降级路径。

## 20. 基础设施依赖清单

F1/G3 前必须落实或明确 mock：Windows/Clang/MSVC runner、受支持浏览器 runner、Safari/macOS 能力、HTTPS 本地域名/证书、CDN/对象存储、Threads 独立 origin、OIDC 测试 client、PostgreSQL、备份区域、签名密钥服务、Git LFS 配额、构建缓存、符号/证据归档和参考测试设备。

每项登记技术 Owner、成本 Owner、凭据来源、配额、数据区域、替代方案和退出/导出路径。没有生产资源时使用可复现本地模拟，不在文档中写入真实密钥、账号或供应商秘密。
