# 12｜内容、存档与版本契约

> 上级文档：[`../01-开发者文档.md`](../01-开发者文档.md)
> 状态：Accepted Baseline
> 最后更新：2026-07-10

## 1. 数据形态

1. **Authoring Source**：JSON、对话/任务图、场景标记与 DCC 源文件。
2. **Validated Model**：C++ ContentCore 通过 Schema、引用、语义和预算检查后的模型。
3. **Baked Package**：规范化二进制索引和资源组成的 `.tgdpack`，带版本、依赖与哈希。
4. **Runtime State**：从不可变定义实例化的 GameCore Session 状态。
5. **Local Persistence**：IndexedDB/原生文件中的快照、操作日志、备份和设置。
6. **Cloud Profile**：服务端修订快照、已接收 Operation 和设备/迁移元数据。

Release 不直接运行 Authoring Source。内容工作台、CLI、Web 预览和发布构建调用同一 C++ 校验/烘焙库。

## 2. Schema 规则

- JSON Schema Draft 2020-12。
- 根对象声明 `$id`、`schemaVersion`、`additionalProperties` 和最小示例。
- 单位进入字段名/类型，例如 `durationTicks`, `distanceMeters`, `sizeBytes`。
- 玩家文本只存本地化 key。
- 默认值只在 Schema/ContentCore 一处定义。
- Schema 后继续检查跨引用、图可达、预算、包依赖和同步语义。
- Native 与 WASM ContentCore 对同一输入产生相同规范模型和诊断代码。

## 3. `.tgdpack` 布局

`.tgdpack` 是项目定义的只读内容容器，不携带可执行代码：

```text
package.manifest.json
indexes/
  entities.bin
  quests.bin
  assets.bin
data/
  gameplay.bin
assets/
  atlases/
  audio/
  fonts/
  fx/
licenses/
```

Manifest 示例：

```json
{
  "packageId": "region_jiangnan",
  "packageVersion": "1.2.0",
  "schemaSetVersion": "1.1.0",
  "contentApiVersion": "1.0",
  "executableRange": ">=1.0.0 <2.0.0",
  "dependencies": [{ "packageId": "core", "range": "^1.3.0" }],
  "files": [{ "path": "data/gameplay.bin", "sha256": "...", "size": 123 }],
  "signature": { "algorithm": "ed25519", "keyId": "release-a", "value": "..." }
}
```

确切签名算法经安全评审锁定；示例表达合同。Web CDN 使用内容哈希 URL，`Cache-Control: immutable`；新 release manifest 原子切换包集合。

## 4. 版本线

| 版本 | 说明 | 示例 |
| --- | --- | --- |
| ExecutableVersion | C++/WASM/Native 代码 | 战斗实现、Axmol/Emscripten 升级 |
| WebShellVersion | HTML/JS/Service Worker/DOM | 缓存策略、桥协议、UI 修复 |
| ContentApiVersion | Runtime 可理解的内容语义 | 新任务/能力节点 |
| SchemaSetVersion | Authoring 文件结构 | 字段拆分、枚举变化 |
| PackageVersion | 单个 `.tgdpack` | 江南内容修复 |
| SaveVersion | 本地/云快照结构 | 世界状态迁移 |
| SyncProtocolVersion | Operation、修订和冲突协议 | 新合并域或错误码 |

不能用一个版本号假装全部同步。客户端、Web Shell、包、存档和服务声明兼容范围。

## 5. 稳定 ID

全局索引记录 ID、类型、拥有包、引入/弃用版本、替代 ID、来源和本地化 key。引用只用 ID，不用路径、显示名、指针或数组下标。

替换 ID 时旧 ID 进入 alias/tombstone；内容和存档迁移器转换，云端保留历史 Operation 可解析映射。禁止重新使用墓碑 ID。

## 6. 本地 Profile

每个 Profile 包含：

```text
ProfileHead
├── profileId / guestOrAccount / localRevision
├── saveVersion / contentApiVersion / packageSet
├── lastCommittedSnapshot / lastSyncedCloudRevision
├── pendingOperationCount / deviceId
└── checksum / migrationHistory

Snapshot
├── global / region / instance state
├── player progression / inventory
├── quests / relationships / records
├── random streams / checkpoints
└── appliedOperationIds compact index

OperationJournal
└── ordered idempotent operations not yet compacted
```

设置与 Profile 分离；输入、显示、音量和多数辅助选项按设备保存，可选同步小型偏好。

## 7. IndexedDB 事务

Web `IStorage` 使用 IndexedDB：

1. GameCore 在安全点生成不可变 Snapshot/Operation bytes。
2. Web Platform 打开 read-write transaction。
3. 写入新快照、操作、校验和与临时 Head。
4. 读回关键 Header/哈希验证。
5. 同一事务提交新 Head；失败则旧 Head 仍有效。
6. C++ 收到成功回调后才向玩家显示“已保存”。

保留最近自动保存轮换、章节/地区关键点和手动导出。`beforeunload` 不是保存机制；每次重大事务立即提交。

调用 `navigator.storage.estimate()` 显示空间，适当时请求 `persist()`。即便获准持久存储，仍提供云同步/导出，因为用户可主动清理站点数据。

## 8. 原生存储等价

Native `IStorage` 用临时文件 + fsync/原子替换 + 备份轮换，但保持异步 API 和同一字节格式。Web 导出档可导入 Windows/移动端；导入前验证版本、包、Profile 所有权和冲突。

平台安全存储只保存刷新 token/设备密钥等小秘密，不保存大型快照。

## 9. 持久 Operation

Operation 只表示可合并的领域事务，不是完整事件溯源：

```text
operationId       UUID/128-bit unique
profileId
deviceId
baseCloudRevision
domain            quest/inventory/relationship/record/settings/...
kind
payloadVersion
payload
logicalSequence
clientCreatedAt   diagnostic only
```

规则：

- 同一 `operationId` 重试只应用一次。
- Operation 先本地提交，再排队上传。
- 服务端用 base revision 检测分叉，不信客户端墙钟排序。
- 奖励 Operation 包含来源/claim ID，重复完成不会重复发放。
- 日志达到阈值或地区里程碑时压缩为新 Snapshot，并保留审计摘要。

## 10. 领域合并策略

| 领域 | 自动合并 | 冲突处理 |
| --- | --- | --- |
| 已发现图鉴/兴趣点 | 集合并集 | 删除通常不传播 |
| 单向成就/技能解锁 | 合法最大进度/并集 | 服务端验证前置 |
| 消耗品/货币 | Operation 账本 | 不用两个快照数值相加 |
| 唯一物品/奖励 | claim ID 去重 | 非法重复隔离并诊断 |
| 任务非互斥节点 | 状态机偏序合并 | 无合法共同后继则冲突 |
| 叙事选择/地区裁定 | 不自动覆盖 | 展示设备 A/B、时间线与后果，玩家选一条或复制 Profile |
| 关系维度 | 带来源的 Operation 重放 | 互斥里程碑进入叙事冲突 |
| 设置 | 设备本地为主 | 明确允许同步的偏好按修订 |

禁止全档 Last-Write-Wins；它会静默丢掉另一设备几小时进度。

## 11. 云修订协议

每个 Profile 云端有单调 `cloudRevision`。客户端同步请求提交：本地已知 revision、Snapshot hash、待同步 Operations、包/Schema 版本和幂等 request ID。

服务端事务：鉴权 → 校验设备/Profile → 去重请求/Operation → 加载基线 → 验证/应用或合并 → 生成新 revision → 写审计/快照 → 返回 accepted/rejected/conflicts 和新 head。

网络失败时本地不回滚。409 表示分叉，不是普通错误；客户端取 conflict bundle，在 DOM UI 中解释并解决。解决本身产生新的幂等 Operation。

## 12. 游客与账号链接

首次打开生成本地 guest Profile 和 device ID，无需注册。链接 OIDC 账号时：

1. Web Shell 完成授权码 + PKCE 等提供方推荐流程。
2. Sync Service 交换/验证身份，返回项目 Session token。
3. 客户端列出本地游客档和账号已有云档。
4. 无冲突可上传/认领；双方都有进度时进入合并预览，绝不静默覆盖。
5. 完成后记录 Profile owner 和恢复方式；原游客档保留备份一段时间。

客户端不保存账号密码；Web 不把长期 token放 `localStorage`。具体会话/Cookie 策略由安全评审决定。

## 13. 迁移

Save/Operation/Package 迁移是逐版本、纯、可重复转换。Web 在 IndexedDB 隔离对象存储中迁移，验证后切 Head；Native 在临时目录迁移。云服务支持滚动部署期间的新旧协议窗口，数据库迁移先向后兼容再清旧字段。

迁移失败保留原快照/日志和诊断，不上传半迁移数据。旧客户端遇到过新的不可逆数据时明确要求更新或只读导出。

## 14. 错误分级

| 级别 | 示例 | 行为 |
| --- | --- | --- |
| Fatal Boot | Core 包哈希/签名、未知 Content API | 阻止进入并回滚 release manifest |
| Package Fatal | 可选地区包损坏 | 隔离/重下，不载入依赖档 |
| Local Save Error | IndexedDB 配额/事务失败 | 保留内存状态、导出/释放空间，不谎称保存 |
| Sync Conflict | 两端叙事/任务无合法合并 | 保留双方，要求选择/复制 Profile |
| Sync Transient | 离线、429、5xx | 本地继续，退避重试 |
| Content Error | 未知 ID、图不可达 | 构建失败；Release 不应出现 |

## 15. 验证层

1. JSON/Schema 与技术基线。
2. ID 唯一、包依赖、引用、图可达与预算。
3. Native/WASM 规范化产物一致。
4. `.tgdpack` 清单、哈希、签名、缓存和回滚。
5. Snapshot 保存/加载/迁移与 IndexedDB 事务故障。
6. Operation 幂等、重复、乱序、重放、压缩。
7. 双设备离线分叉与每个领域的合并/冲突。
8. Guest→OIDC 认领、账号退出、令牌过期和设备撤销。
9. Web↔Native 导出/导入与同一 Profile 同步。
