# 内容与关卡目标｜首两 Beat 真实内容密度

> 任务：`F1-GAME-01-CONTENT-A`；Owner：关卡/内容负责人；必审：战斗、叙事、C++ Core、内容技术、UX、QA。
> 分支：从本交接包提交建立独立 worktree `codex/f1-content-density`；代码快照基线为 `9991a3d`。
> 写入边界：F1 内容源、对应 Schema/生成器/合同测试和 04–07；不修改 Web 宿主、Profile 或 Presentation 实现。
> 最后更新：2026-07-13。

## 目标

把“JSON 中写了 60 分钟”转成玩家真实可玩的非重复内容。第一阶段先让雨渡抵达与沈砚训练两个 Beat 形成可理解、可失败/恢复、真人首玩约 15–17 分钟的连续段落，并建立可复制的内容密度方法；完整 F1 最终仍需 7 Beat、任一结算不少于 60 分钟。

## 当前事实

- `f1-vertical-slice.json` 的 7 个目标预算为 9/8/12/11/7/10/3 分钟，总计 60 分钟。
- 当前两条完整自动功能路线只有 6035/6224 个合格 Tick，约 100 秒，7 个 Beat 均为 0/7 达标。
- 已有 23 交互、7 安全点、7 战斗触发、8 激活、四阶段 Boss 和双结算只是功能骨架；不能通过延迟交互、等待、重复战斗或调慢速度填时。

## 第一阶段交付

1. 为前两 Beat 写逐分钟玩家动词表：地面纵深移动、观察、空间判断、工艺操作、沈砚对话、两架势练习、失败/恢复与反馈；每 30–90 秒应有新的判断或状态变化。
2. 复核雨渡 6 节点和训练 5 步，增加有意义的空间路线、可选观察、错误反馈和训练组合；可选内容不能阻塞主线，重复进入不得重复发奖励。
3. 只使用已有通用 Definition 能力直接落地；遇到“多个 Objective 会合、复合条件、等待/失败替代、环境机关持久状态”等缺口，先写稳定合同提案、Owner 和负向例，不在内容 JSON 中造特殊字段或硬编码 Runtime。
4. 更新唯一 `content/design/f1-vertical-slice.json`、Schema/生成器和 Native/Node 夹具；不得另建第二份流程真相。
5. 为两个 Beat 提供安全点、错序、重复、失败重试、路线返回、时间审计和回放夹具；Presentation 文案/镜头只通过既有可观察事件提出需求。
6. 同步 04–07 的玩家动词、关卡节奏和内容缺口；只有集成实跑通过后才把事实写入 03。

## 内容密度验收

- 未看开发界面的玩家无需口头提示，从获得控制连续完成前两 Beat，首玩中位数目标 15–17 分钟。
- 合格时长来自探索、选择、战斗、对话和工艺判断；空跑、暂停、挂机、失败尝试和重复刷怪按现有审计规则排除。
- 两种架势都在有风险/反馈的情境中使用，不是按一次键即完成的清单。
- 错序、跳过、失败、重复交互和离开再回到安全点不会死锁或重复推进。
- 固定命令回放、MSVC/Clang 和 Web Single 的离散结果一致；证据记录实际分钟数而非只记录 `targetMinutes`。

## 与其他对话的接口

- 需要新通用合同：提交 proposal/fixture，由主开发评审后集成；本分支不直接改 Runtime。
- 需要 HUD/对话状态：提交 `UiEvent`/面板模型需求给 UI 对话，不在内容文件保存布局。
- 需要资产：用 Stable Asset ID、脚点/锚点/状态变体需求交给美术，不用文件名承担玩法语义。

## 非目标

第一阶段不承诺一次完成全部 60 分钟、不做最终台词/配音/美术、不铺三地区，也不把重复敌群和长通道冒充内容。首两 Beat 通过后，按同一方法继续伞巷/工位、返程、Boss 和结算。
