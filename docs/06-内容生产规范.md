# 06｜内容生产规范

> 状态：Active
> 职责：规定内容分类、模板、生产阶段、评审、AI 协作与规模控制
> 最后更新：2026-07-10
> 维护者角色：内容负责人；工具、玩法、叙事、关卡、技术美术与 QA 会签对应模板

## 1. 目标

本规范解决一个根本问题：新角色、新 Boss、新任务和新地区不能每次重新造轮子。内容团队应在通用系统和受验证模板上组合差异，把时间花在玩法、人物和节奏，而不是重复写加载、存档、UI 与失败逻辑。

技术实现与编辑器界面见 [`01-developer/13-编辑器与模板生产.md`](01-developer/13-编辑器与模板生产.md)，本文件只定义内容团队必须交付什么。

Foundation 节点已提供可机器校验的 [`template-registry.json`](../content/templates/template-registry.json) 与对应 [`template-registry.schema.json`](../content/schemas/template-registry.schema.json)；F1 在此合同上实现实际编辑器插件和命令行生成器。

## 2. 内容分类树

```text
World
├── Region
│   ├── Subregion
│   │   ├── Cell / Interior / EncounterSpace
│   │   ├── POI / GatherNode / PublicWork
│   │   └── ScheduleAnchor / TravelLink
│   ├── Chapter / QuestChain / WorldEvent
│   └── RegionState / Resolution
├── Entity
│   ├── Player / NPC / Enemy / Boss
│   └── Interactable / Device / Vehicle
├── Ability
│   ├── Weapon / Stance / Move / Effect
│   └── LanternAction / CraftAction / Cooperation
├── Narrative
│   ├── Dialogue / Evidence / RecordEntry
│   └── Cutscene / Bark / Letter / Rumor
└── Economy
    ├── Item / Material / Tool / Recipe
    └── Shop / PriceRule / RewardTable
```

目录决定所有权，不决定运行时类继承。内容作者组合组件和引用，Runtime 不为每个分类制造深继承树。

## 3. 稳定 ID

ID 使用小写英文、数字和下划线。地区局部内容使用 `<scope>_<type>_<meaning>`；在全项目索引中天然唯一且会跨地区引用的根实体可使用 `<type>_<meaning>`。示例：

- `npc_shen_yan`
- `lq_boss_sheathless_master`
- `hz_quest_small_print_big_road`
- `core_effect_stagger_medium`

2.2 开发对接版已冻结 `npc_*`、`region_*`、`*_chapter_*`、`*_quest_*`、`*_enemy_*` 和 Boss ID；F1 不再做一次“全面改名”。确需更名时必须创建 alias/tombstone 与迁移夹具，之后 ID 永不因显示名、翻译或文件移动而改变。数字序号只用于确有稳定顺序的章节；任务和角色使用语义 ID。

规则：

- ID 由工具生成并查重，不由作者随意复制。
- 删除内容保留墓碑记录；重新使用旧 ID 禁止。
- 引用只用 ID，不用文件路径、显示名或数组下标。
- 复制模板必须生成新 ID；“另存为”不能保留原身份。
- 合并内容时以 ID 冲突为硬错误，不做静默覆盖。

## 4. 内容生命周期

| 阶段 | 产物 | 允许缺失 | 退出条件 |
| --- | --- | --- | --- |
| Pitch | 一页意图卡 | 数值、资产、完整台词 | 与支柱、地区主题和预算一致 |
| Contract | 模板实例与稳定 ID | 最终资产 | Schema、引用和所有权通过 |
| Greybox | 可玩占位 | 高质量表现 | 正常/失败/重试/存档链路完整 |
| Content | 完整机制和文字 | 最终优化 | 与其他系统联动，内容评审通过 |
| Polish | 最终资产、音频、反馈 | 非关键彩蛋 | 性能、可访问性、本地化通过 |
| Shipped | 签名内容包 | 无 | 安装、升级、回滚和遥测验证 |

任何内容都必须知道自己处于哪个阶段。工具和游戏内调试视图要显示阶段，避免占位内容被误判成完成。

## 5. 通用角色模板

角色模板是 NPC、敌人和 Boss 的共同基础，不是一个巨大万能表单。

### 5.1 身份模块

- 稳定 ID、显示名 key、地区/组织、内容阶段、负责人。
- 角色一句话、日常目标、当前压力、不可替代的玩法作用。
- 表现原型、体型、碰撞档、语音档、图标引用。

### 5.2 世界模块

- 出生/归属地、日程状态机、天气/章节覆盖、旅行规则。
- 交互半径、导航档、可达性替代点、缺席代理。
- 归位前后和关键选择后的状态变体。

### 5.3 关系模块

- 三阶段以上关系弧及进入条件。
- 信任、认同、亏欠、熟悉、公开声誉的初值与边界。
- 礼物、指导、同行、交易、争论等支持的交互类型。
- 关系断裂、暂时拒绝与恢复路径。

### 5.4 叙事模块

- 对话入口、条件、冷却、优先级和替代台词。
- 任务链、证据、传闻和《天工录》条目引用。
- 本地化备注、称谓规则、敏感性检查和演出预算。

### 5.5 能力模块

- 阵营、感知、移动、交互、战斗/工艺能力和协作接口。
- 非战斗 NPC 不需要空的战斗树；用组件清单表达能力。
- 可被任务临时添加的状态必须有卸载和存档规则。

## 6. 核心 NPC 模板

每个核心 NPC 的 Pitch 必须回答：

1. 如果删掉此人，哪个玩法或主题会消失？
2. 此人的日常在玩家不介入时怎样继续？
3. 关系每阶段改变什么行为，而不仅是台词？
4. 此人至少在哪两种系统中与玩家互动？
5. 该角色在哪个问题上不会被玩家轻易说服？
6. 地区归位后其工作、路线或责任怎样变化？

生产预算基线：8 个关系场景、12–20 组条件对话、至少 3 个日程阶段、参与 1 条 3–5 步成链任务（可独立拥有，也可作为共同主角/关键参与者）、1 个协作/专业能力、至少 2 个地区状态响应。1.0 的 18 条正式支线链承载 24 个核心 NPC 的角色弧，不等于每人各占一个互不相干的任务 ID。实际可因角色等级调整，但删减必须保留关系弧。

## 7. 敌人与 Boss 模板

### 7.1 普通敌人

必须定义：战术角色、威胁距离、节奏签名、读招、抗性/弱点、工灯交点、群组标签、失败/退场方式、掉落用途和性能档。

一个新敌人如果只是旧敌人换模型和更高数值，应作为变体而非新类型；变体至少改变一项位置规则、一项节奏规则或一项工灯规则。

### 7.2 精英机制战

精英机制战是有稳定 ID、独立工灯交点和一次小型修复结果的命名遭遇，不是普通敌人简单加血。目标 2–4 分钟、2–3 次机制变化、4–7 个核心动作、1–2 个工痕，可复用既有场地/骨架。它使用 `template_elite_encounter`，必须支持失败/重试、辅助模式和正史结算，但不承担地区 Boss 的多方裁定规模。

### 7.3 章节/地区 Boss

Boss 模板包含：

- 叙事问题和归位问题。
- 阶段图，而非线性血量阈值列表。
- 招式集、组合规则、阶段进入/退出和中断恢复。
- 场景机关、NPC 协作和危险边界。
- 每阶段可发现的工痕、证据与错误假设。
- 正常制伏、完整归位、证据不足和辅助模式结果。
- 死亡/失败、重试、跳过演出、存档和断线/崩溃恢复点。
- 战后世界变化、奖励、记录与再次访问行为。
- 性能压力模型和自动战斗沙盒。

章节 Boss 至少 3 次机制变化，地区 Boss 至少 4 个阶段或等价复杂度。阶段必须改变决策，不以换背景色计数。

## 8. 任务模板

任务由有向状态图表达，节点类型限定为：目标、等待、选择、条件、并行、世界事件、奖励、失败替代、结束。任意脚本节点必须经过技术设计审查。

任务合同包含：

- 玩家动机、角色动机、主题问题。
- 前置/互斥、入口、退出和超时替代。
- 每一步玩家动词与预计时长。
- 状态图、持久字段、幂等事件和恢复策略。
- 与关系、地区、经济、工艺、战斗和《天工录》的影响。
- 导航、提示递进、无障碍替代和跳过演出。
- 正常、拒绝、失败、顺序打乱和读档测试矩阵。

成链支线至少三次状态变化，并在最终产生可回访后果。收集若干同类物品只有在过程中引入新空间、方法或角色判断时才可成为链条主体。

## 9. 地区与子地区模板

### 9.1 地区合同

- 地区主题问题、工艺域、社会网络和归位命题。
- 5 小时以上主线时长预算及其活动分布。
- 6 个大型子地区的角色、探索语法和进度关系。
- 主线章图、支线链、Boss、核心 NPC 与兴趣点预算。
- 新增系统/武器、复用系统、禁用系统和未来钩子。
- 流式、内存、导航、音频、天气和时间预算。
- 归位前后状态与跨地区变量。

### 9.2 子地区合同

- 入口、出口、远景地标、安全点、回环和快捷路。
- 通常 30–70 分钟首轮内容，10–25 分钟回访价值；地区终战型可达 90–100 分钟，但拆成 20–35 分钟可保存/退出段。
- 主路径、可选路径、至少一个空间推理或工艺互动。
- NPC 日程锚点、事件槽和遭遇槽，而非固定堆满内容。
- 归位前后至少三类变化。
- 战斗密度、安静段、叙事段、采集和社交预算。
- 流式 Cell、遮挡、导航、相机和性能压力场景。

## 10. 武器与能力模板

武器合同包含共同输入映射、三种架势、资源曲线、12–18 个基础/进阶招式、6–10 个工艺部件、教学顺序、敌人交点、动画/音频/特效预算、AI 使用规则和完整升级线。

能力必须由以下可复用积木表达：触发、目标选择、移动、命中形状、效果、成本、冷却、标签、反馈、取消和网络/回放语义。特殊机制可以扩展积木，但不能直接在角色脚本里偷偷改全局状态。

## 11. 对话与演出模板

- 对话节点只负责条件、选择、文本/语音、动作提示和事件，不直接写复杂玩法状态。
- 每段对话标注预计时长、可打断性、是否允许移动、是否可跳过与回看。
- 长于 90 秒的非关键对话需拆段或加入可玩行为；关键演出长于 3 分钟必须有明确节奏设计。
- 同一内容阶段先用临时配音/节拍验证，再进入正式录音，避免文本未锁就批量返工。
- 角色称谓由关系和场合规则生成，不在每句台词手写分支。

## 12. 内容包与所有权

内容以包发布，例如 `core`, `region_jiangnan`, `region_longquan`, `region_huizhou`, `localization_zh_cn`。每个包声明版本、依赖、资产哈希、入口、兼容的可执行版本、Schema 版本和迁移。

每个稳定 ID 只有一个拥有包；其他包通过公开 ID 引用。地区包不能覆盖核心定义，只能使用扩展点或显式补丁合同。内容加载顺序不应影响结果。

## 13. AI 协作规则

AI 可以加速草案、变体、测试和资料整理，但不能绕过契约与审查。

### 13.1 给 AI 的最小上下文包

- 本任务对应的 00/04/05/06 章节摘要。
- 目标模板与 Schema，不提供整个仓库让其猜结构。
- 允许引用的稳定 ID 清单。
- 内容预算、禁用捷径、完成定义和示例金标准。
- 明确输出格式、文件路径、校验命令和不得修改的范围。

### 13.2 AI 输出门禁

1. Schema 与引用校验。
2. 重复/近重复检测。
3. 世界正史、角色口吻、工艺资料和敏感性人工审查。
4. 在预览器或灰盒中实际游玩。
5. 自动测试与性能测试。
6. 人类负责人署名；“AI 生成”不代表无人负责。

AI 不得自行发明新顶层系统、新正史或长期依赖；此类建议必须作为提案进入 ADR/设计审查。

## 14. 评审会议只看证据

内容评审按固定顺序：

1. 对照 Pitch 复述玩家价值。
2. 现场从干净状态进入内容，不用调试跳过关键链路。
3. 展示正常、失败、拒绝、重试和读档。
4. 展示至少一个与其他系统联动的变式。
5. 对照预算看重复度、时长和性能。
6. 检查归位后世界变化。
7. 记录明确结论：通过、限期修改或退回；不能用“差不多”结案。

## 15. 规模与复用指标

每个里程碑追踪：

- 新内容中由现有模板直接创建的比例，目标 ≥ 90%。
- 为单个内容新增 Runtime 特判的数量，目标为 0。
- 创建一个灰盒 NPC/Boss/任务/子地区从模板到可预览的中位时间。
- 内容校验失败在提交前被本地工具捕获的比例。
- 相同玩法片段在 30 分钟窗口内的重复率。
- 每小时新信息/新决策/新空间/新机制/关系推进占比。

指标用于发现生产问题，不用于逼迫作者用低质量内容刷数量。

## 16. 模板变更

模板字段一旦进入正式内容即视为契约。变更必须：

1. 修改 Schema 和示例。
2. 提供旧实例迁移。
3. 更新编辑器和运行时绑定。
4. 更新契约测试和预览场景。
5. 批量迁移全部内容并生成差异报告。
6. 在 03 记录已经发布的变化。

禁止长期维护 `field`, `field2`, `legacyField` 这种无语义兼容层。

## 17. 内容 Definition of Ready

内容只有满足以下条件才从 Pitch 进入 Contract/Greybox：

- 有稳定 ID、拥有包、模板版本、负责人、目标里程碑和正史层级。
- 能用一句话说清玩家价值、地区主题和删掉它会损失什么。
- 玩家动词、预计时长、入口/退出、失败/拒绝/恢复路径已写明。
- 引用的 NPC、任务、Boss、能力、资产和本地化 key 已存在或有占位合同。
- 关系、奖励、世界状态和同步 Operation 中适用项有明确所有者。
- 有至少一个正常夹具、一个失败/乱序夹具、一个性能/规模夹具。
- 美术/音频预算、可访问性替代、研究来源和敏感性风险已登记。
- 不需要 Runtime 为某个显示名/地区 ID 写临时特判；需要新积木时先做技术设计。

未满足 DoR 的内容可以继续研究、白盒或写作，但不得进入正式包，也不得被路线图计作“实现中”。

## 18. 跨专业签字与退回权

| 内容单元 | 最终负责 | 必须会签 | 可退回原因 |
| --- | --- | --- | --- |
| 核心 NPC/关系弧 | 叙事负责人 | 内容、系统、任务、动画、QA | 只有台词变化、日程死锁、缺地区响应 |
| 普通敌人/精英 | 战斗负责人 | 关卡、动画、VFX/音频、QA | 读招不清、只是换皮加血、群战不可读 |
| Boss | 战斗负责人 | 游戏总监、叙事、关卡、工灯、技术美术、QA | 阶段不改决策、归位无证据、重试/辅助缺失 |
| 任务/对话 | 内容负责人 | 叙事、系统、关卡、可访问性、QA | 只有跑腿、不可恢复、选择无反馈 |
| 子地区 | 关卡负责人 | 内容、战斗、技术美术、性能、QA | 首轮/回访不成立、无地标/回环、超包/内存 |
| 武器/能力 | 战斗负责人 | 动画、工艺设计、UI、性能、QA | 复制旧武器、输入特判、构筑只有数值大小 |

QA 有权因不可复现或证据不足退回；研究/敏感性审阅有权阻止不当真实工艺表达进入 Content；技术负责人只能因合同、性能、安全和维护风险退回，不能单方面改写角色主题。

## 19. F1 金标准实例：沈砚与四时伞祟

F1 不同时实现全部工具页面，而是用两个正式稳定 ID证明模板系统：

### 19.1 `npc_shen_yan`

1. 从核心 NPC 模板生成 ID、关系、日程、交互、本地化和测试夹具。
2. 日程至少含作坊教学、工位工作、雨停休整、Boss 后复盘和缺席代理。
3. F1 只实现“试手”关系阶段，但数据必须能迁移到“托谱/并肩署名”。
4. 互动贯通对话、伞器指导、一次部件调校和战后记录，不制作角色专属 UI。
5. 刷新、拒绝教程、战败、证据不足和完整归位后均有合法对白/任务入口。

### 19.2 `jn_boss_umbrella_wraith`

1. 从章节 Boss 模板建立春骨、夏雨、秋风、冬纸四节点图；F1 可压缩演出，但不可删掉三次决策变化。
2. F1 的两个现场工痕为“混装伞骨的四家刻痕”和“维护账页缺少共同工位”，完成一次伞骨调校试验即可组合成完整归位证据；M1 正式章节再加入纸面/桐油试验，形成更丰富的归位变体，不抬高 F1 已声明路径的门槛。
3. 直接制伏只暂时封存作坊；完整归位恢复联合工位并改变沈砚对白和天工录署名。
4. 失败快照位于 Boss 门前；刷新只能回到门前或已提交结算，不能进入半阶段。
5. 同一内容由 Native/WASM ContentCore 校验、`.tgdpack` 烘焙、Web/Windows 回放和辅助模式共同验证。

F1 还用 `f1_beat_umbrella_lane_first_encounter` 验证“战斗—空间操作—战斗”的内容夹具：双伞偶清场、棚缝检查、积水疏通、诱饵升起、纸鹭清场、路线选择必须是同一有序 Objective 链；敌群激活只引用诱饵 Objective，不允许表现层按动画、计时或物理按键自行刷出第二波。当前三项操作证明 Definition/前置/坐标/波次边界可生产，不代表环境模拟已经完成；正式内容仍需补齐可保存机关状态、可读演出、多种解决路径、失败替代、NPC/世界反馈与 Workbench 时间线预览，并通过 `ISnapshotContributor` / Persistent Operation 接入跨域事务。

F1 的 `f1_beat_canopy_return_encounter` 同时验证最小叠加增援夹具。每项 `QuestEncounterActivationDefinition` 必须显式选择 `mode`：`replace` 可作为 Beat 入场或 Objective 触发的完整切组；`reinforce` 必须由同 Beat Objective 触发，只能点名当前休眠且未被击败的 Actor，并为每个 Actor 提供不与当前 Active 敌人冲突的 Home Pose/Formation Slot。需要由先前选择控制时，必须同时填写稳定的 `requiredSelectionObjectiveId` 与 `requiredSelectionId`；选择 Objective 必须来自已发生的 `choose` 交互，同一 Beat/触发 Objective 的所有变体必须引用同一选择 Objective、option 不重复并完整覆盖该选择的全部 authored options。半门、未来门、无条件/有条件混合或覆盖不全均失败关闭。入口组与增援落点仍须位于该 Beat 安全点的 Encounter aggro 范围内；内容作者不得用 Presentation 计时器、动画回调或物理按键直接刷出增援，倒地重试必须按 Beat/触发 Objective 去重重放已完成边界。

返程的动作证明同样必须内容化。`QuestCombatTriggerDefinition` 需要由先前选择控制时，必须成对填写 `requiredSelectionObjectiveId` / `requiredSelectionId`；同一 Objective 的全部变体必须引用同一项已发生选择、不得重复 option、不得混入无条件变体，并完整覆盖 authored options。当前实例把 `f1_objective_demonstrate_rib_calibration` 写成两项互斥 Definition：春肋要求檐守式重击，冬肋要求翻花式轻击。物理键、动画帧、HUD 状态和 Presentation 计时器都不能直接完成该 Objective；内容作者也不得复制 Objective 来规避完整选项覆盖。

当前返程实例只证明“3 人入口组 + 校准 Objective + 已选春肋时 1 地面伞偶 / 已选冬肋时 1 高位纸鹭 = 4 人混合组 + 对应姿态/Ability 动作证明”可由 JSON/Schema→C++ 合同生产，并不放行任意数量的增援或通用条件图。多单位同时入场、职责组合、多个 Objective 会合、布尔复合条件、动态队形、多攻击令牌、刷出/退场演出、正式数值与 Workbench 条件/动作/时间线预览仍须单独建模和盲测；内容团队不得复制多条单人激活或动作触发器来绕过这些缺口。

这两个实例通过后才把它们当作其他 NPC/Boss 的金标准；“30 分钟生成模板”仅代表骨架，不代表写作、动画和玩法已经完成。

## 21. F1 前两 Beat 内容技术协作合同

`F1-GAME-01-CONTENT-A` 只使用 `f1-vertical-slice.schema.json` 1.4.0 已有的通用 Definition。当前 JSON 可以表达顺序 Objective、带前置的 inspect/operate/talk/choose、互斥选择门、Actor 激活、战斗动作触发、清除结果、安全点与双结算；下表明确本阶段实际采用和刻意不采用的边界。

| 需求 | 1.4.0 做法 | 本阶段边界 |
| --- | --- | --- |
| 雨渡可选观察/跳过 | 一个三选一 `choose` Objective；三项带选择门的 condition 交互汇入同一后续 Objective | 是选择变体，不是任意 Optional Objective；不发奖励 |
| 系缆错误恢复 | 两项方案选择；快结方案由“纠正过载快结”交互完成同一主 Objective | 不持久化绳结网格/机关物理状态 |

| 重复/返回 | Quest 已完成集合、选择集合、击破 Actor 集合与安全点确定性恢复 | Presentation 不保存权威进度；无重复奖励 |

Schema 的数量下限只用于防止本金标准悄悄退回骨架，不证明内容质量或分钟数。任何作者都不得通过复制 Objective、单人激活或具体 ID 的 Runtime 分支绕过下列缺口。

### 21.4 证据夹具与内容评审

- Node 正向夹具必须覆盖三条抵达观察路线 × 两条系缆路线的全部选择门；负向夹具至少删除跳过汇流、快结纠正、选择覆盖、一个动作 Trigger、一个 Actor 激活或一个 Outcome，并验证生成器失败关闭。
- Native 正向夹具必须证明错序无推进、重复幂等、选择不可改写、失败恢复后激活边界可重放，以及两种架势均由实际 Combat Trigger + Outcome 完成。
- Web Single 固定命令路线至少各跑一条迎风/背风组合，检查 Canvas 可操作、错误日志、Quest 状态与固定回放 checksum；Web Shell 的显示文案不作为内容权威。
- 真人报告必须另列逐段实际分钟、卡点、重复感、排除分钟与口头指导次数。若任何玩家需要开发者指出交互点，样本保留为卡点证据但不计“无需指导完成”。

真人报告模板固定在 `tests/fixtures/f1-first-two-beats-playtest-evidence.template.json`，初始状态必须保持 `pending`、P01–P05 为空，不能预填策划估时。`tools/validate-f1-playtest-evidence.mjs` 用唯一 F1 Definition 的 SHA-256、前两 Beat 全 Objective 前缀和三组稳定 selection 校验观察记录，并从单调时间点减去 `loading`、暂停/设置、挂机、失败重试、重复刷取、长距离空跑、强制降速和等待八类区间；卡住、受指导或未完成的样本仍可合法保存，但不会通过盲测门。

- `node tools/validate-f1-playtest-evidence.mjs` 只检查待采集模板没有伪造样本。
- `node tools/validate-f1-playtest-evidence.mjs <实际报告.json> --require-blind-gate` 要求五人、集成玩家构建、非合成录像引用、零开发界面暴露/口头指导、全部完成、三/二/二路线覆盖、中位数 900–1020 秒且任一相邻内容反馈间的合格时间不超过 90 秒。
- `--synthetic` 只验证计算器与排除规则；输出即使其余数值全绿也固定 `actualHumanEvidence=false`，不得复制、改名或截图后写入 P01–P05。
